## 1. 段階共通（各段階の末尾で必ず実施）

各段階（FE / NVS暗号化なし確認 / SB / 封鎖）ごとに、計画書のとおり以下を挟む。**省略禁止**。

1. **読戻し**：`python -m espefuse summary`（ポートは当日環境）、必要な検証ログ。
2. **再起動**：起動異常、`secure boot`、`Flash Encryption` 表示、業務MQTT等の確認は計画書・試験仕様。
3. **4点証跡**：`ProductionTool`／ESP32シリアル／**`espefuse summary` before-after**／`試験記録書.md`（詳細 **`7095`**）。
4. **安定性確認セット（`007-0009`）**：OTA 連続2回、`AP` 10分、`STA`10分。**異常時は停止**して `問題点記録書.md` に記録。

## 3. `ProductionTool` / API（参照のみ）

実コマンドは **`コマンド仕様書.md` §13** の curl 例、および `ProductionTool/README.md`、`LocalServer` の `/api/workflows/production/`・`/api/production/` 系列を当日の稼働版に合わせる。dry-run と本実行の違いは **計画書 6.3d** を正とする。

- [ ] **PT-005a（任意）** `ProductionTool` が `LocalServer` へ不可逆**監査**受理を POST する経路は、**eFuse 実書込みではない**（`SecretCore` の管理器は焼き込み本体を実行しない）。**本番1台目で §6.4 の実機不可逆を既に実施済み**のとき、当日の追加入力が不要なら **空欄でスキップ**してよい（`PRODUCTION_TOOL_DISABLE_IRREVERSIBLE_HANDOFF=1` でも可）。理由: 監査の二重化を避けるため。
- [ ] **注意** `SecretCore` は AP `GET /api/production/state` を照合し、`state` が空でない場合は **`precheck_collected`** を要求する。実機作業後に AP 側状態が一致しないと workflow は**失敗**する（ハードは焼かれない）。その場合は PT-005a を**走らせない**（スキップ）。理由: 実績ある検証ロジックを変えず運用で吸収するため。
