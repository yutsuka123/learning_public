# IoT 設計概要

[重要] 本プロジェクトは ESP32-S3 / Local / Cloud で構成するIoT基盤の段階実装を目的とする。  
理由: 将来のCloud連携を見据えて、現行段階（第3段階）を破綻なく設計するため。

## 1. 目的
- ESP32-S3 と Local アプリ間で安全に制御・状態取得できる構成を確立する。
- 第3段階として、ROM暗号化・セキュアブート・eFuse有効化を前提にした運用へ移行する。
- 第3段階として、MQTT通信をTLS + ログインID/パスワード認証でセキュア化する。

## 2. 構成要素
- Production（メーカー専用の製造、初回書込み、初回セキュア化、eFuse 実行ツール）
- ESP（ESP32-S3）
- Local（現行は Windows + TPM 2.0 前提の LocalServer + Web UI + SecretCore + ローカルMQTTブローカ）
- Cloud（将来対応: 認証・配信基盤）

## 2.1 言語方針
- [厳守] ESP実装は C/C++ を採用する。
- [推奨] ESPのC++実装は可能な範囲で C++20 を利用する。  
  理由: 可読性・型安全性・保守性を高めるため。
- [厳守] `LocalServer` 実装は現行では Windows + TPM 2.0 を前提とし、TypeScript / Node.js を採用する。
- [厳守] Web UIはブラウザベースで実装し、`LocalServer` の REST API を利用する。
- [将来対応] Cloud実装は Python 等を候補とし、第4段階で確定する。

## 3. 段階計画
- [廃止の方針] 第1段階: 本プロジェクトでは実施しない。理由: 第3段階セキュリティ実装を優先するため。
- [廃止の方針] 第2段階: 本プロジェクトでは実施しない。理由: OTA機能より先に通信路保護を完了させるため。
- [厳守] 第3段階（現行対象）: ROM暗号化、セキュアブート、eFuse有効化に加え、MQTTをTLS + ログインID/パスワード認証で運用する。
- [将来対応] 第4段階: Cloudで認証・書換管理（AWS/Google想定）へ拡張。

## 3.1 直近ゴール（2026-03-08時点）
- [重要] OTA が 2面（A/B）で交互に書き込まれることを試験で確認する。
- [重要] OTA 途中失敗時でも旧パーティションから継続起動できることを試験で確認する。
- [重要] ESP32 の Wi-Fi / MQTT / HTTPS / その他設定を内部保存し、メンテナンス用 Wi-Fi AP モードで修正できることを確認する。
- [重要] `k-iot` 系の本番用鍵、署名素材、証明書運用を本番前提で整理する。
- [重要] Secure Boot、Flash Encryption、ROM暗号化を一通り有効化し、動作確認する。
- [重要] 最終的に eFuse 有効化後でも OTA によるバージョンアップが継続できることを直近完了条件とする。
- [重要] AP 共通トップ画面、Production 追加認証、`runPairingSession()` 内部の `createPairingBundle`、`k-user` 再発行後の current/previous key 運用を試験可能な粒度まで固める。

## 4. 設計方針
- [厳守] ESPビルドは PlatformIO を使用し、`platformio.ini` にライブラリバージョンを固定記載する。  
  理由: ビルド再現性とセキュリティ検証の再実行性を担保するため。
- [厳守] 対象チップは ESP32-S3 固定。  
  理由: セキュリティ機能検証の対象差分を抑えるため。
- [禁止] 認証情報（鍵、トークン、秘密値）をGitへアップロードしない。  
  理由: 漏えい時の影響が甚大で回復困難なため。
- [推奨] 認証ロジックは公開可能部分としてモジュール分離する。  
  理由: 秘密情報非依存のコードレビューと再利用性を高めるため。
- [厳守] MQTTブローカ接続はTLSを必須とし、平文MQTT（1883）を利用しない。  
  理由: 通信盗聴・改ざんリスクを低減するため。
- [厳守] MQTT接続時はログインID/パスワード認証を必須とする。  
  理由: 不正クライアント接続を抑止するため。
- [厳守] Wi-Fi設定更新 payload はアプリ層暗号（AES-256-GCM）で保護する。  
  理由: ブローカ内部ログ/誤配信時の漏えいリスクを低減するため。
- [厳守] 鍵方式は `TPM -> S_random -> k-user -> k-device` の階層を採用する。  
  理由: デバイス個別鍵の決定的再生成、別PC複製防止、同一PC再インストール復旧を両立するため。
- [重要] `k-user` の保護方式は `TPM由来方式` と `暗号化ファイル + パスワード方式` を選択可能とし、どちらも `SecretCore` の Rust 実装内でのみ復号・利用する。  
  理由: 冗長構成や別PC復旧を許容しつつ、復号済み秘密の露出を防ぐため。
- [重要] `TPM由来方式` を選択した場合でも、暗号化済みバックアップファイルによる復旧経路を持たせる。  
  理由: TPM 依存だけでは同時稼働冗長構成や障害復旧に対応しにくいため。
- [厳守] `public_id` は公開識別子として扱い、初期値は `IoT_<macアドレスからコロン除去>` を許容する。  
  理由: 運用上のわかりやすさと初期導入時の識別容易化を優先するため。
- [厳守] Wi-Fi設定は二段階確定（未確定保存→接続成功後確定）とし、失敗時は旧設定へロールバックする。  
  理由: 誤設定による遠隔喪失を防ぐため。
- [推奨] 現地復旧としてAPモード（WPA2/Web UI）を実装する。  
  理由: 最悪時の保守性を担保するため。
- [厳守] AP モードは顧客保守と Production で AP 名を分けず、共通トップ画面から分岐する。  
  理由: 現地手順を単純化しつつ、認証と画面分離で機能境界を保つため。
- [厳守] AP 共通トップ画面には共通パスワード認証を要求し、Production 系操作にはメーカーモード専用パスワードの追加認証を要求する。  
  理由: 同一 AP 名でも通常保守と高リスク操作を安全に分離するため。
- [厳守] ESP32 の主要運用設定（Wi-Fi / MQTT / HTTPS / 時刻同期 / その他設定）は内部永続化し、主保存先は NVS とする。  
  理由: 再起動や一時的な通信断後も運用設定を保持し、平文ファイル依存を避けるため。
- [厳守] `k-device`、Wi-Fi / MQTT / OTA / 認証情報は ESP32 の NVS に保存し、鍵は current/previous の 2 スロットで管理する。  
  理由: 初回投入、再ペアリング、48h 猶予、240h 保持後削除の運用を単純に実現するため。
- [推奨] LittleFS は補助的なファイルシステム領域として利用してよいが、主要運用設定の主保存先にはしない。  
  理由: 設定値の一貫性と機密保護を優先するため。
- [厳守] 鍵情報全般、Wi-Fi / MQTT / HTTP / HTTPS / NTP / ユーザー名 / パスワード等の主要接続・認証情報は `NVS` に保存する。  
  理由: 少量の重要設定を一貫したキー管理で扱い、再起動後も安全に復元するため。
- [厳守] 画像、証明書、ログは `LittleFS` の `/images` `/certs` `/logs` に分離保存する。  
  理由: ファイル単位更新、列挙、削除、保守作業をしやすくするため。
- [重要] MQTT / HTTPS 用証明書実体は FW 埋め込みから `LittleFS` `/certs` へ移行し、版数や SHA-256 などの管理メタ情報は `NVS` に保持する。  
  理由: 証明書更新のたびに FW 再ビルドや再配布へ依存しないようにするため。
- [重要] 画像一斉更新は `MQTT` を第一経路とし、`SHA-256` 付き manifest による差分更新、削除、全削除を許可する。  
  理由: 従来の全面書換ではログ等の非対象ファイルを巻き込みやすく、更新時間も長くなるため。
- [厳守] `LittleFS` 更新失敗時は旧ファイル維持とし、`tmp` 書込みと `rename` による安全更新を採用する。  
  理由: 途中失敗や通信断時でも既存運用資産を壊さないため。
- [重要] ファイルログはシリアル出力と別経路で実装し、必要時のみ有効化する。時刻同期済み時は日付パス、未同期時は起動回数パスへ保存し、30 日または 100KB 超過で古いものから削除する。  
  理由: 通常運用時の情報露出を抑えつつ、必要時だけ追跡可能な証跡を残すため。
- [重要] メンテナンス用 Wi-Fi AP モードから、Wi-Fi AP、MQTT ブローカー、HTTPS 配布先などの接続設定を現地修正できるようにする。  
  理由: ネットワーク構成変更後もメンテナンスモードで復旧可能にするため。
- [重要] `Server` と `MQTT Broker` は別責務とし、`1 Server - 複数 Broker` を将来対応として考慮した識別子設計を採用する。  
  理由: 地域分散、台数増加時の分散、冗長化を後から追加しやすくするため。
- [重要] `ESP32` は将来の複数 `Server` 同時接続を考慮し、`serverId` 単位で `brokerId`、`wifiApId`、認証情報、証明書メタ情報、`k-device` を区別できる保存構造を前提とする。  
  理由: 単一接続前提の保存形式に固定すると、冗長構成追加時の手戻りが大きいため。
- [重要] システム全体では複数 `WiFi AP` を許容するが、各 `ESP32` の現行運用は固定1件とする。  
  理由: 大規模建屋での分散配置を許容しつつ、現時点の実装複雑度を抑えるため。
- [重要] テレメトリと `status` は同一 `Broker` を複数 `Server` が subscribe することで共有してよく、OTA は要求元 `Server` に対してのみ応答する。  
  理由: `ESP32` 側の送信分岐を増やさずに、同時稼働冗長構成へ備えるため。
- [厳守] 同一種別の要求が複数 `Server` から到着した場合は、早いもの順に処理する。  
  理由: 現時点で最小限の競合解決ルールを固定し、優先順位の曖昧さを避けるため。
- [厳守] `k-device` 初回投入と `k-user` 再発行後の再ペアリングは、`SecretCore` の `runPairingSession()` を正規経路とし、その内部の `createPairingBundle` を用いた AP モード経由を基本とする。  
  理由: TS 側へ raw key を返さずに、公開 workflow と内部 helper を分離したまま安全に投入するため。
- [厳守] pairing bundle は固定公開鍵による署名検証と、AP モード接続時の ECDH 都度セッション鍵で保護する。  
  理由: 全台共通の固定共有秘密鍵を避け、漏えい影響を局所化するため。
- [厳守] 高リスク処理は `SecretCore` の Rust 主導ワークフローとして実行し、`LocalServer` は開始要求、入力検証、進捗表示、結果表示のみを担当する。  
  理由: TS 側へ高リスク手順知識や中間秘密を極力持たせず、完了判定までを秘密処理層へ閉じ込めるため。
- [厳守] Rust 主導ワークフローの代表は `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` とする。  
  理由: `createPairingBundle` や署名生成などの低レベルAPIを TS が逐次組み立てないようにするため。
- [厳守] `LocalServer` が受け取る高リスク処理の状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` の進捗と最終結果に限定する。  
  理由: UI で必要な可視化を確保しつつ、秘密値や高リスク手順の詳細を TS 側へ広げないため。
- [重要] OTA標準運用は「メーカーが firmware.bin を作成・署名し、顧客環境の LocalServer が実行する」方式とする。  
  理由: 配布物の真正性をメーカー側で担保しつつ、現地更新の手間を抑えられるため。
- [厳守] OTA開始時は LocalServer 側で再認証を要求し、同一ブラウザセッション内では3時間まで再入力不要とする。  
  理由: 高リスク操作の誤実行・離席時実行を抑止するため。
- [重要] eFuse / Secure Boot / Flash Encryption の最終有効化は、顧客向け通常モードと分離した `Production` ツールでのみ実施する。  
  理由: 不可逆処理を工場内専用ツールへ閉じ込め、顧客環境での誤操作や復旧困難事故を防ぐため。
- [厳守] `Production` ツールは `LocalServer` の通常運用 UI と混在させない。  
  理由: 通常運用機能と高リスク製造機能の境界を明確にするため。
- [厳守] `Production` 操作は AP 共通トップ画面の追加認証後にのみ進入可能とし、認証は通常管理者パスワードと別の「メーカーモード専用パスワード」で行う。  
  理由: 同一 AP 名運用でも不可逆操作の入口を厳格に制御するため。
- [推奨] eFuse 書込み本体は、過去のシリアル焼込み中断事故を踏まえ、ESP32 側の専用セキュア化処理で段階実行してよい。  
  理由: ホスト側シリアル依存だけにせず、書込み途中状態をデバイス自身が判定しながら完了または隔離判定へ進められるため。
- [厳守] ESP32 内部でセキュア化する場合でも、通常アプリや通常OTAと共用せず、`Production` 専用のセキュア化フローとして分離する。  
  理由: 高リスク処理を通常運用経路から切り離し、設計単純性と監査性を高めるため。
- [禁止] 顧客向け MQTT 指令や Web UI から eFuse 有効化を直接実行させない。  
  理由: MQTT は通常運用で便利だが、不可逆処理の一般公開経路としては危険性が高いため。
- [厳守] `Production` 実行画面は、事前確認 -> 対象機確認 -> 実行内容確認 -> 最終確認 -> 段階実行 -> 完了判定 の順で進むウィザード形式とする。  
  理由: 危険操作を一括実行させず、各段階で操作者が意図確認できるようにするため。
- [推奨] 最終確認では対象機のシリアル番号、MAC、作業指示番号などの再入力を要求してよい。  
  理由: 対象取り違えや誤クリックを減らせるため。
- [厳守] eFuse 最終有効化に使うファームウェアと手順は、試験機で十分にリハーサルした版のみを本番機へ適用する。  
  理由: 不可逆処理を未検証の組み合わせへ直接適用すると回収不能事故になりやすいため。
- [厳守] ESP32 側の専用セキュア化処理は、十分な専用スタックサイズを持つタスクで実行し、通常処理と競合させない。  
  理由: スタック不足や割込み競合で処理が不安定になることを防ぐため。
- [推奨] セキュア化開始前にスタック高水位、空きヒープ、停止対象タスクを確認し、閾値未達時は開始拒否としてよい。  
  理由: 実行コンテキストが不安定な状態を事前排除できるため。

## 5. セキュリティ境界
- 公開可: 通信フロー、状態遷移、検証ロジック、署名検証の一般処理。
- 非公開: 秘密鍵、実トークン、証明書実体、実運用エンドポイントの機密設定。

## 7. 2026-03-03 決定事項
- [重要] 主対応OSはWindows、副対応OSはmacOSとする。  
  理由: 現行運用を優先しつつ、将来の運用拡張に備えるため。
- [厳守] ローカルMQTT BrokerはMosquittoを利用する。  
  理由: 実績があり軽量で、ローカル運用に適しているため。
- [厳守] OTA配布はHTTPSで実施する（manifest + firmware）。  
  理由: 改ざん・盗聴リスクを低減するため。
- [重要] 同時接続台数は10台を設計前提とする。  
  理由: 現実的な小規模運用要件に合わせるため。
- [厳守] LocalServer/LocalSoft の自動起動方式は Task Scheduler を採用する。  
  理由: Windows運用における導入・保守コストを抑えるため。
- [重要] LocalSoft は SQLite へ保存し、保存期間は日単位で可変（`0` は無期限、既定10日）とする。  
  理由: 小規模運用でのシンプル運用と保持要件の両立のため。
- [重要] LocalSoftForCloud はクラウドとの双方向同期を前提とする。  
  理由: 設定取得と制御反映を双方から実行できるようにするため。
