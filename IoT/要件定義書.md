# IoT 要件定義書

[重要] 本書は第3段階以降の実装要件を定義する。  
理由: 現在の実装優先範囲を固定し、段階混在による手戻りを防ぐため。

## 1. 適用範囲
- 対象: ESP32-S3 + Local（現行は Windows + TPM 2.0 前提、macOSは将来対応）構成
- 段階: 第3段階のみを実装対象、第4段階は将来要件として保持
- [廃止の方針] 第1段階・第2段階は本プロジェクトでは実施しない

## 2. 機能要件
- FR-001: MQTT接続をTLSで確立できること（第3段階）
- FR-002: MQTT接続時にログインID/パスワード認証を必須化できること（第3段階）
- FR-003: TLSハンドシェイク失敗時に接続拒否し、失敗理由をログ出力できること（第3段階）
- FR-004: ログイン認証失敗時に接続拒否し、失敗理由をログ出力できること（第3段階）
- FR-005: ROM暗号化を有効にした状態で動作できること（第3段階）
- FR-006: セキュアブートを有効にした状態で動作できること（第3段階）
- FR-007: 必要なeFuse設定を有効化し、設定状態を追跡できること（第3段階）
- FR-008: Wi-Fi設定更新payloadをAES-256-GCMで復号し、改ざん検知できること（第3段階）
- FR-009: `k-device = HKDF(ikm=k-user, salt=SHA256(base_mac), info="k-device-v1")` 方式でデバイス別鍵を利用できること（第3段階）
- FR-010: Wi-Fi設定更新は未確定保存し、MQTT再接続成功 + CONFIRM受信後に確定できること（第3段階）
- FR-011: 新設定接続失敗時に旧設定へ自動ロールバックできること（第3段階）
- FR-012: AP復旧モード（WPA2/固有パスワード/Web UI）で現地復旧できること（第3段階）
  - AP名: `AP-esp32lab-<MAC(no colon)>`, Pass: `pass-esp32`（将来機密化）
- FR-013: 設定用外部AP (`AP-esp32lab-setting`) が存在する場合、自動的に接続し設定を受け付けること（第3段階）
- FR-014: 初回セットアップ時に `S_random` を生成し、`wrapped_secret = TPM_Encrypt(S_random)` を保存できること（第3段階）
- FR-015: 初回ペアリングで `base_mac` 取得 -> `k-device` 導出 -> デバイス保存を実施できること（第3段階）
- FR-016: `wrapped_secret` と `device_db` のバックアップ/復元が可能であり、同一PC再インストール後に `k-user` / `k-device` を再生成できること（第3段階）
- FR-017: MQTT経由でネットワーク設定（Wi-Fi/MQTT接続情報）を変更できること（第3段階）
- FR-018: `LocalServer` をTypeScript/Node.jsで実装し、REST APIを提供できること（第3段階）
- FR-019: Web UI（ブラウザ）からデバイス管理、OTA配布管理、鍵管理、監査確認を実施できること（第3段階）
- FR-020: OTA配布はHTTPSでmanifest + firmwareを配信できること（第3段階）
- FR-021: ローカルMQTT Broker（Mosquitto）に接続するMQTT Clientを備えること（第3段階）
- FR-022: 同時接続10台規模で運用できること（第3段階）
- FR-023: LocalServer起動時にMQTT `call/status` 要求を自動送信し、応答でonline/offline初期同期できること（第3段階）
- FR-024: online/offline判定はESP32 `status` 通知、status要求応答、Will/切断通知、最終受信時刻で評価できること（第3段階）
- FR-025: PC起動時にTask SchedulerでLocalServerを自動起動できること（第3段階）
- FR-026: `LocalServer` は MQTT経由情報を SQLite へ保存できること（第3段階）
- FR-027: `LocalServer` の保存期間は日単位設定とし、`0` 指定時は無期限保持、既定値は10日とする（第3段階）
- FR-028: `LocalServer` はクラウド連携モード（第4段階準備）で双方向同期を実施できること（第4段階準備）
- FR-029: OTAの標準運用は「メーカー署名済み firmware.bin を顧客環境の LocalServer が実行配布する」方式とする（第3段階）
- FR-030: OTA開始前に LocalServer で管理者再認証を要求し、同一ブラウザセッション内では3時間まで再入力不要とできること（第3段階）
- FR-031: `ProductionTool` は `LocalServer` と分離したメーカー専用ツール群として、工場内でのみ eFuse / Secure Boot / Flash Encryption の最終有効化フローを実行できること（第3段階）
- FR-032: eFuse 最終有効化を ESP32 内部処理で実施する場合は、`ProductionTool` 配下の専用セキュア化処理として実装し、事前条件確認、段階実行、各段階の readback 検証、完了ログ保存、途中中断時の再点検フローを備えること（第3段階）
- FR-033: AP 共通トップ画面は通常運用 UI に対して共通パスワード認証後のみ表示し、ProductionTool 操作は追加認証後の専用画面からのみ実行できること（第3段階）
- FR-034: ProductionTool 操作は通常の管理者再認証とは別の「メーカーモード専用パスワード」を要求し、通常管理者パスワードと共用しないこと（第3段階）
- FR-035: ProductionTool のセキュア化実行は、事前確認 -> 対象機確認 -> 実行内容確認 -> 最終確認 -> 段階実行 -> 完了判定 の順で段階的に進め、各段階で操作者確認を要求できること（第3段階）
- FR-036: eFuse 最終有効化の前に、試験機で十分なリハーサルを完了し、対象ファームウェア・対象手順・対象鍵IDの組み合わせが妥当であることを確認できること（第3段階）
- FR-037: ESP32 内部の専用セキュア化処理は、eFuse 書込みと readback 検証を安全に実行できる十分なタスクスタックサイズを確保し、実行前後のスタック余裕を確認できること（第3段階）
- FR-038: ESP32 は Wi-Fi、MQTT、HTTPS、時刻サーバー、その他運用設定を内部永続保存し、再起動後も保持できること（第3段階）
- FR-039: ESP32 の運用設定保存は NVS を正とし、機密設定や接続設定を平文ファイルへ保存しないこと（第3段階）
- FR-040: LittleFS は補助的なファイルシステム領域として利用可能とするが、Wi-Fi / MQTT / HTTPS などの主要運用設定の主保存先にはしないこと（第3段階）
- FR-041: ESP32 は Wi-Fi AP モードのメンテナンスモードを提供し、Wi-Fi AP、MQTT ブローカー、HTTPS 配布先などの設定を現地修正・保存できること（第3段階）
- FR-042: ESP32 の AP 共通トップ画面はパスワード認証後に表示できること（第3段階）
- FR-043: AP 共通トップ画面から ProductionTool 操作へ進む場合は、メーカーモード専用パスワードによる追加認証を要求できること（第3段階）
- FR-044: `public_id` の初期値を `IoT_<macアドレスからコロン除去>` として生成でき、必要に応じてユーザーが変更できること（第3段階）
- FR-045: 初回投入および再ペアリング時に、`runPairingSession()` 内部の `createPairingBundle` 処理により、AP モード接続中の ECDH セッションと固定公開鍵検証を用いて `k-device` と関連設定を安全に投入できること（第3段階）
- FR-046: ESP32 は `k-device`、Wi-Fi、MQTT、OTA、認証情報を NVS に保存できること（第3段階）
- FR-047: ESP32 は `currentKeyVersion/currentKDevice` と `previousKeyVersion/previousKDevice` の 2 スロットで鍵状態を保持できること（第3段階）
- FR-048: `k-user` 再発行後、ESP32 は旧鍵を稼働時間 48 時間だけ併用し、その後 240 時間の監査保持期間を経て自動削除できること（第3段階）
- FR-049: 高リスク処理は `SecretCore` の Rust 主導ワークフローとして、対ESP32通信開始から完了判定まで実行できること（第3段階）
- FR-050: Rust 主導ワークフローの対象は少なくとも `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` を含むこと（第3段階）
- FR-051: `LocalServer` は高リスクワークフローの進捗 `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` と最終結果のみを受け取り、UI へ表示できること（第3段階）
- FR-052: ESP32 は鍵情報全般、Wi-Fi、MQTT、HTTP、HTTPS、NTP、ユーザー名、パスワード、その他主要接続・認証情報を `NVS` に保存できること（第3段階）
- FR-053: ESP32 は画像、証明書、ログを `LittleFS` の `/images` `/certs` `/logs` に分離保存できること（第3段階）
- FR-054: MQTT / HTTPS 用証明書実体は `LittleFS` の `/certs` から読込でき、版数、SHA-256、アクティブ状態などの管理メタ情報を `NVS` に保存できること（第3段階）
- FR-055: 画像一斉更新は `MQTT` を第一経路として、`SHA-256` 付き manifest による差分更新、削除、全削除を実行できること（第3段階）
- FR-056: 画像または証明書の更新失敗時は旧ファイルを維持し、更新対象外ファイルと `/logs` を保持できること（第3段階）
- FR-057: ESP32 は AP モード時にも `/images` と `/certs` を局所更新できること（第3段階）
- FR-058: ESP32 はファイルログをシリアル出力と別経路で保存し、再起動時または 1KB 到達時に新しいログファイルへ切り替えできること（第3段階）
- FR-059: ESP32 は時刻同期済み時に `/logs/<年>/<月日>/...`、未同期時に `/logs/<起動回数>/...` 形式でログ保存できること（第3段階）
- FR-060: ESP32 はログを 30 日経過または総量 100KB 超過時に、古いものから自動削除できること（第3段階）
- FR-061: `Server` は複数の `ESP32` を管理でき、設計上 `1 Server - 複数 MQTT Broker` を許容できること（第3段階）
- FR-062: `ESP32` は将来の複数 `Server` 同時接続を考慮し、`serverId` ごとに `brokerId`、`wifiApId`、認証情報、証明書メタ情報、`k-device` を区別して保持できること（第3段階）
- FR-063: システム全体では複数 `WiFi AP` を許容し、`ESP32` ごとに異なる `WiFi AP` を設定できること（第3段階）
- FR-064: 各 `ESP32` の現行運用は固定1件の `WiFi AP` 設定を持てること（第3段階）
- FR-065: `k-user` の保護方式は `TPM由来方式` と `暗号化ファイル + パスワード方式` を選択でき、復号処理は `SecretCore` の Rust モジュール内で完結できること（第3段階）
- FR-066: `TPM由来方式` を選択した場合でも、暗号化済みバックアップファイルとパスワードにより別PCで復旧できること（第3段階）
- FR-067: `k-device` は `Server` ごとに別物とする運用と、複数 `Server` 間で共通利用する運用の両方を、ユーザー選択により許容できること（第3段階）
- FR-068: テレメトリおよび `status` は同一 `Broker` を subscribe する複数 `Server` で共有でき、OTA は要求元 `Server` の情報に対してのみ応答できること（第3段階）
- FR-069: 同一種別の要求が複数 `Server` から到着した場合は、早いもの順に処理できること（第3段階）
- FR-070: LocalServer / ProductionTool は、指定USB Wi-Fiインタフェースで `AP-esp32lab-<MAC(no colon)>` を探索し、複数個体を順次メンテナンスできること（第3段階）
- FR-071: AP メンテナンス処理は「探索、接続、ログイン、設定/更新、再起動要求、次個体へ遷移」の順で実行できること（第3段階）
- FR-072: AP 経由更新の最終成否は、再起動後の `status` topic 応答で判定できること（第3段階）
- FR-073: APモード遷移は「起動時ボタン3秒長押し」「MQTT `call/maintenance` 指令」「Wi-Fi/MQTT必須設定未投入時自動遷移」で実行できること（第3段階）
- FR-074: APモード中は MQTT タスクと通常モード専用タスクを起動しないこと（第3段階）
- FR-075: APモード中は青LEDを `0.3ON->0.3OFF->0.3ON->0.3OFF->1.0ON->0.3OFF` パターンで反復し、LCD に `MaintenanceMode` を表示できること（第3段階）
- FR-076: AP Web UI は `user`/`maintenance`/`admin`/`mfg` のロール別権限で画面/API制御できること（第3段階）
- FR-077: LocalServer の「設定 -> 管理者画面」遷移はユーザー名/パスワード認証を必須とすること（第3段階）
- FR-078: LocalServer 管理者画面で `k-user` 発行および `k-device` 発行を実行できること（第3段階）
- FR-079: LocalServer 管理者画面で Wi-Fi USB インタフェース指定により APモード ESP32 を探索・接続し、ネットワーク設定/パスワード/`k-device` を自動設定できること（第3段階）
- FR-080: APモード ESP32 の Webページへ手動遷移するボタンを、デバイス一覧へ表示できること（第3段階）
- FR-081: AP接続中個体と MQTT接続中個体を統合一覧表示し、FW版数/FS版数/書換進捗を表示できること（第3段階）
- FR-082: LocalServer から MQTT接続中のESP32へメンテナンス再起動（`call/maintenance`）指令を送信できること（第3段階）

## 3. 非機能要件
- NFR-001: [厳守] ESPターゲットはESP32-S3のみとする。理由: 機種差異の排除。
- NFR-002: [厳守] PlatformIOでビルドし、`platformio.ini`の依存バージョンを固定する。理由: 再現性確保。
- NFR-003: [厳守] 認証情報はGit管理禁止。理由: 機密漏えい防止。
- NFR-004: [推奨] 認証ロジックは機密値非依存な実装として公開可能にする。理由: 監査容易化。
- NFR-005: [厳守] MQTTはTLS必須（非TLSポート利用禁止）。理由: 通信路の機密性・完全性確保。
- NFR-006: [厳守] MQTTはID/パスワード認証必須。理由: 接続元の最低限認証を担保。
- NFR-007: [厳守] ESP実装言語はC/C++とする。理由: ESP-IDF資産との整合確保。
- NFR-008: [推奨] ESPのC++コードは可能な範囲でC++20を利用する。理由: 保守性向上。
- NFR-009: [厳守] `LocalServer` 実装はWindowsを主対象とし、TypeScript/Node.jsを採用する。理由: 現行運用環境に合わせるため。
- NFR-010: [将来対応] Cloud実装言語はPython等を候補とし、実装開始時に確定する。理由: 要件未確定のため。
- NFR-011: [厳守] `S_random` / `k-user` は平文保存せず、`wrapped_secret` を主とする保護保存物または暗号化バックアップファイルで管理し、Git保存禁止。理由: 鍵漏えい防止と冗長構成時の復旧性を両立するため。
- NFR-012: [厳守] MQTT公開IDは `public_id` を使用し、Wi-Fiステーション名や通常 topic に base_mac を直接使用しない。`public_id` の初期値は `IoT_<macアドレスからコロン除去>` を許容する。理由: 初期導入時の識別容易化と、通常運用での生のMAC露出抑制を両立するため。
- NFR-013: [将来対応] MQTT認証はmTLS拡張可能な設計とする。理由: AWS連携時の移行容易化。
- NFR-014: [厳守] OTA配布物の真正性はメーカー署名で担保し、顧客は署名済み配布物のみ実行可能とする。理由: 誤配布・改ざん配布・旧版再配布のリスクを下げるため。
- NFR-015: [禁止] 顧客向け通常モード、通常OTA、公開MQTT経路から eFuse 有効化や `ProductionTool` フローを起動させない。理由: 不可逆処理を現地運用から分離し、誤操作・不正操作を防ぐため。
- NFR-016: [厳守] `ProductionTool` フローはメーカー認証、閉域ネットワークまたは工場治具環境、監査ログ保存を必須とする。理由: セキュア化処理の責任境界と追跡性を明確にするため。
- NFR-031: [厳守] `ProductionTool`、`LocalServer`、`SecretCore`、`ESP32` の責任範囲は `モジュール仕様書.md` を親定義とし、他文書はこれと矛盾しないこと。理由: 通常運用、秘密処理、製造処理、保存処理の境界を固定するため。
- NFR-017: [重要] eFuse 書込み途中の電源断や通信断があり得る前提で、再実行時は eFuse 読戻し結果を基準に残工程のみを判定し、中途半端な個体は隔離判定できること。理由: 以前のシリアル焼込みで中途半端状態が発生した経験を再発防止へ反映するため。
- NFR-018: [厳守] 隠しボタンや隠し操作は入口秘匿化の補助手段とし、認証そのものはメーカーモード専用パスワードまたは同等以上の認証で担保する。理由: 隠しUIだけでは不正利用防止にならないため。
- NFR-019: [厳守] メーカーモード専用パスワードはリポジトリへ保存せず、通常のOTA再認証用パスワードと分離し、変更・失効可能な管理方式とする。理由: 工場用高権限認証情報の漏えい影響を限定するため。
- NFR-020: [厳守] eFuse 実行タスクは通常業務タスクと分離し、スタックオーバーフローやメモリ逼迫の疑いがある状態で開始しない。理由: eFuse 処理中の異常終了は不可逆事故に直結するため。
- NFR-021: [厳守] eFuse 書込み前に、スタック高水位・空きヒープ・主要周辺処理停止状態を確認し、閾値未達時は処理開始を拒否する。理由: 実行環境が不安定なまま高リスク処理へ入ることを防ぐため。
- NFR-022: [厳守] 機密を含む接続設定の主保存先は NVS とし、LittleFS を利用する場合も機密設定の主保存先へ戻さない。理由: 平文ファイル運用の再発を防ぐため。
- NFR-023: [厳守] 通常運用時のシリアル出力制約は eFuse ではなく FW 制御で実施し、量産通常FWではデバッグ出力・保守コマンドを最小化または停止できること。理由: 問題発生時に署名済み診断FWで一時的に解析する余地を残しつつ、通常運用での情報露出を抑えるため。
- NFR-024: [禁止] シリアル機能停止のみを目的に eFuse の不可逆設定を採用しない。理由: 現地障害解析や再現試験のために一時的なシリアル出力が必要になる可能性を残すため。
- NFR-025: [厳守] 問題解析が必要な場合に限り、メーカー管理下の診断用FWで一時的にシリアル出力を有効化できること。理由: 通常運用の秘匿性と、障害解析の実用性を両立するため。
- NFR-026: [厳守] インターネット接続時も踏み台化されないよう、不要待受停止、未使用ポート閉鎖、管理ポート非公開、認証失敗時の抑止、監査ログ保存を最低限適用する。理由: 侵入後の横展開や第三者攻撃への悪用を防ぐため。
- NFR-027: [禁止] 本プロジェクト要件外の汎用プロキシ、中継、ファイル共有、ポートフォワード機能を本番運用へ同居させない。理由: IoT 管理サーバーが踏み台として悪用される攻撃面を増やさないため。
- NFR-028: [厳守] AP 共通トップ画面は全デバイス共通の固定パスワードで閲覧認証し、将来変更可能な設定値として拡張余地を残す。理由: 現段階の運用簡便性を保ちつつ、将来の認証強化余地を残すため。
- NFR-029: [厳守] `runPairingSession()` の必須設定不足は TS 側で `SecretCore` 呼出し前に検出し、workflow を開始しない。理由: 公開 API を workflow 開始要求に限定したうえで、前段で運用入力不備を確実に排除するため。
- NFR-034: [禁止] `LocalServer` は `POST /api/pairing/bundle` のような `createPairingBundle` 直公開 API を提供しない。理由: 内部 helper を外部公開せず、秘密処理境界を狭く保つため。
- NFR-030: [厳守] `previousKDevice` は旧鍵無効化後も稼働時間 240 時間は監査・障害解析用に保持し、その後は自動削除する。理由: 障害解析余地を残しつつ、旧鍵を恒久保持しないため。
- NFR-032: [厳守] 高リスク処理の逐次オーケストレーションは TS 側で行わず、`SecretCore` が対ESP32通信、検証、完了判定まで責任を持つ。理由: TS 側の改ざんや実装ミスで高リスク処理の順序が崩れることを防ぐため。
- NFR-033: [厳守] `LocalServer` が高リスクワークフローから受け取る情報は進捗状態と結果に限定し、raw key、中間秘密、署名素材、逐次通信手順の内部状態を返さない。理由: UI に必要な可視化だけを残し、秘密処理境界を狭く保つため。
- NFR-035: [厳守] `LittleFS` の資産領域は少なくとも `/images` `/certs` `/logs` に分離し、画像更新や証明書更新がログ領域へ影響しないこと。理由: 更新対象と監査対象を分離し、巻き込み削除を防ぐため。
- NFR-036: [厳守] `LittleFS` のファイル更新は `tmp` 書込み後の `rename` により反映し、途中失敗時は旧ファイルを保持する。理由: 電源断や通信断でも既存資産を壊さないため。
- NFR-037: [厳守] 画像および証明書更新では `SHA-256` を必須とし、一致確認前に本番ファイルへ切り替えない。理由: 破損ファイルや改ざんファイルの反映を防ぐため。
- NFR-038: [厳守] MQTT / HTTPS 用証明書の主保存先は FW 埋め込みではなく `LittleFS` を優先し、移行後は恒久的な再ハードコードを行わない。理由: 更新容易性と証明書ローテーション性を確保するため。
- NFR-039: [厳守] ファイルログはシリアル出力と別関数で実装し、既定では必要最小限のみを有効化できる設計とする。理由: 通常運用での情報露出とフラッシュ消費を抑えるため。
- NFR-040: [厳守] `Server` と `MQTT Broker` は別責務として扱い、`serverId` と `brokerId` を混同しない。理由: 冗長化、地域分散、負荷分散の拡張時に責務境界を保つため。
- NFR-041: [厳守] `ESP32` の保存形式は将来の複数 `Server` 同時接続を考慮し、`serverId` 単位で設定を区別できる構造とする。理由: 単一接続前提の保存形式による手戻りを防ぐため。
- NFR-042: [厳守] `k-user` の復号済み平文は UI、ログ、通常 API、設定ファイルへ出力しない。理由: バックアップ性を確保しても秘密露出を増やさないため。
- NFR-043: [厳守] `k-device` の共有/分離状態、`k-user` の保護方式、復旧方式はメタ情報として保持し、運用開始後に曖昧な状態を残さない。理由: 同時稼働冗長構成での運用事故を防ぐため。
- NFR-044: [重要] 複数 `Broker` 対応でも当面の推奨運用は `1 Server - 1 Broker` とする。理由: 初期運用と障害解析の複雑化を抑えるため。
- NFR-045: [厳守] AP メンテナンスWeb UIの設定参照/設定更新/FW更新/画像更新はログイン認証後のみ許可する。理由: AP到達だけで高リスク操作が可能になる状態を防ぐため。
- NFR-046: [厳守] APモードと通常モードの切替は原則として再起動を伴う。理由: 通信状態・タスク構成・認証状態を確実に初期化するため。
- NFR-047: [推奨] AP Web UI の初期パスワードは、初回セットアップ時または運用開始前に変更する。理由: 固定クレデンシャル継続利用のリスクを下げつつ、現場運用の例外対応余地を残すため。
- NFR-048: [厳守] eFuse 設定機能は `ProductionTool` ツール専用とし、`LocalServer` 管理者画面から実行できないこと。理由: 不可逆処理の誤操作を通常運用系から分離するため。
- NFR-049: [推奨] AP Web UI ロール別パスワード、LocalServer 管理者パスワード、運用用サービスパスワード（MQTT/DB/外部連携）は固定値継続を避け、変更履歴（対象、実施者、日時、理由）を監査ログへ残す。理由: 初期値依存を減らし、他パスワード類も同一運用で追跡可能にするため。

## 4. 制約条件
- [禁止] 実運用の鍵・証明書・トークンをリポジトリへ保存しない。
- [新規利用禁止] ハードコードされた固定秘密値（例: APIキー文字列）の新規追加を禁止。
- [禁止] 自己署名証明書の検証スキップ（開発一時対応を恒久化する運用）。
- [将来対応] Cloud認証はAWS/Googleいずれかを選定し、第4段階で要件確定する。
- [厳守] Wi-Fi MACをdevice_id用途に使用しない（変更可能なため）。
- [厳守] メーカー共通鍵（`K_master_global`）を導入しない。

## 5. 受入観点（第3段階）
- MQTT接続がTLSで成立し、証明書検証に失敗した場合に接続が拒否されること。
- MQTT接続がID/パスワード認証で成立し、認証失敗時に接続が拒否されること。
- ROM暗号化 + セキュアブート + eFuse有効の構成で、MQTT接続機能が維持されること。
- 認証情報を含むファイルがGit差分に現れないこと。
- Wi-Fi設定誤投入時、タイムアウト後に旧設定へロールバックし遠隔管理を継続できること。
- base_mac直接公開なしで `public_id` ベースのMQTT運用が成立すること。
- `public_id` 初期値が `IoT_<macアドレスからコロン除去>` でも運用上識別でき、必要に応じて変更できること。
- バックアップ復元後に既存デバイスとの暗号通信を継続できること。
- 通常運用FWではシリアル高詳細ログや保守コマンドが停止または最小化され、診断用FWでのみ一時的に有効化できること。
- インターネット接続時も不要待受、未使用ポート、汎用中継機能が残っていないこと。
- AP 共通トップ画面にパスワード認証があり、ProductionTool 操作は追加認証で分離されていること。
- `k-user` 再発行後に個体ごとの current/previous 鍵運用、48 時間旧鍵併用、240 時間監査保持後自動削除が成立すること。
- 高リスク処理で `LocalServer` が受け取る情報が `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` の進捗と結果のみで成立すること。
- `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` が `SecretCore` 主導で完了判定まで実行されること。
- 鍵情報全般、Wi-Fi / MQTT / HTTP / HTTPS / NTP / 認証情報が `NVS` から復元されること。
- 画像更新が `/images` の差分更新で成立し、削除・全削除を含めても `/logs` が保持されること。
- 証明書更新後に `LittleFS` `/certs` のファイルと `NVS` メタ情報が整合し、再接続で利用されること。
- ログが時刻同期済みと未同期で定義通りのパスへ保存され、30 日または 100KB 超過で古いものから削除されること。
- `serverId` / `brokerId` / `wifiApId` を用いた識別が文書・設定・実装で矛盾しないこと。
- `k-user` の `TPM由来方式` と `暗号化ファイル + パスワード方式` の両方で、復号済み秘密をユーザーへ露出せずに復旧できること。
- `k-device` の共有運用と分離運用のどちらを選んだかを追跡できること。
- 複数 `Server` 同時稼働前提でも、OTA は要求元 `Server` の情報に対してのみ応答できること。
