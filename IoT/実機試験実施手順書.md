## 事前準備

- LocalServer 起動: `cd IoT/LocalServer && npm run build && npm run start`
- MQTT ブローカー稼働
- ESP32 実機（APモード移行可能な個体）
- USB Wi-Fi アダプタ（LocalServer と同じ PC に接続、`data/settings.json` の `wifiUsbInterfaceName` に設定）
- `試験仕様書.md` を手元に参照

## 7027: ESP32 設定永続保存試験

1. ESP32 を通常運用で起動（MQTT 接続済み状態）。
2. LocalServer 管理画面または API で以下を更新する:
   - Wi-Fi / MQTT / HTTPS / 時刻同期 / その他設定
3. `POST /api/admin/commands/maintenance-reboot` 等で ESP32 を再起動する。
4. 再起動後に `GET /api/admin/ap/settings` または AP 経由で設定を取得し、NVS から復元されていることを確認する。
5. 主保存先が NVS であることをログまたは仕様で照合する。

期待結果: 設定値が再起動後も保持される。

## 7064 残課題: APロールパスワード変更・credentials/rotation

- **7064 済み**: LocalServer 管理者パスワード変更と監査ログ（`npm run test:7064` で実施済み）。
- **残り**:
  1. ESP32 を最新 FW でビルド・書込みすること（`/api/auth/password/change` が 404 の場合は古い FW の可能性あり）。
  2. AP モード機体へ USB Wi-Fi で接続する（起動時3秒長押し、または maintenance-reboot 送信後 20 秒待機）。
  3. `node scripts/runRealDeviceTests.mjs --step 7064` で AP パスワード変更・credentials/rotation を実行。
  4. 手動: `POST /api/admin/ap/password/change` で AP ロール（maintenance/admin/mfg）パスワードを変更する。
  5. 手動: `POST /api/admin/credentials/rotation` で例外継続記録を実行する。
  6. `logs/security-audit.log` で「変更実施」「例外継続」が記録され、平文パスワードが出力されていないことを確認する。

## 7065: AP ログイン必須・RBAC 試験

[重要] 7065 実施前に LocalServer を `npm run build && npm run start` で再起動すること。  
理由: `test-unauth` API を利用するため、ビルド後のプロセス起動が必須。起動中の古いプロセスでは 404 となる。

**自動実行**: `node scripts/runRealDeviceTests.mjs --step 7065` で未認証401・admin→mfg変更403を確認可能。

1. ESP32 を AP モードへ移行（起動時3秒長押し、または `POST /api/admin/commands/maintenance-reboot`）。
2. USB Wi-Fi で `AP-esp32lab-<MAC(no colon)>` に接続する。
3. 未認証で以下へアクセスし `401 unauthorized` を確認:
   - 設定参照 / 設定更新 / FW更新 / 画像更新 API
4. `maintenance` ロールで上記 API 到達可能を確認する。
5. `admin` ロールで certs 更新可能を確認する。
6. `mfg` ロールのみ mfg パスワード変更可能を確認する。

期待結果: 未認証は拒否、各ロールは定義済み権限範囲でのみ成功。

## 7066: AP 探索・順次一括メンテナンス試験

1. `POST /api/admin/ap/batch/start`（`targetSsids` 未指定）を実行する。
2. `AP-esp32lab-<MAC(no colon)>` を探索する。
3. 「探索 -> 接続 -> ログイン -> 設定更新 -> 再起動要求 -> 次個体」を 1 台ずつ順次実行する。
4. `GET /api/admin/ap/batch/:batchId` で結果を取得する。

期待結果: 複数台を順次処理し、個体ごとの成功/失敗理由が追跡できる。

## 7067: AP 更新後の最終成功判定試験

1. 7066 の AP 更新フローを実行する。
2. 再起動前の AP 切断状態と再起動後 `status` topic 応答を比較する。
3. `waitForStatusRecoveryByDevice` 経由で `public_id` / `fwVersion` / `configVersion` を収集し成否判定する。
4. AP 切断のみでは成功扱いにしない。再起動後 `status` 応答で所定項目が収束した場合のみ成功とする。

## 実施順序の推奨

1. 7027（NVS永続）→ 単体で確認可能
2. 7064 残（APパスワード変更・credentials/rotation）→ AP接続後に実施
3. 7065（ログイン必須・RBAC）→ AP接続後に実施
4. 7066（一括メンテ）→ 7065 後に実施
5. 7067（再起動後 status 判定）→ 7066 の更新フロー実行後に実施
