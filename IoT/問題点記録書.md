# IoT 問題点記録書

[重要] 本書は開発時に発生した問題と原因・対策を追跡管理する。  
理由: 同一障害の再発防止と、切り分け時間の短縮のため。

## 記録 #0001
- 見出し: ESP32からMQTT(1883)へ到達できない
- 種類: コマンド
- 分野: サーバー
- 状態: 解決済
- 試験番号: 7001
- 日時: 2026-03-02

### 症状
- ESP32 側ログで `WiFiClient connect timeout 3000 ms` が継続し、`kMqttInitDone` を受信できない。

### 原因
- ブローカPCのWi-Fiネットワークプロファイルが `Public` だったが、FWルール `mqtt` は `Private` のみ適用だった。
- その結果、同一LAN内のESP32から `172.16.1.59:1883` への受信許可が適用されなかった。

### 対策
- ブローカPCのFW適用プロファイルを実ネットワーク条件に合わせる（Public/Private整合）。
- 必要に応じて MQTT受信ルールを Public へも適用。
- 接続検証時にネットワークプロファイル種別を初期確認項目へ追加。

### 再発防止
- [厳守] `運用記録_ESP32_MQTT接続検証_2026-03-02.md` の「重要な注意点」を接続試験前チェックとして運用する。
- [推奨] 将来は IoT専用LAN を `Private` とし、ルールをサブネット限定する。

## 記録 #0002
- 見出し: ESP32のみMQTT over TLS証明書検証に失敗する（-9984）
- 種類: デバッグ
- 分野: サーバー
- 状態: 解決済
- 試験番号: 7002
- 日時: 2026-03-06

### 症状
- ESP32ログで `(-9984) X509 - Certificate verification failed` が継続し、MQTT初期化が完了しない。
- 一方でPC側クライアント（`mosquitto_pub/sub`、`mqtt.fx`）は同一Brokerへ接続できる。

### 原因
- DNS名運用時は `mqttUrl=<パソコン名>` をESP32側で解決できず、`DNS Failed` が発生した。
- `mqttUrl=172.16.1.59` へ変更後も、ESP32（Arduino-ESP32 / mbedTLS）で証明書照合条件が厳密に一致せず、証明書検証エラーが発生した。

### 対策
- サーバ証明書のSANを以下構成へ調整して再発行した。
  - `IP:172.16.1.59`
  - `DNS:172.16.1.59`
  - `DNS:CF-FV_1`（任意）
  - `DNS:localhost`（任意）
- ESP32接続先は `SENSITIVE_MQTT_URL="172.16.1.59"` を維持し、DNS依存を回避した。
- CA証明書(`ca.crt`)をESP32の `SENSITIVE_MQTT_TLS_CA_CERT` に同期反映した。

### 再発防止
- [厳守] 接続試験時は `mosquitto_pub/sub` とESP32ログを同時確認し、DNS・証明書・認証の失敗層を分離して判定する。
- [推奨] SANは `IP` と `DNS` の両方を登録し、クライアント実装差異の影響を避ける。

## 記録 #0003
- 見出し: ESP32 から LocalServer OTA HTTPS(4443) へ接続できない
- 種類: デバッグ
- 分野: サーバー
- 状態: 解決済
- 試験番号: 7010
- 日時: 2026-03-07

### 症状
- `POST /api/commands/ota` に対して ESP32 は `ota start queued` を受信し、`esp32lab/notice/otaProgress/<device>` へ `prepare / retry / error` を publish する。
- しかし OTA 本体では `HTTPClient connect(): failed connect to mqtt.esplab.home.arpa:4443` および `returnError(): error(-1): connection refused` が出力され、更新処理へ進めない。

### 原因
- PC側 `LocalServer` では `OTA HTTPS started. httpsPort=4443` を確認しており、同一PCからの `curl -k https://127.0.0.1:4443/ota/manifest.json` / `firmware.bin` は取得できる。
- `Test-NetConnection -ComputerName 172.16.1.59 -Port 4443` は PC上で成功しているため、LocalServer の待受自体は成立している。
- Windows 側 `4443/TCP` 許可後の再試行でも、ESP32 では引き続き `connection refused` が再現した。
- 追加切り分けの結果、問題は二段構えだった。
- `https://172.16.1.59:4443/...` の IP直指定では `tcpProbe` は成功するが、`(-9984) X509 - Certificate verification failed` で TLS ハンドシェイクが失敗した。
- `https://mqtt.esplab.home.arpa:4443/...` の hostname 指定では ESP32 側DNS解決が失敗するが、fallback IP `172.16.1.59` へ接続しつつ、TLS検証ホスト名を `mqtt.esplab.home.arpa` のまま維持すると `Certificate verified.` となり OTA が成功した。
- 以上より、根本原因は「単純な到達性不足」ではなく、「ESP32のDNS不達」と「IP直指定時の証明書検証条件不一致」の組み合わせだった。

### 対策
- ESP32 OTA処理を `HTTPClient` 依存から外し、`WiFiClientSecure::connect(ip, port, tlsHostName, caCert, ...)` を用いる実装へ変更した。
- OTA URL が hostname の場合は、`WiFi.hostByName()` 失敗時に `SENSITIVE_MQTT_FALLBACK_IP` を利用して接続先IPを補完するようにした。
- TLS証明書検証は接続IPではなく hostname のまま維持し、`mqtt.esplab.home.arpa` 名義の証明書と一致させた。
- 実機再試験で `executeSingleOtaAttempt success.` と再起動後の `StartUp` status を確認し、OTA成功を確認した。
- OTA進捗通知は `write` を5%刻みに間引き、`done` は再起動前に待ち時間を確保して publish するよう調整した。
- LocalServer 側でも `start-up` status 受信時に `otaPhase=done` / `otaProgressPercent=100` / `otaDetail=rebooted after ota` へ収束させ、画面が古い `write` 状態で残らないようにした。

### 再発防止
- [厳守] OTA用HTTPSポート追加時は、Node起動確認だけでなく「別端末（ESP32）」からの到達確認を必ず行う。
- [推奨] `コマンド仕様書.md` または `環境仕様書.md` に、OTA配布ポートのFirewall受信許可手順を明記する。
- [厳守] ESP32側で hostname URL を使う場合は、DNS不達時の fallback IP と TLS検証ホスト名を分離して扱う。理由: 「接続先IP」と「証明書照合名」は同一である必要がないため。
- [禁止] 証明書検証を回避するために `setInsecure()` へ戻さない。理由: MQTT/OTAともに同一LAN内でもなりすまし検知を維持するため。

## 記録 #0004
- 見出し: 新ルータ運用開始後、ESP32 が NTP 同期待ちで MQTT 接続開始まで進まない
- 種類: 試験
- 分野: サーバー
- 状態: 解決済
- 試験番号: 7028
- 日時: 2026-03-08

### 症状
- `WiFi AP=172.17.1.1` `DNS=172.17.1.100` 環境で ESP32 書込み後、Wi-Fi 接続までは成功する。
- シリアルログで `applyWifiDnsConfiguration success. primary=172.17.1.100` と `wifi event handler: DNS re-applied after GOT_IP. dns1=172.17.1.100` は出る。
- しかし直後に `connectToWifiRouter DNS resolved setting. dns1=172.17.1.1` となり、`timeService::syncNow` がタイムアウトして `mainTaskEntry: skip mqtt connect until UTC sync for startup will registration.` のまま MQTT 接続へ進まない。

### 原因
- 暫定切り分けでは、CoreDNS 自体の名前解決機能は生きている。  
  - `nslookup mqtt.esplab.home.arpa 172.17.1.100` -> `172.17.1.100`
  - `nslookup ntp.esplab.home.arpa 172.17.1.100` -> `172.17.1.100`
  - `nslookup pool.ntp.org 172.17.1.100` -> 成功
- `Test-NetConnection -ComputerName 172.17.1.100 -Port 8883` も成功しているため、MQTT ポート到達性自体は成立している。
- 初回は `w32tm /stripchart /computer:ntp.esplab.home.arpa /samples:1 /dataonly` と `w32tm /stripchart /computer:172.17.1.100 /samples:1 /dataonly` が `0x80072746` で失敗したが、その後 `W32Time` 起動により応答成功へ変化した。
- 診断用に ESP32 の `timeServerUrl` を `172.17.1.100` 直指定へ変更すると、`timeService::syncNow succeeded.` と `NTP SYNC OK` を確認できた。
- よって主原因候補は以下のいずれか。
  - ESP32 の最終利用 DNS が `172.17.1.100` ではなく `172.17.1.1` へ戻っている
  - `ESP32` の `hostByName(mqtt.esplab.home.arpa)` と `SNTP(configTime)` で利用している解決経路/タイミングが異なる
  - `mqtt.esplab.home.arpa` に対する DNS 応答だけが ESP32 で取得できていない

### 対策
- [完了] `172.17.1.100` 上で UDP/123 の待受と `W32Time` 応答を確認する。
- [厳守] ESP32 の Wi-Fi 接続完了後に、実際に参照される DNS が `172.17.1.100` のままか再確認する。
- [厳守] `172.17.1.100:8883` で Mosquitto 等の MQTT ブローカー待受が継続している前提で、NTP サービスだけを個別に切り分ける。  
  理由: `8883/TCP` は到達しており、MQTT 側とNTP側を分離して調査できるため。
- [推奨] `mqtt.esplab.home.arpa` の A レコードが想定どおり `172.17.1.100` を返すか CoreDNS 設定を再確認する。  
  理由: 現行 `Corefile` では `172.17.1.100` を返す運用へ更新済みであり、クライアント側キャッシュや経路差異を切り分けるため。
- [推奨] 次段階の診断として、`mqtt.esplab.home.arpa` も一時的に `172.17.1.100` 直指定へ変更し、MQTT TLS 接続自体が成立するか確認する。  
  理由: MQTT 名解決問題と TLS/認証問題をさらに分離できるため。
- [完了] `MQTT` と `OTA` の接続順序を「IP設定ありなら先にIPで接続試行、TLS検証の正規名はFQDN維持」へ変更する。  
  理由: ローカル環境では DNS 不安定区間より先に IP で疎通を確立し、起動時間短縮と切り分け容易性を両立するため。
- [完了] `beta.8` 実機再試験で `connectToMqttBroker success. state=0` と `publishStatusNotice success.` を確認する。  
  理由: `IP優先 + FQDN保持` の併用仕様が実機で有効であることを確認できたため。

### 再発防止
- [厳守] 新ルータ切替時は「DNS Aレコード」「UDP/123」「ESP32実機ログ」を1セットで確認する。
- [禁止] NTP 未成立のまま MQTT 不達を DNS 単独の問題と断定しない。  
  理由: 現行起動シーケンスでは NTP 未同期だけで MQTT 処理が未実行になるため。
