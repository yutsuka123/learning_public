- 7028: 実施済（NG）
  - 日時: 2026-03-08
  - 実施内容: `python -m platformio run`、`python -m platformio run -t upload --upload-port COM9`、`nslookup mqtt.esplab.home.arpa 172.17.1.100`、ESP32 シリアルログ確認。
  - 結果要約: `mqtt.esplab.home.arpa` は `172.17.1.100` への DNS 問い合わせで解決でき、ESP32 書込みも成功した。ESP32 ログでも `applyWifiDnsConfiguration success. primary=172.17.1.100` と `wifi event handler: DNS re-applied after GOT_IP. dns1=172.17.1.100` を確認した。一方で起動後は `connectToWifiRouter DNS resolved setting. dns1=172.17.1.1` が出力され、続く `timeService::syncNow` がタイムアウトして `mainTaskEntry: skip mqtt connect until UTC sync for startup will registration.` のまま MQTT 接続処理へ進めなかった。  
    理由: 現行実装では UTC 同期完了まで MQTT 接続を開始しないため、NTP 側の成立を先に解消する必要があるため。
  - 追記(2026-03-08): `nslookup ntp.esplab.home.arpa 172.17.1.100` は `172.17.1.100` を返し、`nslookup pool.ntp.org 172.17.1.100` も成功した。さらに `nslookup mqtt.esplab.home.arpa 172.17.1.100` は `172.17.1.100` を返し、`Test-NetConnection -ComputerName 172.17.1.100 -Port 8883` も `TcpTestSucceeded=True` を確認した。  
    理由: CoreDNS の現行Aレコードと MQTT ポート到達性は成立しており、残課題が `NTP同期` と `ESP32 で最終表示される DNS=172.17.1.1` の切り分けへ絞れたため。
  - 追記(2026-03-08): `w32tm /stripchart /computer:ntp.esplab.home.arpa /samples:1 /dataonly` および `w32tm /stripchart /computer:172.17.1.100 /samples:1 /dataonly` は、いずれも `0x80072746` エラーで時刻取得できなかった。  
    理由: `172.17.1.100` 側の NTP サービス実体または `UDP/123` 応答に問題が残っている可能性が高いため。
  - 追記(2026-03-08): `W32Time` 起動後に `w32tm /stripchart /computer:172.17.1.100 /samples:3 /dataonly` と `w32tm /stripchart /computer:ntp.esplab.home.arpa /samples:3 /dataonly` は成功し、`UDP/123` 待受も確認できた。そのうえで ESP32 の時刻同期先を診断用に `172.17.1.100` 直指定へ変更して再書込みしたところ、`timeService::syncNow succeeded.` と `NTP SYNC OK` を確認し、その後 `MQTT RECONNECT` へ進んだ。  
    理由: `ESP32 -> 172.17.1.100:123/UDP` は成立しており、NTP サービス実体の問題ではないことを確認できたため。
  - 追記(2026-03-08): NTP 直指定成功後も `mqtt.esplab.home.arpa` については `hostByName(): DNS Failed for mqtt.esplab.home.arpa` が再現し、`pingBrokerHost failed. hostByName failed and fallback is unavailable.` により MQTT 初期化は失敗した。  
    理由: 切り分けの結果、現在の主問題は `ESP32` 上の MQTT 向け DNS 名前解決経路であり、NTP 自体は別問題として解消したため。
  - 追記(2026-03-08): `MQTT` と `OTA` の接続順序を「IP設定ありなら先にIPで接続試行、TLS検証の正規名はFQDN維持」へ変更し、`beta.8` を `COM9` へ再書込みした。実機ログで `timeService::syncNow succeeded.`, `connectToMqttBroker success. state=0`, `publishStatusNotice success. ... sub=start-up onlineState=Online` を確認した。  
    理由: `hostByName(mqtt.esplab.home.arpa)` の不安定性を避けつつ、ローカル環境での起動時間短縮とTLS検証整合を両立できることを確認したため。
