- 7010: 実施済（NG）
  - 日時: 2026-03-07
  - 実施内容: `IoT_F0D0F94EB580` へ `firmwareVersion=1.1.0-beta.2` / `sha256=e44290dfdc8cb9e52a281b77030f3ca5f1f78dbd7e2da942754630cb1f74ce60` を含む OTA 要求を送信。
  - 結果要約: ESP32 側で `ota start queued` と `otaProgress` publish は確認できたが、`HTTPClient connect(): failed connect to mqtt.esplab.home.arpa:4443` / `returnError(): error(-1): connection refused` により OTA 本体は失敗した。  
    理由: PC側 `LocalServer` の HTTPS 配布ポート `4443/TCP` へ ESP32 から到達できず、Windows 側受信許可が不足している可能性が高いため。
  - 追記(2026-03-07): Windows 側で `4443/TCP` 許可後に再試行したが、`GET /api/devices` では引き続き `otaPhase=error` / `otaDetail=http status -1` を確認した。  
    理由: Firewall 許可以外にも、ESP32 から見た `4443/TCP` 到達経路に未解決要因が残っているため。
- 7011: 実施済（OK）
  - 日時: 2026-03-07
  - 実施内容: ESP32 OTA 実装を `HTTPClient` 依存から外し、`hostname URL + fallback IP + hostnameでTLS証明書検証` 方式へ変更後、`POST /api/commands/ota` で `https://mqtt.esplab.home.arpa:4443/ota/firmware.bin` を指定して再試験。
  - 結果要約: ESP32 ログで `resolveOtaTargetIp fallback will be used. host=mqtt.esplab.home.arpa fallbackIp=172.16.1.59`、`Certificate verified.`、`executeSingleOtaAttempt success.`、`OTA SUCCESS`、`rst:0xc (RTC_SW_CPU_RST)` を確認した。`GET /api/devices` でも再起動後に `onlineState=online` / `detail=StartUp` / `lastSeenAt=2026-03-07T08:26:50.563Z` を確認した。  
    理由: DNS不達環境でも、TLS検証ホスト名を維持したまま fallback IP へ接続すれば OTA が成立することを実機で確認できたため。
  - 追記(2026-03-07): ESP32 側で `write` 通知を5%刻みに間引き、`done` 通知を再起動前に待つよう改善したうえで、LocalServer 側でも `start-up` status 受信時に OTA 完了表示へ収束させる補正を追加した。再試験で `GET /api/devices` が `otaProgressPercent=100` / `otaPhase=done` / `otaDetail=rebooted after ota` を返すことを確認した。  
    理由: 実機OTA成功後にブラウザが `95%/write` で残留しないことを確認するため。
