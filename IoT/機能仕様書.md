# IoT 機能仕様書

[重要] 本書は段階別の機能仕様を定義し、現行対象を第3段階に固定する。  
理由: 実装優先順位を明確化し、機能追加時の混線を防止するため。

## 1. 段階別機能

### 1.1 第1段階 [旧仕様]
- F1-001: ESP起動通知（MQTT Publish）
- F1-002: Local GUIからLED制御指令（MQTT Publish）
- F1-003: ESP側でLED制御指令の受信・反映（MQTT Subscribe）
- [廃止の方針] 本プロジェクトでは実施しない。

### 1.2 第2段階 [旧仕様]
- F2-001: LocalからFW書換モード遷移要求
- F2-002: ESPのHTTPS OTA（`esp_http_client`）
- F2-003: 分割転送、再開転送対応
- F2-004: OTA進捗MQTT通知（%と状態）
- F2-005: デュアルROM領域を交互利用した更新
- [廃止の方針] 本プロジェクトでは実施しない。

- F3-025: デュアルROM（2面）で非実行領域へ書込み、検証成功時に次回起動先を切替えできる
- F3-026: ESP32 の Wi-Fi / MQTT / HTTPS / 時刻同期 / その他運用設定を NVS へ永続保存できる
- F3-027: メンテナンス用 Wi-Fi AP モードの Web UI から、Wi-Fi / MQTT / HTTPS / その他設定を修正し、保存できる
- F3-028: Wi-Fi AP や MQTT ブローカー変更後でも、メンテナンスモード経由で設定修正し復旧できる
- F3-029: LittleFS は補助ファイル領域として利用可能とするが、主要運用設定の主保存先にはしない

- F3-031: AP 共通トップ画面から Production 画面へは、メーカーモード専用パスワードの追加認証で遷移できる
- F3-032: AP 共通トップ画面に `shortId`、`public_id`、版数、Production対応/非対応、eFuse / Secure Boot / Flash Encryption / シリアル状態を表示できる
- F3-033: `public_id` の初期値を `IoT_<macアドレスからコロン除去>` とし、未ペアリング時は `pending` と表示できる
- F3-034: `runPairingSession()` 内部の `createPairingBundle` 処理により、`k-device` と Wi-Fi / MQTT / OTA / 認証情報を AP モード経由で一括投入できる
- F3-035: ESP32 は `k-device` を NVS の current/previous 2 スロットで保持できる
- F3-036: `k-user` 再発行後、旧鍵を稼働時間 48h だけ猶予し、その後 240h 監査保持後に自動削除できる
- F3-037: 新鍵切替済み個体では旧鍵の高権限操作を禁止し、未切替個体では旧鍵の高権限操作を許可できる
- F3-038: `LocalServer` は高リスク処理について、入力検証、ワークフロー起動要求、進捗表示、結果表示のみを担当する
- F3-039: `SecretCore` は `runPairingSession()` により、AP モード投入の対ESP32通信開始から完了判定まで実行できる
- F3-040: `SecretCore` は `runKeyRotationSession()` により、`k-user` 再発行後の再ペアリング処理を対ESP32通信開始から完了判定まで実行できる
- F3-041: `SecretCore` は `runProductionSecureFlow()` により、Production 高リスク処理の進行、検証、完了判定を実行できる
- F3-042: `SecretCore` は `runSignedOtaCommand()` により、署名付き OTA 開始指示の生成から送達確認・完了判定まで実行できる
- F3-043: 高リスクワークフローの進捗状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` を正規値として扱う

## 2. 状態遷移（ESP）
- `booting` -> `mqttTlsConnecting` -> `mqttAuthenticating` -> `running`
- [推奨] 状態遷移をMQTTへ逐次通知する。
- Wi-Fi更新時: `running` -> `wifiUpdatePending` -> `wifiReconnectTesting` -> `wifiConfirmWait` -> `running`
- ロールバック時: `wifiReconnectTesting` -> `wifiRollback` -> `running`
- 現地復旧時: `running` -> `apRecovery` -> `running`
- AP 共通画面認証時: `apRecovery` -> `apTopAuth` -> `apTop`
- Production 追加認証時: `apTop` -> `productionAuth` -> `productionReady`
- key 並存時: `running` -> `keyGraceRunning` -> `keyExpiredRetained` -> `keyDeleted`
- 高リスクワークフロー時: `workflowQueued` -> `workflowRunning` -> `workflowWaitingDevice` -> `workflowVerifying` -> `workflowCompleted`
- 高リスクワークフロー異常時: `workflowQueued|workflowRunning|workflowWaitingDevice|workflowVerifying` -> `workflowFailed`

## 3. 異常時要件
- TLS検証失敗時は接続を打ち切り、再試行間隔を設ける。
- MQTT認証失敗時は接続を打ち切り、再試行回数制限を適用する。
- 署名検証失敗時は旧ROM側へフォールバックする。
- eFuse適用不可時は処理停止し、Localへ失敗理由を通知する。
- Wi-Fi更新復号失敗/AES-GCM認証失敗時は設定適用を中止する。
- Wi-Fi更新の確定待ちタイムアウト時は旧設定へ戻す。
- OTAのSHA256不一致時は更新確定せず、失敗通知する。
- OTA再試行上限超過時はNG通知し、旧起動領域を維持する。
- AP 共通トップ画面パスワード失敗時は状態表示を許可しない。
- Production 追加認証を3回連続失敗した場合は AP 再接続または 5 分待機を要求する。
- `createPairingBundle` の必須設定不足時は TS 側で `SecretCore` 呼出し前にエラーとする。
- bundle の署名検証失敗または復号失敗時は `k-device` や設定値を保存しない。
- previous key 自動削除結果は UI に出さず、監査ログのみに残す。
- 高リスクワークフロー失敗時は `failed` を返し、TS 側は内部手順を再構成せずエラー表示と再試行要求のみに留める。
- [禁止] `POST /api/pairing/bundle` のような `createPairingBundle` 直公開 API を提供しない。

## 4. AP 共通トップ画面
- [厳守] AP 名は `esplab-ap-<shortId6>` とする。
- [厳守] AP 名は顧客向けメンテナンスと Production で共通化する。
- [厳守] 共通トップ画面には共通パスワード認証を要求する。
- [厳守] Production 画面へ進む場合はメーカーモード専用パスワードの追加認証を要求する。
- [重要] Production 追加認証成功後 10 分間は再入力不要とし、10 分経過後は高権限操作直前に再認証する。
- [重要] 共通トップ画面の表示項目は以下とする。
  - `shortId`
  - `public_id`
  - firmware 版数
  - `Production対応 / 非対応`
  - eFuse 実施済 / 未実施
  - Secure Boot 有効 / 無効
  - Flash Encryption 有効 / 無効
  - シリアル状態（開放 / 制限 / 停止）
  - current `keyVersion`
  - previous key 状態（`grace` / `expired-retained` / `deleted`）
  - Production 画面遷移可否

## 5. `k-user` 再発行の逐次再ペアリング
### 5.1 正常系
1. 管理者が `LocalServer` で `k-user` 再発行を開始する。
2. TS が管理者認証、対象確認、影響確認を行う。
3. TS は `runKeyRotationSession()` を `SecretCore` へ要求する。
4. `SecretCore` が新 `k-user` 系統と新 `keyVersion` を生成し、進捗を `queued` / `running` として返す。
5. 作業者は対象個体を 1 台ずつ AP モードへ入れる。
6. `SecretCore` は `waiting_device` 状態で対ESP32通信開始を待つ。
7. `SecretCore` が必要に応じて bundle 生成、送達、署名検証、復号確認を実行する。
8. ESP32 は `current/previous` キーと設定値を NVS に更新する。
9. `SecretCore` は `verifying` を経て `completed` または `failed` を返す。
10. TS は当該個体を「切替済」または「失敗」に更新する。

### 5.2 UI 表示
- 旧鍵猶予中
- 未切替台数
- 個体ごとの切替済 / 未切替
- 旧鍵高権限可 / 不可
- current `keyVersion`
- previous key 状態
- previous key 保持残り時間
- 最終エラー理由

## 6. `k-device` 初回投入
### 6.1 正常系
1. 新規個体を AP モードへ入れる。
2. 作業 PC が `esplab-ap-<shortId6>` に接続する。
3. 共通トップ画面パスワード認証を行う。
4. TS が必須 `requestedSettings` を検証する。
5. TS は `runPairingSession()` を `SecretCore` へ要求する。
6. `SecretCore` は `queued` / `running` を返しつつ、`public_id`、`k-device`、`keyVersion`、Wi-Fi / MQTT / OTA / 認証情報を含む bundle を生成する。
7. `SecretCore` は対ESP32通信を継続し、署名検証、復号、保存完了を確認する。
8. `SecretCore` は `verifying` を経て `completed` または `failed` を返す。
9. 保存完了後、通常運用へ移行する。

### 6.2 UI 表示
- `public_id`（未発行時は `pending`）
- セキュア状態
- シリアル状態
- 入力設定チェック結果
- `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed`
- エラー理由
- 監査記録済み表示

## 7. LocalServer workflow API
- [厳守] `LocalServer` が外部公開する高リスク処理 API は以下の workflow 系に限定する。
- `POST /api/workflows/pairing/start`
  - 用途: `runPairingSession()` 開始要求
- `POST /api/workflows/key-rotation/start`
  - 用途: `runKeyRotationSession()` 開始要求
- `POST /api/workflows/production/start`
  - 用途: `runProductionSecureFlow()` 開始要求
- `POST /api/workflows/signed-ota/start`
  - 用途: `runSignedOtaCommand()` 開始要求
- `GET /api/workflows/{workflowId}`
  - 用途: 進捗状態と結果取得
- [厳守] 応答項目は `workflowId`、`state`、`result`、`errorSummary` を基本とし、raw key や中間秘密を含めない。
- [禁止] TS が `createPairingBundle`、署名、暗号化、対ESP32送達を個別 API で逐次制御する構成を採用しない。
