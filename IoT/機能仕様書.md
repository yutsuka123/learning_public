# IoT 機能仕様書

[重要] 本書は段階別の機能仕様を定義し、現行対象を第3段階に固定する。  
理由: 実装優先順位を明確化し、機能追加時の混線を防止するため。

## 2. 状態遷移（ESP）
- `booting` -> `mqttTlsConnecting` -> `mqttAuthenticating` -> `running`
- [推奨] 状態遷移をMQTTへ逐次通知する。
- Wi-Fi更新時: `running` -> `wifiUpdatePending` -> `wifiReconnectTesting` -> `wifiConfirmWait` -> `running`
- ロールバック時: `wifiReconnectTesting` -> `wifiRollback` -> `running`
- 現地復旧時: `running` -> `apRecovery` -> `running`
- AP 共通画面認証時: `apRecovery` -> `apTopAuth` -> `apTop`
- ProductionTool 追加認証時: `apTop` -> `productionAuth` -> `productionReady`
- key 並存時: `running` -> `keyGraceRunning` -> `keyExpiredRetained` -> `keyDeleted`
- 高リスクワークフロー時: `workflowQueued` -> `workflowRunning` -> `workflowWaitingDevice` -> `workflowVerifying` -> `workflowCompleted`
- 高リスクワークフロー異常時: `workflowQueued|workflowRunning|workflowWaitingDevice|workflowVerifying` -> `workflowFailed`

## 3. 異常時要件
- TLS検証失敗時は接続を打ち切り、再試行間隔を設ける。
- MQTT認証失敗時は接続を打ち切り、再試行回数制限を適用する。
- 署名検証失敗時は旧ROM側へフォールバックする。
- eFuse適用不可時は処理停止し、Localへ失敗理由を通知する。
- Wi-Fi更新復号失敗/AES-GCM認証失敗時は設定適用を中止する。
- Wi-Fi更新の確定待ちタイムアウト時は旧設定へ戻す。
- OTAのSHA256不一致時は更新確定せず、失敗通知する。
- OTA再試行上限超過時はNG通知し、旧起動領域を維持する。
- AP 共通トップ画面パスワード失敗時は状態表示を許可しない。
- ProductionTool 追加認証を3回連続失敗した場合は AP 再接続または 5 分待機を要求する。
- `createPairingBundle` の必須設定不足時は TS 側で `SecretCore` 呼出し前にエラーとする。
- bundle の署名検証失敗または復号失敗時は `k-device` や設定値を保存しない。
- previous key 自動削除結果は UI に出さず、監査ログのみに残す。
- 高リスクワークフロー失敗時は `failed` を返し、TS 側は内部手順を再構成せずエラー表示と再試行要求のみに留める。
- [禁止] `POST /api/pairing/bundle` のような `createPairingBundle` 直公開 API を提供しない。
- `fileSync` の `SHA-256` 不一致時は新ファイルへ切り替えず、旧ファイルを維持する。
- `fileSync` の途中失敗時は `/logs` を含む更新対象外ファイルを保持する。
- 証明書更新失敗時は旧証明書を維持し、接続再開時も前回有効証明書を継続利用する。
- ログ削除条件に未達の場合は、ログ整理のために画像や証明書を削除しない。
- 複数 `Server` 同時稼働時も、OTA は要求元 `Server` に対してのみ結果を返す。
- `k-user` の復号済み平文は UI や通常 API に出さない。
- `WiFi AP` はシステム全体では複数存在してよいが、各 `ESP32` の現行運用は固定1件とする。
- パスワード/認証情報の変更 API 失敗時は旧値を維持し、監査ログへ失敗理由を残す。
- 例外継続（固定値運用）登録時に `reason` / `scope` / `expiresAt` のいずれかが欠落している場合は登録を拒否する。

### 3.1 ESP側 MQTT TLS 再試行戦略
- [重要] `connectToWifiRouter()` は Wi-Fi 接続を 3 回まで試行し、1 回ごとに約 7 秒のポーリングを行う。
- [重要] `connectToWifiRouter()` は明確な失敗状態（`WL_CONNECT_FAILED` / `WL_NO_SSID_AVAIL`）を検知したら、同一 attempt 内の待機を打ち切って次の attempt へ移る。
- [重要] `connectToWifiRouter()` の attempt 間バックオフは 1200ms とする。
- [重要] `connectToMqttBroker()` は TLS / 認証 / broker 到達失敗時に 10 回まで再試行し、各再試行間隔は 200ms とする。
- [重要] `connectToMqttBroker()` の再試行は、`DNS失敗` / `証明書期限切れ` / `証明書不一致` / `TLSハンドシェイク失敗` / `認証失敗` / `Broker到達不可` / `タイムアウト` を同一系列の失敗として扱い、各失敗を `ERR111`〜`ERR117` 相当の要約へ集約する。
- [厳守] 失敗時に TLS を無効化したり、平文 MQTT へ自動降格したり、証明書検証を省略したりしない。
- [厳守] Wi-Fi が切断された場合は MQTT 単独の再試行を続けず、Wi-Fi 再接続完了後に MQTT 再接続をやり直す。
- [重要] 10 回の再試行を超えた場合は即時の高頻度ループを止め、低頻度の再接続待機へ移行する。
- [推奨] 低頻度待機へ移行した際は、管理画面へ `接続エラー原因` と最終失敗時刻を表示できるよう、直近の失敗要約を保持する。
- [厳守] MQTT 認証失敗時は認証情報を推測で再設定せず、管理画面側の状態表示とエラー原因表示を優先する。
- [厳守] Wi-Fi が切れた場合は MQTT だけを再試行せず、Wi-Fi -> MQTT -> ReConnect publish の順で再接続チェーンを組み直す。
- [重要] `mainTaskEntry()` は MQTT エラー検知時に再接続起点を戻し、再接続チェーンの再起動は少なくとも 5000ms 間隔で実施する。
- [重要] エラー要約には `DNS失敗`、`証明書不一致`、`認証失敗`、`Broker到達不可`、`タイムアウト` のような切り分け可能なラベルを残す。
- [禁止] 再試行戦略の都合で秘密値をログへ出力しない。

## 4. AP 共通トップ画面
- [厳守] AP 名は `AP-esp32lab-<MAC(no colon)>` とする。
- [厳守] AP 名は顧客向けメンテナンスと ProductionTool で共通化する。

- [重要] ProductionTool 追加認証成功後 10 分間は再入力不要とし、10 分経過後は高権限操作直前に再認証する。
- [重要] 共通トップ画面の表示項目は以下とする。

  - `public_id`
  - firmware 版数

  - eFuse 実施済 / 未実施

## 4.1 環境センサー運用
- [重要] `BME280` は `I2C` モードで使用し、`SCL/SDA` は既存 `LCD` と共有する。
- [厳守] `BME280` の `CSB` は 3.3V 固定とし、`SPI` モードへ入れない。
- [重要] `SDO` を `GND` に接続した場合は `0x76`、3.3V に接続した場合は `0x77` として扱う。
- [推奨] 初回配線後は起動ログの `I2C scan` と `detectBme280Address` 出力でアドレス確認を行う。
- [禁止] `LCD` と `BME280` へ別タスクから直接 `Wire` アクセスし、同時送信競合を発生させない。

| F3-114 | `GET /api/workflows/{workflowId}` は `get_workflow_status` を通じて workflow の `queued/running/waiting_device/verifying/completed/failed` を返す。 |

- [厳守] `wrapped_k_user.bin` が存在する場合は `s_random → HKDF` 導出を使用しない。インポート済みが優先。
- [厳守] k-user の平文をコンソール・ログ・ファイルに出力しない。fingerprint のみ表示可。
- [推奨] マイグレーション実行前に LocalServer を停止し、SecretCore のみ起動した状態で行う。
- [厳守] バックアップファイルは `scrypt + AES-256-GCM` を使用し、平文 `k-user` を保存しない。
- [禁止] バックアップ処理で raw `k-user` を API 応答へ返却しない。
- [厳守] `SecretCoreIpcClient` は low-level 実装に限定し、業務コードは `SecretCoreFacade` を経由する。
- [厳守] 通常運用の IPC は保護モードを使用し、互換の平文IPCは手動起動時の限定保守用途にとどめる。
- [重要] Stage1 の Rust MQTT 化は command publish のみであり、subscribe/受信解析/OTA HTTPS 配布はまだ TS 側担当である。

## 5. `k-user` 再発行の逐次再ペアリング
### 5.1 正常系
1. 管理者が `LocalServer` で `k-user` 再発行を開始する。
2. TS が管理者認証、対象確認、影響確認を行う。
3. TS は `runKeyRotationSession()` を `SecretCore` へ要求する。
4. `SecretCore` が新 `k-user` 系統と新 `keyVersion` を生成し、進捗を `queued` / `running` として返す。
5. 作業者は対象個体を 1 台ずつ AP モードへ入れる。
6. `SecretCore` は `waiting_device` 状態で対ESP32通信開始を待つ。
7. `SecretCore` が必要に応じて bundle 生成、送達、署名検証、復号確認を実行する。
8. ESP32 は `current/previous` キーと設定値を NVS に更新する。
9. `SecretCore` は `verifying` を経て `completed` または `failed` を返す。
10. TS は当該個体を「切替済」または「失敗」に更新する。

### 5.2 UI 表示
- 旧鍵猶予中
- 未切替台数
- 個体ごとの切替済 / 未切替
- 旧鍵高権限可 / 不可
- current `keyVersion`
- previous key 状態
- previous key 保持残り時間
- 最終エラー理由

## 6. `k-device` 初回投入
### 6.1 正常系
1. 新規個体を AP モードへ入れる。
2. 作業 PC が `esplab-ap-<shortId6>` に接続する。
3. 共通トップ画面パスワード認証を行う。
4. TS が必須 `requestedSettings` を検証する。
5. TS は `runPairingSession()` を `SecretCore` へ要求する。
6. `SecretCore` は `queued` / `running` を返しつつ、`public_id`、`k-device`、`keyVersion`、Wi-Fi / MQTT / OTA / 認証情報を含む bundle を生成する。
7. `SecretCore` は対ESP32通信を継続し、署名検証、復号、保存完了を確認する。
8. `SecretCore` は `verifying` を経て `completed` または `failed` を返す。
9. 保存完了後、通常運用へ移行する。

### 6.2 UI 表示
- `public_id`（未発行時は `pending`）
- セキュア状態
- シリアル状態
- 入力設定チェック結果
- `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed`
- エラー理由
- 監査記録済み表示

## 7. LocalServer workflow API
- [厳守] `LocalServer` が外部公開する高リスク処理 API は以下の workflow 系に限定する。
- [進捗][2026-03-16] `POST /api/workflows/pairing/start`
  - 用途: `runPairingSession()` 開始要求
  - TS 側の REST 窓口、`requestedSettings` 事前検証、`ap/configure` 系入力からの変換 helper は実装済み
  - Rust 側は `createPairingBundle` 相当の内部生成、AP モード到達・認証 precheck、ESP32 `GET /api/pairing/state` 参照、`POST /api/pairing/session`、`POST /api/pairing/bundle-summary`、`POST /api/pairing/transport-session`、`POST /api/pairing/transport-handshake`、`POST /api/pairing/secure-bundle` まで実装済み
  - workflow は `queued -> running -> waiting_device -> verifying -> completed(OK)` へ到達し、AP 側 `state=applied` と `savedCurrentKeyVersion` を照合する
- [進捗][2026-03-21] `POST /api/workflows/key-rotation/start`
  - 用途: `runKeyRotationSession()` 開始要求
  - TS 側の REST 窓口、Pairing 互換入力の正規化、新 `k-user` 発行、再導出 `k-device` の request 反映、IPC 受理、workflow 状態取得までは実装済み
  - Rust 側は Pairing と同じ secure bundle 送達・AP `state=applied` 完了判定経路を再利用する。未実施なのは 7040 実試験と複数台切替時の運用確認である
- [進捗][2026-03-21] `POST /api/workflows/production/start`
  - 用途: `runProductionSecureFlow()` 開始要求
  - [進捗][2026-03-22] TS 側の REST 窓口、`dryRun` 必須の最小入力検証、`stepPlan` / `expectedMac` / `expectedFirmwareVersion` / `minimumFreeHeapBytes` / `minimumStackMarginBytes` / `precheckSnapshot` / AP 接続情報の検証、IPC 受理、workflow 状態取得までは実装済み
  - [進捗][2026-03-22] Rust 側は dry-run 限定で `queued -> running(precheck) -> waiting_device(lock/stage_prepare) -> verifying(readback/final_judgement) -> completed(OK)` の骨格を返せる。`precheckSnapshot` に明示 NG がある場合と `dryRun=false` の場合は `failed` で停止する
  - [進捗][2026-03-22] AP 接続情報が与えられた場合は ESP32 AP `POST /api/production/precheck` にログイン付きで到達し、firmware/MAC/free heap/stack margin の観測値を workflow precheck へ取り込める
  - [重要] 不可逆処理本体、実機読戻し、隔離判定の最終化、再開判定は継続実装とする
- `POST /api/workflows/signed-ota/start`
  - 用途: `runSignedOtaCommand()` 開始要求
- `GET /api/workflows/{workflowId}`
  - 用途: 進捗状態と結果取得
- [厳守] 応答項目は `workflowId`、`state`、`result`、`errorSummary` を基本とし、raw key や中間秘密を含めない。
- [禁止] TS が `createPairingBundle`、署名、暗号化、対ESP32送達を個別 API で逐次制御する構成を採用しない。
- [重要][2026-03-21] 現在の end-to-end 完了確認済み API は `POST /api/workflows/signed-ota/start`、`POST /api/workflows/pairing/start`、`GET /api/workflows/{workflowId}` である。`key-rotation` は同等の実装経路まで到達したが実試験未了、`production` は開始 API・IPC・状態取得の骨格まで実装済みで本体は継続実装である。
