# IoT 鍵一覧仕様書

[重要] 本書は `IoT` プロジェクトで扱う鍵、鍵関連保存物、鍵関連識別子を一覧化し、用途・保存場所・生成タイミングを明確化する。  
理由: `k-user / k-device` 系と `k-iot` 系の役割混同を防ぎ、実キー生成時の事故を防止するため。

[厳守] 本書には実キー値・実パスワード・実証明書秘密鍵を記載しない。  
理由: 文書経由の機密漏えい防止のため。

[禁止] 実キー値を `IoT/ESP32/header/sensitiveData.h` や公開リポジトリへ記載しない。  
理由: セキュアブート鍵、Flash Encryption鍵、OTA署名鍵の漏えいは復旧コストが極めて高いため。

## 1. 命名方針
- `k-user` / `k-device`
  - LocalServer 側の通信・アプリ層暗号用途
- `k-iot`
  - ESP32 セキュア化系の総称名
- `k-iot-***`
  - ESP32 セキュア化系の用途別論理名
- [厳守] `k-user` 系と `k-iot` 系は完全分離する。

## 2. LocalServer 系鍵一覧
| 鍵名/保存物 | 目的 | キー値の保存場所 | 生成タイミング | 備考 |
|------|------|------|------|------|
| `S_random` | `k-user` 導出元のランダム秘密 | 平文保存しない。TPM 復号後のメモリ内のみ | LocalServer 初回セットアップ時 | [厳守] ディスク保存禁止 |
| `wrapped_secret` | `S_random` の TPM 保護保存物 | LocalServer 専用管理領域ファイル | 初回セットアップ時 | 単体では別PC利用不可 |
| `k-user` | ユーザー環境ルート鍵 | 平文保存しない。起動時メモリ内のみ | LocalServer 起動時 / 再発行時 | `S_random` + `S_app` から HKDF 生成 |
| `k-user-backup.enc` | `k-user` 復旧用の暗号化バックアップ保存物 | LocalServer 専用管理領域ファイルまたは安全媒体 | バックアップ作成時 | [厳守] パスワードと組み合わせて `SecretCore` 内でのみ復号 |
| `k-device` | デバイス単位通信鍵 | LocalServer 側: 平文保存しない。ESP32 側: NVS の current/previous スロット | デバイス登録時 / 通信開始時 / 再ペアリング時 | `k-user` + `base_mac` から HKDF 生成。`Server` ごと分離/共通を選択可 |
| `k-pairing-session` | AP モード中の pairing bundle 暗号化用セッション鍵 | 永続保存しない。`SecretCore` と ESP32 のセッション内のみ | AP モード接続ごとに ECDH で都度生成 | [厳守] TS へ返却しない |
| `ap-common-password` | AP 共通トップ画面の閲覧認証 | 現在は全デバイス共通固定値。将来変更可能な設定値 | 初期導入時 | [重要] ProductionTool 追加認証とは別 |
| `maker-mode-password` | ProductionTool 用高権限操作の追加認証 | リポジトリ保存しない。運用安全領域で管理 | 初期導入時 / 更新時 | [厳守] AP 共通パスワードと分離 |

| `k-iot-flash-encryption` | Flash Encryption によるファームウェア格納保護 | 生成元保管: オフライン暗号化媒体。デバイス側: eFuse / OTP 領域 | 試験用: リハーサル直前。**当面** 本番も同一バイナリを用いる（`todo_old20260429.md`）。本番専用は **`020-0001`** 後 | [厳守] 実値を `sensitiveData.h` へ書かない |

| `k-iot-pairing-verify` | pairing bundle 署名検証用固定公開鍵 | ESP32 ファーム内の検証用領域 | Pairing 仕様確定後、出荷ファーム生成時 | [重要] 秘密鍵ではなく公開鍵のみ ESP32 へ保持 |

## 5. 実キー作成タイミング
### 5.1 今すぐ作らない鍵
- `k-iot-secure-boot`
- `k-iot-flash-encryption`
- `k-iot-ota-sign`

[重要] 上記は現時点では論理名のみ確定し、実キーはまだ作成しない。  
理由: ESP-IDF の適用手順、eFuse 書込み順序、OTA署名検証方式を確定する前に実キーを作ると、再生成や手順やり直しのリスクが高いため。

### 5.2 試験用キーを作るタイミング
- [重要] `k-iot-secure-boot` 試験用: Secure Boot v2 の試験手順と焼込みリハーサルを開始する直前
- [重要] `k-iot-flash-encryption` 試験用: Flash Encryption の試験手順と焼込みリハーサルを開始する直前
- [推奨] `k-iot-ota-sign` 試験用: OTA署名検証の実装・E2E 試験着手直前

### 5.3 本番用キーを作るタイミング
- [重要][2026-04-29] **`k-iot-secure-boot` / `k-iot-flash-encryption` は当面、試験用と同一素材を本番にも用いる**（`IoT/todo_old20260429.md`）。次に **別ファイルの本番専用**が必要になったときは `todo.md` **`020-0001`** を実施し、`コマンド仕様書.md` **§10.2.5a** に従う。
- [厳守] **`020-0001` 実施後**の `k-iot-secure-boot` 本番専用: `ProductionTool` で本番機へ eFuse 書込みを行う直前
- [厳守] **`020-0001` 実施後**の `k-iot-flash-encryption` 本番専用: `ProductionTool` で本番機へ Flash Encryption を有効化する直前
- [重要] `k-iot-ota-sign` 本番用: OTA を外部ネットワークや運用環境で正式運用開始する直前

### 5.4 生成前に必須の確認
- `コマンド仕様書.md` に生成手順が記載されていること
- `試験仕様書.md` に試験用キーでのリハーサル試験が記載されていること
- `問題点記録書.md` に既知制約が残っていないこと
- eFuse 書込み順序とロールバック不可条件を理解していること
- 鍵保管場所とバックアップ方針が決まっていること

## 6. 保存場所ルール
- [厳守] `k-user` / `k-device` の平文値はリポジトリへ保存しない。
- [厳守] ESP32 側 `k-device` は NVS に current/previous の 2 スロットで保持し、平文ファイルへ保存しない。
- [厳守] `k-user-backup.enc` は平文鍵ではなく、`SecretCore` 内でのみ復号可能な暗号化保存物として扱う。
- [厳守] `k-user` の保護方式は `TPM由来方式` と `暗号化ファイル + パスワード方式` を許容し、選択状態をメタ情報で保持する。
- [厳守] `k-iot-secure-boot` の秘密鍵はオフライン署名PCまたはHSMへ保存し、ESP32 ソースへ保存しない。
- [厳守] `k-iot-flash-encryption` の実値は公開文書、公開リポジトリ、`sensitiveData.h` に記載しない。
- [厳守] OTA HTTPS 配布用証明書秘密鍵は LocalServer の配布ディレクトリと分離し、アクセス権を限定する。
- [厳守][2026-04-29] **原則**として試験用キーと本番用キーは分離する。**例外（当面）**: `k-iot-secure-boot` / `k-iot-flash-encryption` は **`todo_old20260429.md`** の方針により**同一素材**で試験・本番の両方に用いる。分離が必要になったら **`020-0001`** を実施する。
- [推奨] **本番専用**秘密鍵（`020-0001` 実施後）は暗号化済み外部媒体へ二重保管し、保管責任者を文書で明確化する。

## 6.2 `ProductionTool` での取扱い
- [重要] eFuse / Secure Boot / Flash Encryption の最終有効化は、顧客向け通常モードではなく `ProductionTool` でのみ実施する。
- [厳守] `k-iot-secure-boot` の秘密鍵はオフライン管理を維持し、`ProductionTool` が直接保持するのは署名済み成果物または公開鍵ダイジェストまでとする。
- [推奨] `k-iot-flash-encryption` は個体ごとに生成し、可能であれば ESP32 内部の専用セキュア化処理で eFuse へ書込み、外部へ再出力しない。
- [厳守] ESP32 内部処理を採用する場合でも、通常アプリとは別の専用セキュア化フローとして実装し、通常OTAや通常MQTT経路からは呼び出さない。
- [厳守] 中断後は「全部焼けた前提」で扱わず、eFuse 読戻し結果と製造ログで状態判定する。
- [補足] 以前のシリアル焼込みで中途半端状態が発生した経験を踏まえ、完了まで一気に進める設計よりも、再実行判定可能な段階設計を優先する。

## 6.5 鍵の責任範囲
- [厳守] `ProductionTool` は `k-iot-secure-boot`、`k-iot-flash-encryption`、製造用成果物の投入責務のみを持ち、通常運用の `k-user` / `k-device` 管理主体にならない。
- [厳守] `LocalServer` は鍵メタ情報、状態表示、操作要求のみを扱い、raw key を保持しない。
- [厳守] `SecretCore` は `k-user` / `k-device` 導出、`k-user-backup.enc` の復旧、`runPairingSession()` 内部の `createPairingBundle` 生成、署名、暗号化の主体であり、raw key を TypeScript へ返さない。
- [厳守] `LocalServer` が外部公開する高リスク処理は `/api/workflows/...` 系に限定し、`createPairingBundle` を直公開しない。
- [厳守] `ESP32` は `k-device` current/previous と接続設定を NVS に保持し、未検証 bundle を受理しない。
- [重要] `ESP32` は将来の複数 `Server` 同時接続を考慮し、`serverId` ごとに `brokerId`、`wifiApId`、`kDeviceProfileId` を区別できるようにする。
- [厳守] 詳細な責任境界は `モジュール仕様書.md` を親定義とする。

## 6.3 `k-iot-flash-encryption` の運用整理
- [重要] `k-iot-flash-encryption` は初回書込みまたは初回セキュア化時に個体 eFuse へ設定する。
- [厳守] OTA のたびに `k-iot-flash-encryption` を再投入しない。
- [補足] 初回設定後は ESP32 個体内の Flash Encryption 機構が内部利用するため、`LocalServer` や通常 OTA 配布側は保持しない。
- [将来対応] 再製造、個体再生、重大障害時の再評価が必要な場合のみ運用見直しを行う。

## 6.1 OTA配布ファイル保護方針
- [重要] 当面は `firmware.bin` 自体の追加暗号化は採用しない。
- [厳守] 代わりに HTTPS/TLS、証明書検証、Secure Boot、Flash Encryption、OTA時再認証で保護する。
- [将来対応] 顧客配布時に `.bin` 流出後の可読性低減要求が強くなった場合は、追加暗号化を別途評価する。
- [補足] 顧客配布を行う場合でも、追加暗号化だけで完全秘匿はできないため、費用対効果評価を必須とする。

## 6.4 AP / Pairing 用認証素材
- [厳守] AP 共通トップ画面パスワードは現時点では全デバイス共通固定値とするが、将来変更可能な設定値として拡張余地を残す。
- [厳守] ProductionTool 用追加認証は `maker-mode-password` で実施し、AP 共通パスワードと分離する。
- [厳守] pairing bundle の暗号化は `k-pairing-session` を使い、TS へ raw secret を返さない。
- [厳守] ESP32 は `k-iot-pairing-verify` で bundle 署名を検証する。

## 7. `sensitiveData.h` の扱い
- [重要] `IoT/ESP32/header/sensitiveData.h` へは、論理名、キーID、プレースホルダのみを記載する。
- [禁止] 実運用の Secure Boot 秘密鍵、Flash Encryption 実値、OTA署名秘密鍵を記載しない。
- [将来対応] `sensitiveData.h` のプレースホルダは、秘密保管先の参照名またはビルド時注入方式へ置換する。
