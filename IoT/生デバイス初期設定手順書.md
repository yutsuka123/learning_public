# IoT 生デバイス初期設定手順書

[最重要] 本書は「生デバイスを 0 から設定し、`final secure` 適用直前まで安全に到達させる」ための固定手順を定義する。  
理由: 設定未投入（特に `k-device`）のまま進めると MQTT 暗号化通信が成立せず、誤った切り分けや文鎮化リスク増大につながるため。

[厳守] 本書の対象は **不可逆工程前**（`007-B` 進入前）までとする。  
理由: `eFuse` 書込みなどの不可逆工程は別ゲート（`7099` / `7101` / `PC-012`〜`PC-014` / `7095`）を満たしてから実施するため。

[禁止] `Wi-Fi` / `MQTT` / `NTP` / `k-device` 未確認のまま `esp32s3_secure_final` を先に適用しない。  
理由: 起動停止時の回復経路が狭まり、原因切り分けが難化するため。

## 1. 対象と完了条件
- 対象: `DEV-002` などの生デバイス（初期個体）。
- ゴール: `esp32s3_secure` で `AP` / `STA` / `NTP` / `MQTT` / `OTA` の事前確認を完了し、`final secure` 採用可否を判定可能な状態にする。
- 非対象: `007-B` 不可逆工程（`Flash Encryption` / `Secure Boot` / 封鎖）。

## 3. 運用フロー（ProductionTool 主、LocalServer 常駐）
[最重要][2026-04-05] 本手順では、操作主体を `ProductionTool` とし、`LocalServer` は監視・鍵発行・AP設定投入の基盤として常駐させる。  
理由: 「手動コマンド主体」だと担当者ごとに順序がぶれやすく、`k-device` 未投入や証跡漏れが発生しやすいため。

### 3.1 起動順（固定）
1. `LocalServer` を起動し、管理APIが応答することを確認する。
2. `SecretCore` を既定 Pipe `\\.\pipe\iot-secret-core-ipc` で起動する。
3. `ProductionTool` を起動し、対象機選択・dry-run 可能状態まで進める。
4. その後に `DEV-002` へ `esp32s3_secure` 書込みと設定投入を行う。

### 3.2 役割分担（固定）
- `ProductionTool`: 対象機識別、dry-run、作業実行の主導、監査ログ出力。
- `LocalServer`: AP探索、`k-device` 発行、`api/admin/ap/configure`、オンライン状態監視。
- `SecretCore`: 鍵処理（`k-user` / `k-device`）と IPC 保護。

### 3.3 フェーズ進行フロー（0 -> final secure 前）
1. 開始前チェック（識別情報、設定方針、記録先準備）
2. `esp32s3_secure` 書込み（不可逆なし）
3. AP/ネットワーク/鍵投入（`k-device` 含む）
4. `7099`（AP/STA/OTA）実施
5. `7101`（ProductionTool 連携）実施
6. `final secure` 採否判定（条件充足時のみ）

## 4. 開始前チェック
- [厳守] `DEV-001` は解析保全対象として切り離し、今回の対象個体を `DEV-002` として明確化する。
- [厳守] `COM` 番号、`チップ実測MAC`、`deviceName`（起動後）を `デバイス管理表.md` へ先に記録する。
- [重要] `IoT/ESP32/.env` と `IoT/ESP32/header/sensitiveData.h` の設定方針一致（AP資格情報、`production-hmac-v1`）を確認する。
- [重要] `sensitiveData.json` に `wifi` / `mqtt` / `timeServer` / `credentials.keyDevice` の項目があることを確認する。

## 5. 0 からの設定手順（不可逆なし）
### 5.1 ファーム書込み（安全版）
1. `python -m platformio run -e esp32s3_secure -d IoT/ESP32`
2. `python -m platformio run -e esp32s3_secure -t upload --upload-port <COMx> -d IoT/ESP32`
3. シリアルで起動確認（`secureFinalBuild=0`、`setup completed` を確認）

### 5.2 設定投入（AP/ネットワーク/鍵）
1. まず AP 起動可否を確認（長押し3秒、`AP-esp32lab-...`）。
2. LocalServer 管理APIまたは管理画面で `k-device` を対象 `deviceName` 向けに発行する。
3. `api/admin/ap/configure`（または同等画面操作）で `Wi-Fi` / `MQTT` / `NTP` / `k-device` を投入する。
4. 再起動後シリアルで `NTP SYNC OK` と `MQTT` 接続系ログを確認する。

### 5.3 事前安定性確認（`7099`）
1. AP 5分観測（`ping 192.168.4.1`、`/api/health`）。
2. STA 5分観測（`onlineState=online` 維持）。
3. OTA 1回目、OTA 2回目（各回で起動復帰確認）。
4. 異常なしなら `試験記録書.md` `7099` 当日欄へ記録する。

### 5.4 連携確認（`7101`）
1. `LocalServer` / `SecretCore` / `ProductionTool`（既定Pipe）を起動。
2. dry-run を実施し、対象機一致・監査ログ出力・停止条件判定を確認。
3. `試験記録書.md` `7101` に証跡を記録。

## 6. 停止条件（即時）
- `k-device required but unavailable` が出る。
- `MQTT RECON FAIL` が継続し、`onlineState` が復帰しない。
- `AP` が起動しない、または `api/health` 到達不可。
- `DEV-002` の識別情報（`MAC` / `deviceName` / `COM`）が記録と一致しない。
- [厳守] 上記発生時は `final secure` へ進まず、`問題点記録書.md` に記録してから再開判断する。

## 7. `final secure` 進入判定（本書の終点）
- [重要] 次の全条件を満たす場合のみ `esp32s3_secure_final` 採用判定を行う。
  - `7099` がクローズ可能
  - `7101` が `OK`
  - `PC-012`〜`PC-014` が `OK`
  - `7095` 当日欄へ反映済み
- [禁止] 条件不足で「先に secure_final を焼いて様子を見る」を行わない。

## 10. `DEV-002` パーティション変更前の保全対象
- [最重要][2026-04-05] パーティション変更前に、`DEV-002` を「すぐ起動できる状態」へ戻せる材料を揃えてから作業する。
- [厳守] 次の項目を変更前に記録する。
  - `COM`
  - `チップ実測MAC`
  - `deviceName`
  - `public_id`
  - `FW版`
  - `Wi-Fi` / `MQTT` / `NTP` 設定投入済みであること
  - `k-device` 投入済みであること
  - `7099` / `7101` の直近結果
- [厳守] ローカル側の復帰材料（`LocalServer` 設定値、対象機向け設定値、必要バックアップ対象）を確認し、未整理なら先に `todo.md` `008-0050` を実施する。
- [禁止] 復帰条件が曖昧なままパーティション変更へ進まない。

### 10.1 変更前に固定する `DEV-002` ベースライン
- [重要][2026-04-05] 現時点の既知ベースラインは以下とする。
  - `COM=COM4`
  - `チップ実測MAC=80:B5:4E:F9:CE:04`
  - `AP表示MAC=04:CE:F9:4E:B5:80`
  - `deviceName=IoT_04CEF94EB580`
  - `public_id=""`（最新試験ログ上は未設定）
  - `FW版=1.1.0-beta.31`
  - `build env=esp32s3_secure`
  - `7099=OK`
  - `7101=最終再確認前`
- [厳守] 上記値はパーティション変更直前に再採取し、`デバイス管理表.md` と `試験記録書.md` 当日欄へ同日反映する。

### 10.2 主復帰材料（論理復元用）
- [最重要] 主復帰手段は「旧 raw NVS をそのまま書き戻すこと」ではなく、**新CSV前提で安全版FWを書き込み、設定値を再投入して起動を戻すこと**とする。
- [理由] パーティション変更後は `NVS` のサイズや後続領域配置が変わるため、旧 `0x9000/0x5000` 前提の raw イメージを新CSVへ主経路で書き戻すのは整合リスクがあるため。
- [厳守] 変更前に次の参照元が揃っていることを確認する。
  - `IoT/ESP32/data/sensitiveData.json`
    - `wifi` / `mqtt` / `timeServer` / `credentials.keyDevice` の復元元
  - `IoT/LocalServer/data/settings.json`
    - `otaFirmwareVersion` と運用設定の復元元
  - `IoT/LocalServer/data/securityState.json`
    - LocalServer 側の運用状態参照元
  - `IoT/LocalServer/data/secure-backups/nvs-secret-backup-20260325-113605.json`
    - `keyDevice` 系の参照元（機密値の文書転記は禁止）
  - `IoT/LocalServer/data/secure-backups/internal-settings-backup-20260326-101418/`
    - `settings.json` / `securityState.json` / `wrapped_k_user.bin` / `wrapped_secret.bin` の退避済み参照元
  - `IoT/ESP32/header/version.h`、`IoT/LocalServer/.env`、`IoT/LocalServer/data/settings.json`
    - 版数3点同期の確認対象
- [重要] `wrapped_k_user.bin` / `wrapped_secret.bin` は `DEV-002` 個体のパーティション変更そのものでは書き換えないが、`7101` 再確認まで戻すには Local 側復帰材料として保持が必要である。

### 10.3 念のため保存する補助バックアップ
- [重要] 旧レイアウトの raw バックアップは**念のため保存**として取得してよい。
- [厳守] ただし用途は「旧CSVへ完全に戻す場合の参考」または「差分解析」であり、新CSVへの主復旧手段にしない。
- [推奨] 変更前に以下を採取する。
  - 現行パーティションCSVのコピー（`ESP32/partitions/esp32s3_from_device_16MB.csv`）
  - 現行 `NVS raw` バックアップ（現行レイアウトの `0x9000` から `0x5000`）
  - 直前のシリアル起動ログ
  - `7099` / `7101` の直近成功証跡

### 10.4 パーティション変更後の標準復帰手順
1. `DEV-002` へ新パーティションCSV前提の `esp32s3_secure` を書き込む。
2. 必要なら `uploadfs` または AP 設定投入で `Wi-Fi` / `MQTT` / `NTP` / `k-device` を再投入する。
3. 再起動後、シリアルで `setup completed`、`NTP SYNC OK`、MQTT 接続系ログを確認する。
4. `onlineState=online` を確認し、`7099` の最小順（起動 -> AP -> STA -> OTA）で戻りを確認する。
5. 最後に `7101` を再確認し、`ProductionTool` dry-run と既定 Pipe 連携を戻す。
- [厳守] 変更後に `k-device required but unavailable` が出た場合は、まず `credentials.keyDevice` の復元を優先し、`7083` や `7101` を先に再開しない。

### 10.5 復帰完了条件
- [最重要] 「すぐ起動できる状態」に戻った判定は、少なくとも次を満たした場合のみ `OK` とする。
  - シリアルで `setup completed` を確認
  - `onlineState=online` を維持
  - AP で `ping 192.168.4.1` と `/api/health` が成功
  - OTA 後に `otaPhase=done` と `online` 復帰を確認
  - `7101` の dry-run / 連携確認が通る
- [禁止] 「APだけ見えた」「OTAだけ通った」の片側成立だけで復帰完了扱いにしない。
