# IoT セキュリティ方針（認証情報取扱い）

[重要] 本方針は認証情報の漏えい防止に加え、**ソースコードや実行物を別PCへ複製されても秘密なしでは不正利用できない設計**を最優先とする。  
理由: 本プロジェクトでは「鍵そのものの抜き取り」だけでなく、「TypeScript/実行物を複製し、別PCで判定ロジックを書き換えて悪用される」懸念が主要リスクであるため。

## 1. 定義
- 認証情報: 秘密鍵、トークン、証明書実体、パスワード、秘密ID、署名素材、`S_random`、`k-user`、`k-device`、`k-iot-***` 導出元秘密。
- 認証ロジック: 認証判定アルゴリズム、検証フロー、状態遷移制御、署名対象データ定義、鍵選択ロジック。
- 公開設定: タイムゾーン、表示設定、ポート番号、OTA配布元種別など、漏えいしても直接侵害へ繋がらない設定。
- 秘匿設定: `S_random`、`k-user`、MQTT/HTTPS認証情報、秘密鍵、証明書秘密鍵、`wrapped_secret` の保管領域情報。
- ローカル安全領域: Windows の TPM を主対象としたハードウェア保護領域、および `wrapped_secret` 専用保管領域。

## 2. 想定脅威
- [重要] 設定ファイルやソースコードが抜き取られる。
- [重要] TypeScript/JavaScript 実装が別PCへコピーされ、判定ロジックを書き換えた偽サーバーが作られる。
- [重要] 運用PCは現在1台前提だが、将来の再インストール/移設に備えてバックアップ・復元を許容する。
- [補足] 運用はオフライン/オンラインの両方を想定するため、外部KMS依存を前提にしない。
- [補足] 運用OSは現在Windows中心だが、将来macOS対応を想定する。

## 3. 基本方針
- [厳守] 認証情報ファイルはGitへ一切アップロードしない。
- [禁止] Pull Request、Issue、設計書、コメントへ認証実値を貼り付けない。
- [重要] セキュリティ境界は「TypeScriptコード」ではなく「秘密情報を持つ正規インスタンス」で定義する。
- [重要] 認証ロジックの秘匿を安全性の主軸にしない。理由: TS/JS/exe 化のみでは解析・改変耐性の本質解決にならないため。
- [厳守] LocalServerが正規サーバーであることの判定は、デバイス側で検証可能な暗号学的証拠（HMAC/署名/nonce/timestamp）に基づくこと。
- [厳守] 第3段階のMQTTはTLS + ログインID/パスワード認証を必須とする。
- [禁止] MQTTを平文通信のみで運用しない。
- [推奨] 認証ロジックは公開可能として分離し、機密値は外部注入する。

## 4. ローカルサーバー鍵管理方針
- [厳守] `S_random` は LocalServer セットアップ時に CSPRNG で生成する。
- [厳守] `wrapped_secret = TPM_Encrypt(S_random)` を保存し、`S_random` の平文は保存しない。
- [厳守] `k-user` は `S_random` から HKDF で起動時に再生成する。
- [厳守] `k-user` は `settings.json` / `.env` / ソースコード / ドキュメントへ平文保存しない。
- [厳守] `k-user` の保護方式は `TPM由来方式` と `暗号化ファイル + パスワード方式` を許容し、復号は `SecretCore` の Rust モジュール内でのみ行う。
- [重要] `TPM由来方式` を選択した場合でも、暗号化済みバックアップファイルによる復旧経路を持たせる。
- [重要] `wrapped_secret` を別PCへ単独コピーしても利用できない状態を基本とするが、復旧時は別途定義した暗号化バックアップファイルとパスワードを利用してよい。
- [推奨] LocalServer本体は `SecretCore` 抽象層経由で秘密へアクセスし、秘密処理実装をアプリ本体から分離する。

## 5. バックアップ・復元方針
- [重要] 同一PC再インストール時の復元を許容する。
- [厳守] バックアップ対象は `wrapped_secret`、暗号化済み `k-user` バックアップファイル、`device_db`、必要メタ情報とする。
- [禁止] `S_random` / `k-user` / `k-device` の平文エクスポート。
- [重要] `wrapped_secret` 単体は別PCでは復号できないが、暗号化済みバックアップファイル + パスワードによる復旧は許容する。
- [推奨] バックアップ実行時と復元実行時に監査ログを残す。

## 6. `k-device` / 派生鍵方針
- [厳守] `k-device` は `HKDF(ikm=k-user, salt=SHA256(mac_address), info="k-device-v1")` で生成する。
- [重要] `k-device` は LocalServer/`SecretCore` 側では必要時に導出し、平文ファイル保存しない。
- [重要] ESP32 側では current/previous の 2 スロット運用を前提に NVS へ保存する。
- [厳守] 初回投入と再ペアリングは `runPairingSession()` を正規経路とし、内部の `createPairingBundle` 処理で行い、TS へ raw `k-device` を返さない。
- [厳守] `k-device` は `Server` ごとに分離する運用と、複数 `Server` 間で共通利用する運用の両方を許容し、選択状態をメタ情報で保持する。
- [仕様変更] `public_id` の初期値は `IoT_<macアドレスからコロン除去>` を許容する。
- [厳守] ただし Wi-Fi ステーション名や通常 topic に base_mac を直接使用しない。
- [禁止] メーカー共通鍵（`K_master_global`）の導入。

## 7. ロジック改ざん・別PC複製への対策
- [重要] ソースコードや実行物が別PCへコピーされても、TPM で `wrapped_secret` を復号できなければ正規の `k-device` / HMAC / 署名を生成できない設計とする。
- [厳守] ESP32側は「LocalServerが送ったから正しい」とは判断せず、メッセージのHMAC/署名/nonce/timestampを必ず検証する。
- [厳守] OTA開始、重要設定更新、鍵操作などの高権限コマンドは、署名検証を通過しなければ受理しない。
- [厳守] 署名対象には少なくとも `command` / `target` / `timestamp` / `nonce` / `keyVersion` を含める。
- [重要] 複数 `Server` 同時稼働時でも、テレメトリや `status` は同一 `Broker` を subscribe する複数 `Server` が共有してよい。
- [厳守] OTA や高リスク要求は要求元 `Server` の情報に対してのみ応答し、同一種別の要求が複数到着した場合は早いもの順に処理する。
- [禁止] 「exe化したから安全」「難読化したから安全」という扱い。
- [推奨] 将来はコード署名・配布物ハッシュ検証・改ざん検知ログを追加する。

## 8. 実装ルール
- [厳守] 公開設定と秘匿設定を分離し、公開設定のみ `settings.json` 等へ保存する。
- [厳守] 機密は環境変数またはローカル安全領域から読み込む。
- [厳守] APメンテナンスモードのユーザー/パスワードは `IoT/LocalServer/.env`、`IoT/ESP32/.env`、`IoT/ESP32/header/sensitiveData.h` で管理し、ESP32側の `.env` と `sensitiveData.h` は同一情報を維持する。
- [厳守] LocalServer の「設定 -> 管理者画面」は `username` / `password` 認証を必須とし、未認証で鍵発行・AP一括設定・メンテナンス再起動を実行できないようにする。
- [厳守] サンプル設定は `*.example` のみを公開し、実値はローカル専用にする。
- [推奨] `.env.example.sample.txt` はダミー値のみを保持し、必須キーの説明コメントを十分に記載する。
- [新規利用禁止] ソースコードへの秘密値ハードコード。
- [禁止] 機密ファイル名を分かりにくく偽装して管理する運用。
- [厳守] TLS証明書検証を実施し、検証スキップをデフォルト化しない。
- [厳守] Wi-Fi設定payloadはAES-256-GCMで暗号化し、平文保存・平文配信を禁止する。
- [推奨] 認証・署名・鍵導出・SecretStore はモジュール分離し、将来 TS からネイティブ実装へ差し替え可能にする。
- [厳守] `k-iot-secure-boot` の秘密鍵、`k-iot-flash-encryption` の実値、`k-iot-ota-sign` の秘密鍵を `sensitiveData.h` や公開リポジトリへ記載しない。
- [厳守] `sensitiveData.h` には `k-iot` 系の論理名・プレースホルダ・参照名のみを置く。
- [厳守] 試験用キーと本番用キーを分離し、本番用秘密鍵はオフライン保管を基本とする。
- [厳守] 鍵情報全般、Wi-Fi / MQTT / HTTP / HTTPS / NTP / ユーザー名 / パスワード等の主要接続・認証情報は `NVS` を主保存先とする。
- [厳守] 画像、証明書、ログは `LittleFS` の `/images` `/certs` `/logs` に分離保存する。
- [厳守] MQTT / HTTPS 用証明書実体は `LittleFS` `/certs` に保存し、`NVS` には版数、SHA-256、アクティブ状態などのメタ情報のみ保持する。
- [禁止] 証明書や秘密鍵実体を、移行完了後も恒久的に FW へハードコードし続けない。
- [厳守] `LittleFS` 更新は `SHA-256` 必須、`tmp` 書込み、成功後 `rename` とし、失敗時は旧ファイルを保持する。
- [禁止] 画像更新や証明書更新時に `/logs` を巻き込んで削除しない。
- [厳守] `fileSync` と AP モード更新は認証後にのみ実行し、無認証のファイル差替経路を設けない。
- [厳守] `Server` と `MQTT Broker` は別責務として扱い、`serverId` / `brokerId` / `wifiApId` を混同しない。
- [厳守] システム全体では複数 `WiFi AP` を許容するが、各 `ESP32` の現行運用は固定1件とする。
- [厳守] AP 共通トップ画面には共通パスワード認証を要求し、Production 操作にはメーカーモード専用パスワードの追加認証を要求する。
- [厳守] AP 共通パスワードは現時点では全デバイス共通固定値とするが、将来変更可能な設計余地を残す。
- [厳守] `SecretCore` の `createPairingBundle` は `runPairingSession()` 内部 helper とし、`targetDeviceId` `sessionId` `keyVersion` `requestedSettings` を入力として bundle を生成する。
- [厳守] `requestedSettings` の必須項目（Wi-Fi / MQTT / OTA / 認証情報）は TS 側で事前検証し、不足時は `SecretCore` を呼び出さない。
- [厳守] pairing bundle は固定公開鍵による署名検証と、AP モード接続時 ECDH の都度セッション鍵で保護する。
- [厳守] `k-pairing-session` は永続保存せず、TS へ返却しない。
- [禁止] `POST /api/pairing/bundle` のような `createPairingBundle` 直公開 API を提供しない。
- [厳守] メーカーモード専用パスワードは通常管理者パスワードと分離し、リポジトリや仕様書へ実値を記載しない。
- [禁止] 隠しボタンのみで `Production` 入口へのアクセスを許可しない。
- [推奨] 隠しボタンは専用ログイン画面を開くための補助手段とし、認証・監査・操作確認は別途実施する。
- [厳守] 通常運用時のシリアル出力制約は FW 制御で行い、eFuse による不可逆停止は採用しない。
- [禁止] シリアル停止のみを目的に eFuse を書き込まない。理由: 不具合解析時に一時的なシリアル出力を許可する署名済み診断FWの適用余地を残すため。
- [重要] 量産通常FWではシリアル高詳細ログ、秘密値出力、危険な保守コマンドを停止または最小化する。
- [厳守] 診断用FWはメーカー管理下の署名済み配布物としてのみ使用し、顧客環境で恒久運用しない。
- [厳守] インターネット接続時も踏み台化防止のため、不要待受停止、未使用ポート閉鎖、管理ポート非公開、認証失敗抑止、監査ログ保存を最低限適用する。
- [禁止] 本プロジェクト要件外の汎用プロキシ、中継、ファイル共有、ポートフォワード機能を本番運用へ同居させない。

## 9. 管理者権限侵害の扱い
- [重要] 運用PCの管理者権限を奪われた場合、当該PC上での完全防御は困難であることを前提とする。
- [重要] 本件では「運用PC管理者責任」を一定範囲で許容する。
- [厳守] ただし、運用PC侵害と「ソース/設定の単純コピーによる別PC不正利用」は区別して対策する。
- [推奨] 当該PC侵害時の被害範囲を減らすため、鍵バージョン管理・失効手順・再ペアリング手順を用意する。

## 10. 監査ルール
- [厳守] コミット前に機密ファイル混入チェックを実施する。
- [推奨] `IoT/.gitignore` を定期見直しし、新規機密種別を反映する。
- [厳守] 少なくとも以下を監査ログへ残す。
- LocalServer初期鍵生成
- TPM による `wrapped_secret` 生成/復号
- バックアップ作成
- 復元実行
- OTA開始署名要求
- 鍵ローテーション
- 失効/再ペアリング
- AP 共通トップ画面認証成功/失敗
- `Production` ログイン
- `runPairingSession()` 要求
- `createPairingBundle` 内部生成失敗
- `runPairingSession()` 中の署名検証失敗 / 復号失敗 / NVS 保存失敗
- `fileSync` の `SHA-256` 不一致
- `/certs` 更新失敗による証明書切替中止
- ログ削除条件発火（30日超過 / 100KB超過）
- 48h 到達による旧鍵失効
- 240h 到達による previous key 自動削除
- セキュア化の各段階開始/完了/中断
- 隔離判定
- [003-0018] アンインストール時の消去対象・結果・日時（`安全消去ポリシー.md` 参照）

## 11. インシデント時対応
- [厳守] 認証情報流出疑い時は、直ちに無効化・ローテーションを実施する。
- [厳守] 別PC複製や偽サーバー疑い時は、対象デバイス側で `keyVersion` を更新し、旧鍵系統を失効する。
- [厳守] 原因、影響範囲、再発防止策を記録する。

## 11.5 アンインストール・廃止時の安全消去（003-0018）
- [厳守] LocalServer / Production / LocalSoft 廃止時は `安全消去ポリシー.md` に従い、機密データを上書き消去する。
- [禁止] 通常削除のみで機密残留を許容する運用。
- 理由: 論理削除では復元ツールで機密が読み出されるリスクがあるため。

## 12. 補足
- 認証ロジック自体の公開は可能とする。ただし、公開物に秘密値・秘密導出情報を含めない。
- [補足] TypeScript/Node.js を採用しても、本方針は維持可能である。安全性の主軸はコード秘匿ではなく、OS保護された秘密とデバイス側検証である。
