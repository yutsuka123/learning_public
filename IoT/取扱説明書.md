# IoT 取扱説明書

[重要] 本書は `IoT` システムの通常運用、復旧、再ペアリング、サポート依頼時の一次案内をまとめる。  
理由: 現場で「どの順で戻すか」「バックアップが無いときに何をしてよいか」を一冊で判断できるようにするため。

[厳守] 本書は運用手順の案内に限定し、秘密鍵・実パスワード・raw `k-device`・`wrapped_secret` の実値は記載しない。  
理由: 取扱説明書は広く参照されやすく、機密漏えい経路になりやすいため。

## 1. この説明書で扱う対象
- `LocalServer`
- `ESP32` の通常運用と AP 復旧
- `MQTT TLS` 接続
- `wrapped_secret` / `device_db` の復元
- バックアップ未保持時の再ペアリング
- サポートパッケージの生成依頼
- 出荷後のファームウェア更新の基本：**§2.1**（OTA を主、`ProductionTool` と社内手順、シリアルはオプション）
- MQTT 経由 OTA とネットワーク境界：**§2.2**（署名検証一式が揃うまでの留意）

## 2. 日常運用の基本
1. `LocalServer` を起動する。
2. MQTT ブローカと ESP32 の接続状態を確認する。
3. 管理画面で `TLS接続状態`、`認証状態`、`接続エラー原因` を確認する。
4. 異常があれば、まずネットワーク、次に認証、最後に証明書の順で切り分ける。

[厳守] 異常時に推測で設定を書き換えず、原因が分かるまで先にログと状態を確認する。  
理由: 誤った再設定は原因を隠し、復旧時間を長引かせるため。

### 2.1 出荷後のファームウェア更新（OTA を主とする）**[重要][`007-0013`]**

[厳守] 出荷後の**通常**のファームウェア更新は、MQTT 指令に基づく **HTTPS 経由の OTA（無線更新）** を正とする。  
理由: 更新の経路・版数管理・セキュリティ上の評価を運用側で一元化するため（詳細は `機能仕様書.md` §1.5）。

[推奨] 製造工程などで機器側のロックダウン（「閉じる」手順）を行うときは、**製造用の `ProductionTool`** と、社内で定めた手順書に従う。個人の読みだけで、eFuse やセキュアブート関連の項目を試さない。  
理由: 手順単位での監査と、対象個体の取り違え防止のため。

[重要] **UART／USB シリアル**を使った書込みは、**標準運用では使わず**、保守・開発・復旧などで**わざとシリアルを開いて**実施するときのみ、**オプション（例外運用）**として扱う。実施するときは次を認識すること。  
- 物理接続やケーブル経由の操作は、窃取・第三者による不正書込み・机上のセキュリティ前提の変更につながる。  
- OTA と別経路となるため、**誰が・いつ・どの版を入れたか**を台帳と突き合わせる。運用チームまたは担当者間で必要性を確認してから行う。

[厳守] シリアル経由の更新を、**恒久運用の「いつもの更新手段」として前提化しない**。必要なときだけオプションとして選び、その都度リスクと手順を共有する。

詳細仕様・禁止事項は **`機能仕様書.md` §1.5** を正とする。

### 2.2 MQTT 経路の OTA とネットワーク境界**[重要][`008-0016`]**

[重要] MQTT 指令により **HTTPS の firmware を適用する**運用において、`LocalServer`・MQTT ブローカ・ESP32 をつなぐ LAN は **信頼できるネットワーク**（管理上の権限がある利用者のみが操作でき、第三者がブローカに接続できる公開構成にしないなど）として扱う。  
理由: コマンド署名（HMAC 等）の装置側検証や関連インフラが `todo.md` の **`008-0006`〜`008-0009`** のとおり**本編で常時満たされるまで**には段階があり、その間は **不正 MQTT 指令**の混入リスクをネットワークと運用手順側で抑制するため。

[参照] 実装フェーズごとの詳細・試験観点は **`MQTTコマンド仕様書.md`** と **`機能仕様書.md`** を正とする。本項目はユーザー向けの最低限ルールのみ述べる。

## 3. バックアップがある場合の復元
### 3.1 復元前確認
- 対象個体の `public_id`、`deviceName`、`MAC`、`keyVersion` を確認する。
- 復元元が正しいバックアップであることを確認する。
- 復元対象に秘密値が混入していないことを確認する。

### 3.2 復元順序
1. `LocalServer` を停止する。
2. `wrapped_secret` を元の保管先へ戻す。
3. 必要に応じて `device_db` を戻す。
4. `LocalServer` を再起動する。
5. 起動後に鍵状態とデバイス一覧を確認する。

[重要] `wrapped_secret` は元の TPM 前提でのみ使える。別PCで無理に復元しようとしない。  
理由: 別PC復元は別経路の復旧設計に切り替えるべきであり、失敗を重ねても回復しないため。

## 4. バックアップが無い場合
[重要] `wrapped_secret` と `device_db` のバックアップが無い場合、旧鍵を無理に復元しない。  
理由: 不完全な復旧を続けると、接続不良と鍵不整合が重なり、かえって停止範囲が広がるため。

### 4.1 正規対応
1. 対象個体を AP 復旧モードへ入れる。
2. 既存の認証情報・鍵状態を前提にせず、再ペアリングを実施する。
3. `LocalServer` で対象個体を確認し、再ペアリング開始前に `public_id`、`deviceName`、`MAC`、`keyVersion` を記録する。
4. `runPairingSession()` 経路で新しい `k-user` / `k-device` 系統を投入する。
5. `device_db` を再構成し、管理台帳と試験記録の対象個体情報を更新する。
6. 再起動後に `TLS接続状態`、`認証状態`、`接続エラー原因` を確認し、通常運用へ戻せることを確認する。

### 4.2 実施時の注意
- 旧個体識別子と新個体識別子を混同しない。
- 旧パスワードの再利用を前提にしない。
- `wrapped_secret` の再取得や旧 `keyStore.json` の再利用を試みない。
- 再ペアリング後は古い `device_db` 断片を残さず、復旧に使った一時ファイルを安全消去する。
- 成功後は試験記録書と問題点記録書へ経緯を残す。
- 失敗した場合は、途中で保存した新しい設定をそのまま使い続けず、いったん AP 復旧モードへ戻して再実施する。

## 5. サポート依頼時の案内
- `サポートパッケージ仕様書.md` に従って、秘密を含まないログと状態情報を集める。
- 証跡は `LocalServer` の画面状態、接続エラーの要約、時刻、対象個体情報を優先する。
- パスワードや秘密鍵の手入力を依頼された場合は、まず正規の再ペアリング手順を確認する。

[禁止] サポート調査のために秘密鍵や実パスワードを共有する。  
理由: 調査用の一時共有が、そのまま恒久的な漏えい経路になるため。
