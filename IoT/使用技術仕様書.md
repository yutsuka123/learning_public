# IoT 使用技術仕様書

[重要] 本書は `IoT` プロジェクトの実装言語・主要技術を定義する。  
理由: 実装開始後の言語ブレを防ぎ、設計と実装の整合性を保つため。

## 1. 言語方針
- ESP:
  - [厳守] C/C++ を採用する。
  - [推奨] C++を利用する場合は可能な範囲で C++20 を採用する。
- Local:
  - [厳守] 現行は Windows + TPM 2.0 を前提とする。
  - [将来対応] macOS は同等のハードウェア拘束方式を別途定義する。
  - [厳守] `LocalServer` は TypeScript / Node.js を採用する。
  - [厳守] 管理画面はブラウザWeb UI構成とする。
  - [厳守] 秘密処理は `SecretCore`（Rust）へ分離し、`LocalServer` と責任境界を分ける。
- Production:
  - [厳守] `Production` はメーカー専用ツールとし、通常運用の `LocalServer` と混在させない。
  - [重要] `Production` は工場内の初回書込み、初回セキュア化、eFuse 実行、製造ログ保存を担当する。
- Cloud:
  - [将来対応] Python 等を候補とし、要件確定時に選定する。

## 2. 技術スタック（現時点）
- ESP:
  - PlatformIO
  - ESP-IDF
  - MQTT（TLS + ID/パスワード認証）
  - Wi-Fi設定暗号化（AES-256-GCM）
- Local:
  - mosquitto（ローカルブローカ）
  - TypeScript / Node.js（`LocalServer`）
  - REST API（ローカル管理API）
  - Web UI（ブラウザ）
  - OTA配布（HTTPS: manifest + firmware）
  - Task Scheduler（Windows自動起動）
  - [重要] 秘密処理は TypeScript本体から分離し、**Rust製 SecretCore** として別プロセス化する。
  - [厳守] SecretCore は現行では TPM を利用して `wrapped_secret` を管理する。
  - [厳守] TS ↔ SecretCore の層間通信は X25519 DH鍵交換 + AES-256-GCM 暗号化 + nonce認証で保護する。
  - [厳守] 高リスク処理は `SecretCore` の Rust 主導ワークフローで完結させる。
  - [禁止] SecretCore から `k-user` / `k-device` の raw value や中間秘密を返却する API を設けない。
  - [推奨] SecretCore は難読化ビルド（obfstr + LLVM O3 + strip + zeroize）を適用する。
  - 詳細は `LocalServer秘密処理別層仕様書.md` を参照。
- LocalSoft:
  - Python
  - SQLite（保持期間: 日単位、`0` は無期限）
- LocalSoftForCloud:
  - Python
  - クラウド双方向同期
  - 暗号化payload生成（AES-GCM, HMAC-SHA256）
- Production:
  - 製造用実行ツール
  - 署名済み成果物投入
  - eFuse / Secure Boot / Flash Encryption の最終有効化
  - 製造監査ログ
- Cloud:
  - 未確定（Python系を第一候補）
  - ユーザー単位ルート鍵保護、デバイス鍵導出、MQTT ACL制御

## 2.1 責任範囲
- `Production`: 製造、初回書込み、初回セキュア化、eFuse 実行、製造監査。
- `LocalServer`: 通常運用 UI、REST API、入力検証、一覧表示、進捗表示、通常監視。
- `SecretCore`: TPM 拘束秘密管理、`k-user` 再生成、`k-device` 導出、高リスクワークフロー実行、署名、内部成果物生成。
- `LocalSoft`: MQTT 通知保存、SQLite 保持、履歴参照。
- `LocalSoftForCloud`: クラウド双方向同期。
- `ESP32`: AP モード、bundle 検証/復号、NVS 保存、MQTT/OTA 実行、鍵状態管理。
- [厳守] 詳細な責任境界は `モジュール仕様書.md` を親定義とする。

## 2.2 Rust 主導ワークフロー
- [厳守] `SecretCore` が提供する高リスクワークフローは `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` を基本とする。
- [厳守] `LocalServer` が受け取る進捗状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` に限定する。
- [禁止] TS 側で `runPairingSession()` 内部の `createPairingBundle`、署名、検証、対ESP32通信手順を逐次組み立てて高リスク処理を完了させる実装。

## 3. C++20運用方針（ESP）
- [推奨] `platformio.ini` のビルド設定で C++20 を優先する。
- [禁止] C++20が利用可能な箇所で、互換性理由なく旧標準へ戻す運用。
- [厳守] 依存ライブラリ制約でC++20が利用できない場合は、理由を設計書へ記録する。

## 4. セキュリティ連携要件
- [厳守] 言語選定に関わらず、認証情報はGitへアップロードしない。
- [厳守] 認証ロジック公開時は秘密値を分離する。
- [禁止] サンプルコードへ認証実値を直書きしない。
- [厳守] base_macの直接公開を避け、public_id運用を採用する。
- [厳守] Wi-Fi設定はアプリ層暗号（AES-256-GCM）で送受信する。
- [厳守] LocalServer本体は `wrapped_secret` 以外の秘密を直接ファイル管理せず、SecretCore 層を通じて利用する。
- [厳守] 秘密処理別層との通信は、ネットワーク公開しないローカルIPC（Windows: Named Pipe / macOS: Unix Domain Socket）を基本とする。
- [禁止] SecretStore層から `k-user` や `k-device` の raw value を返却するAPIを設けない。
- [推奨] SecretStore層は「署名」「検証」「派生鍵生成」「バックアップ復元」の限定APIだけを公開する。

## 6. 決定事項（2026-03-03）
- [重要] 主対応OSはWindows、副対応OSはmacOS。
- [厳守] ローカルMQTT BrokerはMosquitto。
- [厳守] OTAはHTTPS配布（manifest + firmware）。
- [重要] 同時接続台数は10台想定。
