# IoT OTA仕様書

[重要] 本書はESP32のOTA更新仕様を定義する。  
理由: MQTT指令、HTTPS取得、進捗通知、再試行、整合性確認を一貫した手順で実装するため。

[厳守] OTAは「指令はMQTT」「実ファイル取得はHTTPS（ESP32がpull）」で実施する。  
理由: サーバーからESP32への直接書換を避け、NAT配下でも安定運用するため。

[禁止] 検証省略（SHA256未検証、TLS検証無効）を本番運用に持ち込まない。  
理由: 改ざん・破損・なりすまし更新を防止するため。

## 1. OTA方式
### 1.1 方式①（主系）
- LocalServer（またはCloudServer）から MQTT で `otaStart` を指令する。
- ESP32は指令を受信後、指定URLへHTTPS接続してOTAを開始する。
- ESP32は分割書込みを行い、LCD表示とMQTT通知で進捗を報告する。

### 1.1.1 標準運用モデル
- [重要] OTAの標準運用は **メーカー署名 + 顧客実行** とする。
- 役割分担:
  - メーカー: `firmware.bin` の作成、署名、正式版管理
  - 顧客: メーカー署名済み配布物を自環境 LocalServer から実行
- [厳守] 顧客が実行できるのは、メーカー署名済み配布物のみとする。
- [禁止] 顧客が未署名ファームウェアや独自改変ファームウェアを標準運用で配布すること。
- 理由: 配布物の真正性担保をメーカー側に集約しつつ、現地更新の運用負荷を抑えるため。

### 1.1.2 `Production` との分離
- [厳守] eFuse / Secure Boot / Flash Encryption の最終有効化は、標準OTAではなくメーカー専用の `Production` フローでのみ実施する。
- [禁止] 顧客環境の通常OTA、通常Web UI、公開MQTT経路から eFuse 有効化を実行しない。
- [厳守] `LocalServer` は通常OTA実行主体とし、`Production` は製造時セキュア化主体として責任範囲を分離する。
- [推奨] `Production` の入口は、通常画面に常時表示しない専用認証導線または工場内専用画面とする。
- [厳守] 隠しボタンは補助手段であり、認証は通常の管理者パスワードと分離したメーカーモード専用パスワードで実施する。
- 理由: OTA と不可逆なセキュア化処理を混在させると、更新失敗時の切り分けと事故封じ込めが難しくなるため。

### 1.2 方式②（副系）
- ESP32 APモード（メンテナンスモード）での書換を提供する。
- 本書の詳細対象は方式①とする（方式②は別途拡張）。

## 2. 方式①の詳細仕様
### 2.1 MQTT指令（otaStart）
- トピック: `esp32lab/call/otaStart/<deviceName|all>`
- `args` 必須項目:
  - `firmwareVersion`（更新後バージョン）
  - `firmwareUrl`（`https://.../firmware.bin`）
  - `manifestUrl`（`https://.../manifest.json`）
  - `sha256`（配布前計算済み SHA256 16進文字列）
  - `chunkSize`（分割書込みサイズ。例: 4096）
  - `sameProgressRetryMax`（同一進捗率リトライ回数。固定値: `3`）
  - `sameProgressRetryIntervalSeconds`（同一進捗率リトライ間隔秒。固定値: `5`）
  - `fullRestartRetryMax`（最初から再試行回数。固定値: `3`）

### 2.1.1 OTA実行時の再認証
- [厳守] LocalServer の Web UI から OTA を開始する際は、管理者パスワードの再入力を要求する。
- [厳守] 再認証成功後は、同一ブラウザセッション内かつ 3時間以内に限り再入力を省略できる。
- [推奨] セッション管理はサーバー側で行い、Cookie は `HttpOnly` `Secure` `SameSite=Strict` を使用する。
- [重要] ブラウザ終了時の失効は「セッションCookieを永続化しない」ことで可能な範囲で実現する。
- 理由: OTA は高リスク操作であり、ローカル運用でも誤操作・なりすまし・離席時操作を抑止したいため。

### 2.2 HTTPS取得と分割書込み
- ESP32は `manifestUrl` を取得し、配布メタ情報を確認する。
- 続けて `firmwareUrl` をHTTPSで取得し、分割書込みする。
- 書込み中は受信済みバイト数から進捗率を算出する（0〜100%）。
- [推奨] 進捗通知は1%以上変化時、または5秒ごとに送信する。

### 2.3 進捗通知（LCD + MQTT）
- LCD表示: `OTA xx%` を常時更新する。
- MQTT通知:
  - トピック: `esp32lab/notice/otaProgress/<deviceName>`
  - 送信項目例:
    - `otaSessionId`
    - `firmwareVersion`
    - `progressPercent`（0〜100）
    - `phase`（`download` / `write` / `verify` / `done` / `error`）
    - `retryCountSameProgress`
    - `retryCountFullRestart`
    - `detail`
- LocalServer画面はデバイス別に最新進捗を表示する。

### 2.4 リトライ制御
- 同一進捗率で停止した場合:
  - 5秒おきに最大3回まで再試行する。
  - 3回失敗で「最初から再試行」へ移行する。
- 最初から再試行:
  - 最大3回まで実施する。
  - 3回失敗でOTA失敗として終了し、MQTTへNG通知する。

### 2.5 整合性確認（SHA256）
- サーバー側で配布前に `firmware.bin` の SHA256 を計算し、`otaStart` で送信する。
- ESP32は受信・書込み完了後に同じSHA256を計算し、指令値と一致比較する。
- 一致時のみ更新完了とし、次回起動パーティションを更新先へ切替える。
- 不一致時はNG終了し、更新確定を行わない。

### 2.6 デュアルパーティション運用
- ESP32は2面（A/B）構成で更新する。
- 非実行側へ書込み後、検証成功時のみ `boot` 対象を更新側へ切替える。
- 次回起動で更新側を起動し、起動成功後に確定通知を送る。
- [厳守] 連続OTA時に、書込み先パーティションが 0 / 1 で交互に切り替わることを試験で確認する。
- [厳守] OTA の途中失敗、整合性不一致、未確定起動失敗時は、旧パーティションから継続起動できることを試験で確認する。
- 理由: 更新失敗時でも現行運用面を保持し、現地復旧不能事故を防ぐため。

## 3. 完了・異常通知
- 完了時:
  - MQTTで `result=OK` と `firmwareVersion`、`sha256Verified=true` を通知する。
- 異常時:
  - MQTTで `result=NG`、`errorCode`、`detail`、`retryCount` を通知する。
- [推奨] 失敗要因は `networkTimeout` / `httpStatus` / `shaMismatch` / `writeError` などに分類する。

## 4. セキュリティ要件
- [厳守] HTTPS証明書検証を有効化する。
- [厳守] MQTT経路はTLS + ID/パスワード認証を必須とする。
- [厳守] 配布URLは短寿命トークン化または閉域配布を推奨する。
- [禁止] ファームウェア実体や署名鍵を公開領域へ平文配置しない。

### 4.1 証明書・鍵の必須要素
- [厳守] OTA HTTPS 配布サーバー証明書を用意する。
- [厳守] ESP32 側で検証する CA 証明書または同等の信頼設定を用意する。
- [厳守] Secure Boot 用署名鍵を用意する。
- [厳守] 標準運用のメーカー署名に必要な署名鍵を管理する。
- [推奨] OTA運用時の追加署名を採用する場合は `k-iot-ota-sign` を別鍵で用意する。

### 4.2 ファームウェア追加暗号化の採用方針
- [重要] 当面の推奨構成では、`firmware.bin` 自体の追加暗号化は採用しない。
- 理由:
  - HTTPS/TLS で通信路保護ができる。
  - Secure Boot により真正性を確保できる。
  - Flash Encryption によりデバイス保存後の読出し耐性を高められる。
  - 追加暗号化は鍵管理、復旧、切り分けを複雑化しやすい。
- [将来対応] 顧客配布で「配布サーバーから `.bin` が流出しても読みにくくしたい」要求が強くなった場合のみ、配布ファイル追加暗号化を再評価する。
- [補足] 顧客へ配布する以上、追加暗号化をしても完全秘匿は困難であり、費用対効果を都度評価する。

### 4.3 当面の推奨構成
- [厳守] OTA配布は HTTPS/TLS 必須とする。
- [厳守] ESP32 はサーバー証明書検証を実施する。
- [厳守] Secure Boot を有効化する。
- [推奨] Flash Encryption を有効化する。
- [厳守] OTA実行時は LocalServer 側で再認証する。
- [推奨] OTA開始コマンドは HMAC / 署名で保護する。
- [重要] 標準運用は「メーカー署名済み配布物を顧客の LocalServer が実行する」方式とする。

### 4.4 `Production` 時のセキュア化方針
- [重要] 過去のシリアル焼込みで中途半端状態が発生した経験を踏まえ、工場内セキュア化では ESP32 内部の専用処理を採用してよい。
- [厳守] ただし内部処理を採用する場合でも、通常アプリではなく「`Production` 専用セキュア化フロー」として実装する。
- [厳守] `Production` の UI/操作フローは、以下の段階で実行する。
  - 第1段階: `Production` 入口からメーカー専用ログイン画面を開く
  - 第2段階: メーカーモード専用パスワードで認証する
  - 第3段階: 対象機情報（シリアル、MAC、版数、未セキュア化状態）を確認する
  - 第4段階: 実行予定内容（Secure Boot、Flash Encryption、eFuse項目）を表示し、操作者確認を取る
  - 第5段階: 最終確認として対象機識別子や確認文言の再入力を要求する
  - 第6段階: セキュア化処理を段階実行し、各段階ごとに進捗・読戻し検証結果を表示する
  - 第7段階: 完了判定または隔離判定を表示し、製造ログへ記録する
- [厳守] 専用セキュア化フローは次を満たすこと。
  - 実行前の事前条件確認（対象機識別、電源安定、版数、鍵ID、未セキュア化状態）
  - 各段階ごとの eFuse 読戻し検証
  - 段階完了ログの LocalServer 保存
  - 中断後の再実行時に、読戻し結果から残工程のみ判定する設計
  - 中途半端状態の個体を出荷禁止として隔離判定できること
- [厳守] 本番個体へ適用する前に、同一手順・同一ファームウェア系統・同一鍵運用前提で試験機リハーサルを十分に完了していること。
- [厳守] ESP32 内部の専用セキュア化処理は、eFuse 書込み・readback・ログ出力を安全に実行できる専用タスクで実施し、十分なスタックサイズを確保すること。
- [推奨] セキュア化開始前に `uxTaskGetStackHighWaterMark` 相当の方法でスタック余裕を確認し、空きヒープや他タスク停止状態とあわせて開始可否判定に用いる。
- [禁止] スタック余裕未確認、ヒープ逼迫、通常通信タスク過負荷の状態で eFuse 処理を開始しない。
- [推奨] 段階間で「次へ」を手動確認させる方式、または危険段階のみ二重確認を要求する方式としてよい。
- [推奨] Secure Boot 署名鍵はオフライン管理し、`Production` が扱うのは署名済み成果物または公開鍵ダイジェストまでとする。
- [推奨] Flash Encryption 鍵はデバイス個体ごとに生成・書込みし、通常運用へ持ち出さない。
- [禁止] 「途中まで焼けたら必ず全部焼ける」と仮定した設計にしない。
- 理由: eFuse は不可逆であり、電源断・通信断・内部例外を完全には排除できないため、完了志向よりも読戻し検証と隔離判定を優先すべきである。

## 5. 変更履歴
- 2026-03-09: `LocalServer` を通常OTA主体、`Production` を製造時セキュア化主体として責任分離を明記し、旧製造用名称を `Production` ベースへ統一。理由: OTA と不可逆セキュア化の責任境界を文書上で明確にするため。
- 2026-03-08: OTA途中失敗時は旧パーティションから継続起動できることを要件へ追加。理由: 2面OTAの安全性確認で「交互切替」だけでなく「失敗時の旧面維持」を明示するため。
- 2026-03-08: eFuse 実行前の十分な試験機リハーサルと、専用処理タスクのスタックサイズ確認要件を追加。理由: eFuse 失敗防止には操作手順だけでなく実行時安定性確認も必要であるため。
- 2026-03-08: `Production` の隠し入口、専用パスワード、段階実行UIを追加。理由: 工場内専用機能の認証と誤操作防止フローを OTA 文書上でも明確化するため。
- 2026-03-08: `Production` と、ESP32内部専用セキュア化フローの条件を追加。理由: 工場内限定の eFuse 最終有効化を OTA 標準運用と分離し、過去の中途半端焼込み経験を再発防止へ反映するため。
- 2026-03-08: OTA標準運用を「メーカー署名 + 顧客実行」へ確定。理由: 顧客配布の可能性を踏まえつつ、真正性担保と運用負荷のバランスが取れる方式を正式採用するため。
- 2026-03-08: 顧客配布可能性を踏まえた当面の推奨構成を追加。理由: 追加暗号化の是非、OTA時再認証、証明書要件を実装前に固定するため。
- 2026-03-07: 新規作成。方式①（MQTT指令 + HTTPS OTA）の詳細仕様、リトライ制御、SHA256検証、2面切替を定義。
