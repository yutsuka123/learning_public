# IoT ROM容量分析資料

[最重要] 本書は `ESP32 esp32s3_secure` / `esp32s3_secure_final` の ROM 使用量を、**実測値・ファイル別・機能別**で分析した資料である。  
理由: 「何が重いのか」「本当に異常なのか」「どこから削るべきか」を次セッション以降も同じ根拠で判断できるようにするため。

[厳守] 本書の数値は 2026-04-05 時点の `esp32s3_secure_final` ビルド成果物（`firmware.bin` / `firmware.elf` / `firmware.map` / `.pio/build/.../src/*.o`）を基準とする。  
理由: 推測ではなく、実ビルドに基づく定量評価を残すため。

[重要] ファイル別・機能別の表は「リンク後寄与の近似把握」を目的として、**コンパイル済みオブジェクトサイズ**も併用する。  
理由: `firmware.map` 単独では機能群の俯瞰がしづらいため、相対比較に有効な指標を補助的に使う。

[禁止] 本書の表だけを見て即削除判断しない。  
理由: 大きいものの中には `TLS` や `OTA` のように削れない基盤機能も含まれるため。

## 1. 結論（先に要点）

- [結論1] ROM 使用率が高い最大要因は、**コード量の増加だけでなく、16MB実機に対してパーティション定義が実質4MBしか使っていないこと**である。
- [結論2] `secure_final` だけが異常に大きいわけではない。`esp32s3_secure` も `99.6%` であり、問題は既に開発安全版の時点から存在する。
- [結論3] アプリ側の主因は `MQTT系`、`AP HTTP/Pairing系`、`機密設定/NVS/証明書系`、`OTA系` の複合である。
- [結論4] 「機能のわりに多すぎる」のは半分正しい。**MQTT+TLS+OTA+AP WebServer+Pairing+JSON+LittleFS+証明書管理**まで同居させると重くなるのは妥当だが、現状のパーティション設定が厳しすぎて問題を増幅している。
- [結論5] したがって優先順位は、`(A) パーティション再設計` を先に検討し、同時に `(B) ログ・機能の減量` を安全側で進めるのが妥当である。

[重要] `firmware.elf` のトップシンボルを見ると、「単一巨大関数1つ」で膨らんでいるのではなく、**ログ整形・TLS・MQTT/OTA/AP処理**が複合して大きい。

## 6. 主因の分析

### 6.1 一番の主因

- [最重要] **パーティション定義が狭すぎること**が第一主因である。
- 16MB 実機なのに、現在の CSV は 4MB 分しか使っていない。
- そのため OTA 2面運用で各 app 面が 1.25MB に固定され、現在のファームでは余裕がほぼゼロになっている。

### 6.2 アプリ側の主因

- [重要] `MQTT系` が最大。単なる publish/subscribe だけでなく、暗号化 payload、status、network/set/get、JSON、HTTP連携を同居させている。
- [重要] `maintenanceApServer.cpp` が非常に大きい。AP認証、role管理、pairing、transport handshake、production precheck を1ファイルで抱えている。
- [重要] `sensitiveData.cpp` も大きい。NVS、LittleFS、証明書ファイル、JSON移行、ハッシュ計算まで担当しており責務が広い。
- [重要] `log.cpp` の増加は、詳細ログと `printf` 系整形関数の引き込みに直結している。

### 6.3 妥当かどうか

- [評価] `MQTT TLS + OTA + AP WebServer + Pairing + JSON + LittleFS + 証明書管理 + NVS` を1つのバイナリへ載せるなら、**1MB超級になること自体は不自然ではない**。
- [評価] ただし「現機能なら当然このサイズ」という意味ではない。`maintenanceApServer.cpp` と `MQTT/mqtt.cpp` に機能集中があり、整理余地は大きい。
- [評価] よって結論は **“完全に異常ではないが、現行構成は重すぎる。しかもパーティションがそれ以上に不利”** である。

## 7. 削れそうな箇所（優先度付き）

### 7.1 優先度A: パーティション

- [最優先] `app0/app1` を 16MB 実機に見合うサイズへ再設計する。
- [理由] ここを変えない限り、数十KB削ってもすぐ再び上限へ達する。
- [厳守] `DEV-002` だけで先行検証し、`DEV-001` には適用しない。

### 7.2 優先度B: ログ

- [有力] `CORE_DEBUG_LEVEL=5` のまま最終運用する必要があるか再評価する。
- [有力] `secure_final` 用に、成功系の詳細ログを落とし、失敗系ログだけ残す。
- [注意] rollback前後、NVS失敗原因、OTA失敗原因は削らない。

### 7.3 優先度C: APサーバー巨大化対策

- [有力] `maintenanceApServer.cpp` を「認証」「network settings」「pairing」「production precheck」に分割し、将来は compile-time で一部無効化可能にする。
- [有力] 本番機で不要な API（たとえば開発用補助API）があればビルドフラグで除外する。

### 7.4 優先度D: MQTT巨大化対策

- [有力] `mqtt.cpp` の受信分岐をコマンド別ファイルへ分割し、不要コマンドをコンパイル対象から外せる構成へ寄せる。
- [有力] `image package` や `file sync` のような重い管理系コマンドを、常時同梱すべきか見直す。

### 7.5 優先度E: 設定/証明書処理

- [有力] `sensitiveData.cpp` の「NVS本体」「証明書ファイル」「JSON移行処理」を分離する。
- [有力] 初回移行専用コードが常時必要かを再評価する。

## 8. 候補パーティション案の評価

### 8.1 ユーザー提案案

- `app0=4MB`
- `app1=4MB`
- `NVS=100KB`
- `filesystem=6MB`

### 8.2 評価

- [評価] **概ね妥当** である。現在の app 面 1.25MB に対して大幅な余裕を確保でき、ROM 99.6% 問題の第一対策として理にかなっている。
- [評価] `app0/app1=4MB` は現在実測（約1.3MB）の約3倍超であり、今後の IDF寄せ・ログ増減・セキュア化差分を考えても安全側である。
- [評価] `filesystem=6MB` も十分大きく、画像・ログ・証明書・一時ファイルの運用に余裕がある。
- [評価] `NVS=100KB` も現在より拡張されるため、設定値・将来の鍵状態管理に対して安全側である。

### 8.3 注意点

- [厳守] `otadata` は別途確保が必要である（少なくとも現行同等の `0x2000`）。
- [厳守] `coredump` 領域も別途確保が必要である。
- [厳守] app パーティション開始オフセットは `0x10000` 境界へ整列させる必要がある。
- [重要] `NVS=100KB` をそのまま先頭へ置くと `otadata` / `app0` と衝突するため、実CSVでは offset を組み直す必要がある。
- [重要] 上記案でも 16MB 全体に対して 1MB 以上の余りが見込めるため、将来予約領域や `coredump` 拡張余地は十分に残せる。

### 8.4 候補案A の実CSV案

- [重要][2026-04-05] 候補案AをそのままCSVへ落とす場合、以下の配置が妥当候補となる。

| パーティション | offset | size | 補足 |
| --- | --- | --- | --- |
| `nvs` | `0x9000` | `0x19000` | `100KB` |
| `otadata` | `0x22000` | `0x2000` | 現行同等 |
| `app0` | `0x30000` | `0x400000` | `4MB`、`0x10000` 境界整列 |
| `app1` | `0x430000` | `0x400000` | `4MB`、OTA 2面維持 |
| `spiffs` | `0x830000` | `0x600000` | `6MB`、`LittleFS` 用 |
| `coredump` | `0xFF0000` | `0x010000` | 最終 `64KB` |

- [評価] `app0` 開始を `0x30000` に上げることで、`NVS=100KB` と `otadata=0x2000` を保持したまま `0x10000` 境界整列を満たせる。
- [評価] `spiffs` 終端は `0xE30000`、`coredump` 開始は `0xFF0000` のため、その間に `0x1C0000`（約 `1.75MB`）の将来予約領域を残せる。
- [評価] 最終終端は `0x1000000` であり、16MB フラッシュ上限内に正しく収まる。
- [重要] `0x24000` から `0x2FFFF` までの未使用領域は、`app0` を `0x10000` 境界へ合わせるための整列ギャップであり、設計上許容される。
- [推奨] `coredump` を高位アドレス側へ寄せることで、`spiffs` 後方に連続した予備領域を確保し、次回の拡張や `filesystem` 再配分をしやすくする。

### 8.5 `otadata` / `coredump` 再評価メモ

- [重要][2026-04-05] 次回は、候補案Aを基準にしつつ `otadata` / `coredump` の増量価値も比較する。
- [評価] `otadata=0x2000` は ESP32 の標準的な dual OTA 運用では通常十分であり、**優先度は低い**。
- [理由] `otadata` は OTA メタ情報用であり、アプリ本体や `filesystem` のように大きな容量消費源ではないため。
- [評価] `coredump=0x010000`（64KB）は最小寄りであり、今後 `abort` や例外解析を重視するなら **128KB`〜`256KB` への拡張は検討価値がある**。
- [注意] `coredump` を増やす場合も、今回の第一目的は ROM 99.6% 回避と `DEV-002` の安全起動復帰であるため、`app0/app1=4MB` を削ってまで先に拡張しない。
- [推奨] 比較順は以下とする。
  1. 候補案A: `otadata=0x2000`、`coredump=0x010000`
  2. 候補案B: `otadata=0x2000`、`coredump=0x020000`
  3. 必要時のみ `coredump=0x040000` まで比較
- [禁止] `otadata` / `coredump` の議論だけを先行して、`DEV-002` の保全完了前に実機切替へ進まない。
