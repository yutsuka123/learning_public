# 本番セキュア化出荷準備試験計画書

[重要] 本書は、`Secure Boot / ROM暗号化 / Flash Encryption / eFuse` を含む本番セキュア化を、慎重かつ確実に出荷状態まで進めるための実行計画を定義する。  
理由: 不可逆処理の誤操作は復旧不能リスクが高く、実装・試験・運用・証跡を一体で管理する必要があるため。

[厳守] 本書は「段階ゲート方式」を前提とし、各段階で `go/no-go` を明示する。  
理由: 1段でも異常があれば直ちに停止し、被害を最小化するため。

[禁止] 本番鍵の実値、実パスワード、秘密鍵ファイルの実パスを本書へ記載しない。  
理由: 文書経由の機密漏えい防止。

## 1. 対象と目的

### 1.1 対象

- 不可逆処理: `Secure Boot` / `ROM暗号化` / `Flash Encryption` / `eFuse`
- 鍵:
  - `k-iot-secure-boot`
  - `k-iot-flash-encryption`
  - `k-iot-pairing-verify`（検証用公開鍵）

### 1.2 目的

- 出荷対象1台目を安全にセキュア化し、手順の妥当性を確定する。
- 不可逆処理の前に、非不可逆検証を可能な限り完了する。
- 保存先、バックアップ可否、証跡保全を明確化する。

### 1.3 不可逆の定義

- [厳守] 本書で不可逆とみなすのは「eFuse に書込み後、通常手段で元に戻せない設定/鍵投入」である。
- [重要] `Secure Boot` 有効化、`Flash Encryption` 有効化、eFuse の書込み禁止/読出し禁止設定は不可逆扱いとする。
- [禁止] 「あとで戻せる前提」で本番機へ適用しない。

## 2. 全体フロー（段階ゲート）

### 2.1 フェーズ定義

1. フェーズA: 非不可逆準備完了（文書/環境/接続/認証）
2. フェーズB: 試験機で非不可逆検証完了（導線・判定・停止条件）
3. フェーズC: 試験機で不可逆リハーサル（必要最小範囲）
4. フェーズD: 本番1台目で段階実行
5. フェーズE: 出荷判定レビューと展開可否判断

### 2.2 go/no-go ルール

- [厳守] 次段へ進む条件は「当該段階の証跡が3点そろうこと」。
  - `ProductionTool` 判定ログ
  - ESP32 シリアルログ
  - `試験記録書.md` 記録
- [厳守][2026-03-25] `007` クローズ判定では、上記3点に加えて `espefuse summary` の before/after を必須化し、4点証跡で判定する。
- [厳守][2026-03-25] 各段階（FE/SB/NVS/JTAG-DL封鎖）の完了判定前に、安定性確認として「OTA 2回連続」「APモード 10分連続」「通常モード 10分連続」を実施し、`abort`/再起動ループ/通信断がないことを確認する。
- [禁止] 証跡未確定で次段階へ進まない。
- [厳守] `NG` が1つでも出たら即停止し、`問題点記録書.md` に切り分けを記録する。

## 3. 非不可逆を先に終わらせる方策

### 3.1 先行完了対象

- 画面/認証/導線確認（例: 7019, 7020, 7021）
- precheck 閾値確認（例: 7023）
- 中断時再判定ロジックの dry-run 検証（7022 の非不可逆版）
- 本番前リハーサル完了確認（7024）

### 3.2 停止条件

- AP 到達不安定、`LocalServer` 未起動、認証不整合、ログ欠落
- しきい値未達（`minimumFreeHeapBytes`, `minimumStackMarginBytes`）
- 手順逸脱（想定外コマンド、順序違反）

- [厳守] 鍵生成主体は `ProductionTool` 単体ではない。実キーはオフライン端末で `espsecure.py` 等により生成し、`ProductionTool` は eFuse 投入実行と監査導線を担う。
- [厳守] 標準の鍵粒度は以下とする。
  - `k-iot-flash-encryption`: デバイスごと新規生成（再利用禁止）
  - `hmac_nvs_key`: デバイスごと新規生成（再利用禁止）
  - `k-iot-secure-boot`: 製品群ごと運用可（漏えい時の影響は製品群へ波及）
- [要確認][2026-03-26] 例外運用Bとして、実キーを `sensitive` 系ファイルへ暗号化保持する案を比較対象にする。
  - [厳守] 平文キー保存は禁止。暗号化ファイル形式 + 別経路パスフレーズ保管を前提にする。
  - [厳守] `006-0011` 実行前に、バックアップ取得、復元手順、復元確認結果を証跡化する。
  - [禁止] 例外運用Bを採用しても公開リポジトリへの混入を許容しない。
  - [理由] 本番移行後の再現性と復旧性を定量比較してから実行可否を判断するため。

## 5. eFuse 項目一覧（何をどこに書くか）

[厳守] eFuse 名称は ESP-IDF / チップ世代で差異があるため、実行当日に `python -m espefuse --port COMx summary` の実機項目名で照合する。  
理由: 名称不一致で誤ビットへ書く事故を防ぐため。

### 5.1 ESP32-S3 eFuse ブロック構造（公式準拠）

[重要] ESP32-S3 には 11 個の eFuse ブロック（各256ビット）がある。鍵格納に使えるのは以下:


| ブロック名        | 別名           | 用途制約                                                            |
| ------------ | ------------ | --------------------------------------------------------------- |
| `EFUSE_BLK4` | `BLOCK_KEY0` | 汎用（Secure Boot / Flash Encryption / HMAC / ユーザー）                |
| `EFUSE_BLK5` | `BLOCK_KEY1` | 同上                                                              |
| `EFUSE_BLK6` | `BLOCK_KEY2` | 同上                                                              |
| `EFUSE_BLK7` | `BLOCK_KEY3` | 同上                                                              |
| `EFUSE_BLK8` | `BLOCK_KEY4` | 同上                                                              |
| `EFUSE_BLK9` | `BLOCK_KEY5` | [禁止] XTS_AES 鍵には使用不可（HWエラッタ）。Secure Boot digest / HMAC / ユーザーのみ |


[厳守] Flash Encryption 鍵は `BLOCK_KEY0`〜`BLOCK_KEY4` の範囲に書く。`BLOCK_KEY5` へ書くと HW が参照できず起動不能になる。

### 5.2 現時点で確認済みの観測項目（実機証跡あり）

- `SPI_BOOT_CRYPT_CNT`（Flash Encryption 有効/無効、3ビット）
- `SECURE_BOOT_EN`（Secure Boot 有効/無効）
- `ENABLE_SECURITY_DOWNLOAD`（Secure ROM download mode 有効化）

### 5.3 本番運用で管理する eFuse 項目（チェックリスト）

[致命的修正][2026-03-25] ESP32-S3 では **Flash Encryption を先に有効化し、Secure Boot v2 を後** に有効化する。  
理由: Secure Boot v2 有効化時に `RD_DIS` への write-protect が適用されるため、先に Flash Encryption 鍵の read-protect を設定しないと FE 鍵が読取可能なまま固定されるため（ESP-IDF 公式 Security Features Enablement Workflows 準拠）。

### 5.4 eFuse に書く鍵の対応表と生成コマンド


| 鍵                        | eFuse反映先                                                                     | 生成コマンド                                                                                                                                       | 投入コマンド                                                                       | 備考                                                    |
| ------------------------ | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------------- |
| `k-iot-secure-boot`      | `BLOCK_KEYm` (`SECURE_BOOT_DIGEST0`) + `SECURE_BOOT_EN`                      | `espsecure generate-signing-key --version 2 --scheme rsa3072 sb_signing_key.pem`                                                             | `espefuse burn-key-digest BLOCK_KEYm sb_signing_key.pem SECURE_BOOT_DIGEST0` | 秘密鍵(PEM)はデバイスへ置かず、公開鍵ダイジェストのみ eFuse へ。ダイジェストは自動生成される。 |
| `k-iot-flash-encryption` | `BLOCK_KEYn` (`XTS_AES_128_KEY` or `XTS_AES_256_KEY`) + `SPI_BOOT_CRYPT_CNT` | `espsecure generate-flash-encryption-key fe_key.bin` (AES-128) / `espsecure generate-flash-encryption-key --keylen 512 fe_key.bin` (AES-256) | `espefuse burn-key BLOCK_KEYn fe_key.bin XTS_AES_128_KEY`                    | AES-256の場合2ブロック消費。デバイスごと一意推奨。                         |
| `k-iot-pairing-verify`   | eFuse 非対象                                                                    | ビルド時組込み                                                                                                                                      | なし                                                                           | FW 検証領域へ公開鍵として保持。                                     |


[厳守] Secure Boot v2 の鍵投入には `burn-key-digest`（`burn-key` ではない）を使用する。理由: RSA-3072 秘密鍵 PEM から公開鍵ダイジェストを自動算出して eFuse へ焼くコマンドであり、`burn-key` を使うと生の鍵バイナリとして扱われて目的が不一致になるため。  
[重要] FE 鍵は `burn-key` で投入し、投入時にデフォルトで read-protect + write-protect が自動適用される。  
[重要] SB ダイジェストは `burn-key-digest` で投入し、デフォルトで write-protect が適用されるが **read は可能なまま**（SB の検証に必要なため）。

### 5.6 シリアル運用（FW制御）と最低限セキュリティ

[重要][2026-05-02変更] 本プロジェクトでは、**試験中はシリアル出力を残し、最終完成時だけシリアル抑止済みの量産通常FWへ更新する**前提で運用する。必要時のみ、これに加えて `DIS_USB_SERIAL_JTAG` による書換え不能化を最終クローズで扱う。  
理由: 1号機ノウハウどおり、他の不可逆・証跡・安定性確認が問題ないことを見届けるまでは復旧余地を残し、最後にセキュア化を完成させるため。


| レベル | 方針                               | 推奨タイミング      | リスク/注意          |
| --- | -------------------------------- | ------------ | --------------- |
| S1  | シリアルログは FW 版で制御（通常版は最小ログ、診断版で詳細） | 試験機段階から適用    | 版管理が甘いとログ漏えい    |
| S2  | DLモード封鎖 + JTAG封鎖（eFuse）          | 本番1台目レビューOK後 | 再書込み経路が大きく制約される |
| S3  | シリアル抑止済み量産通常FWへの更新 + 必要時のみUSB/JTAG書換え不能化 | すべての不可逆・証跡確認完了後 | 故障解析の復旧難易度が最大化  |


[厳守] S2/S3 へ進む前に、診断用FWでの代替解析手段（署名済み限定）を確立する。
[禁止] 段階途中でシリアル閉鎖を前倒ししない。シリアル書換え不能化と量産通常FWへの更新は、他の不可逆・証跡・安定性確認が完了した後の最終段でのみ扱う。
[重要] シリアル出力停止は、`UART_PRINT_CONTROL` / `DIS_USB_SERIAL_JTAG_ROM_PRINT` の eFuse 焼成ではなく、**シリアル抑止済みの量産通常FWへ更新することで実現する**。`ProductionTool` は最終クローズ判断と証跡記録を担い、FW切替そのものは量産通常FWの配布・更新手順で管理する。

### 5.6b Flash Encryption 鍵長選択（AES-128 vs AES-256）


| 観点            | AES-128（256-bit鍵）          | AES-256（512-bit鍵）                             |
| ------------- | -------------------------- | --------------------------------------------- |
| eFuse ブロック消費  | **1ブロック**（`BLOCK_KEY0` のみ） | **2ブロック**（`BLOCK_KEY0` + `BLOCK_KEY1`）        |
| セキュリティ強度      | 128bit セキュリティ（現在の標準水準）     | 256bit セキュリティ（量子コンピュータ耐性向上）                   |
| ブロック残数（全6中）   | 5ブロック残（SB/HMAC/将来用に余裕大）    | 4ブロック残（SB + HMAC + 予備1）                       |
| Key Purpose   | `XTS_AES_128_KEY`（1種）      | `XTS_AES_256_KEY_1` + `XTS_AES_256_KEY_2`（2種） |
| 将来の NVS 暗号化   | HMAC鍵用に1ブロック残せる            | HMAC鍵用に1ブロック残せる（ただし余裕は減る）                     |
| 実装難易度         | シンプル                       | 2ブロックの連続配置が必要（`BLOCK_KEY5` 前後注意）              |
| ESP-IDF デフォルト | AES-128 が推奨デフォルト           | 要明示設定                                         |


**本プロジェクト推奨: AES-128 (`XTS_AES_128_KEY`)**  
理由: ① 現状の IoT 製品レベルでは AES-128 で十分な強度 ② ブロックを節約してSB/HMAC/NVS暗号化に余裕を持たせる ③ 実装シンプル化。ブロック割当て: `BLOCK_KEY0`=FE、`BLOCK_KEY1`=SB、`BLOCK_KEY2`=NVS-HMAC（採用時）。

### 5.6c NVS Encryption 採用要否と方式選択（HMAC方式 vs Flash Encryption方式）

**NVS Encryption とは**: NVS（Wi-Fi設定、MQTT設定、鍵情報等）をフラッシュ上で暗号化する機能。Flash Encryption がある場合でも NVS パーティション単体を追加で暗号化できる。

#### 本プロジェクトの NVS 保存データ（実コード確認済 `sensitiveData.cpp`）


| カテゴリ          | 保存内容                                   | 機密レベル                    |
| ------------- | -------------------------------------- | ------------------------ |
| `wifi`        | Wi-Fi SSID / Password                  | **高**（ネットワーク侵入経路）        |
| `mqtt`        | MQTT URL / User / **Password** / TLS設定 | **高**（MQTT接続認証情報）        |
| `server`      | API サーバー URL / User / **Password**     | **高**（LocalServer接続認証情報） |
| `ota`         | OTA サーバー URL / User / **Password**     | **高**（OTA改ざんリスク）         |
| `timeServer`  | NTP URL / Port / TLS                   | 低（固定URL）                 |
| `credentials` | `keyDevice`（ペアリング鍵 Base64）/ バージョン管理情報  | **最高**（デバイス固有秘密鍵）        |


[重要] `credentials.keyDevice` はペアリング鍵そのものであり、これが漏洩するとデバイスのなりすましが可能になる。

#### 方式比較


| 観点                | Flash Encryption のみ（現方針）         | + NVS Encryption HMAC方式            |
| ----------------- | -------------------------------- | ---------------------------------- |
| NVS保護の仕組み         | FE でフラッシュ全体を暗号化 → NVS もブロック単位で保護 | FE に加え、NVS パーティションを別鍵で AES-XTS 暗号化 |
| 鍵の保存先             | FE 鍵（eFuse）のみ                    | FE 鍵 + HMAC 鍵（eFuse `BLOCK_KEY2`）  |
| eFuse 追加消費        | 0                                | **1ブロック**（`BLOCK_KEY2`）            |
| 強度                | FE が唯一の保護（単一障害点）                 | FE + 独立した HMAC 鍵（多層防御）             |
| `keyDevice` 漏洩リスク | FE が破られると全 NVS が読める              | FE + NVS 鍵の両方が必要（縦深）               |
| 実装の複雑さ            | シンプル（現状のまま）                      | やや複雑（HMAC鍵生成 + NVS初期化実装変更）         |
| OTA後の継続動作         | 問題なし                             | HMAC鍵が eFuse に固定されるため OTA でも継続可    |
| 鍵ローテーション          | 困難（FE 鍵は不可逆）                     | 困難（HMAC 鍵も eFuse 不可逆）              |


#### 懸念事項（Flash Encryption のみの場合）

1. `**keyDevice`（ペアリング鍵）の単一障害点**: Flash Encryption 鍵（BLOCK_KEY0）が解析された場合、NVS内の全データ（MQTT Password、Wi-Fi Password、**keyDevice**）が一度に漏洩する。
2. **物理アクセス攻撃**: JTAG/DL を封鎖しても、フラッシュ IC をはんだ除去して直接読み取る「チップ外し攻撃」では FE 鍵なしに暗号化データにアクセスされる可能性は排除できない（ただし ESP32-S3 では FE 鍵が eFuse に保護されるため実用上の障壁は非常に高い）。
3. **FW バグによる NVS 平文ログ流出**: シリアルログ経由でデバッグ中に機密値を誤出力するリスクは FE で防げない（FW レイヤーの問題）。`[禁止] パスワード値をログへ直接出力しない` は既にコードに記載済みだが継続注意。

**[旧仕様][2026-04-23] NVS Encryption HMAC方式は採用見送りとする。**  
理由: `Flash Encryption` でフラッシュ上の NVS を保護でき、`Secure Boot` / `MQTT TLS` / `JTAG/DL封鎖` の現行セキュリティ層で要件を満たせるため。加えて、NVS暗号化を維持するための追加検証・運用負荷が大きく、現行構成に対する費用対効果が低いと判断した。

#### NVS Encryption HMAC方式 採用検討記録（旧仕様）
[旧仕様] 以下の工程は採用検討時の記録であり、現在の実施対象ではない。

[重要] NVS Encryption HMAC方式は FE/SB とは独立した追加保護だが、**sdkconfig変更・FW実装変更**を伴うため、不可逆な eFuse 書込みの前に FW 側の作業をすべて完了する必要がある。  
[重要] ESP-IDF公式 `NVS Encryption`（ESP32-S3）では、**HMAC方式は `nvs_keys` パーティションを必要としない**。`nvs_keys` が必要なのは Flash Encryption ベース方式。

**工程 A: 既存 `nvs` パーティション維持確認（可逆・設計確認）**

現状パーティション（維持）:

```
nvs,      data, nvs,      0x9000, 0x5000
otadata,  data, ota,      0xe000, 0x2000
```

[厳守] HMAC方式では `nvs_keys` を新設しない。  
[理由] 公式仕様上、HMAC方式は eFuse 内の `HMAC_UP` 鍵から NVS鍵を導出し、鍵パーティションを使用しないため。  
[重要] 既存 `nvs` 領域の物理配置を変えないため、**パーティション変更起因のデータ消失リスクを回避**できる。

**工程 B: sdkconfig.defaults の変更（可逆・ビルド変更）**

```

CONFIG_NVS_ENCRYPTION=y
CONFIG_NVS_SEC_KEY_PROTECTION_SCHEME_HMAC_BASED=y
CONFIG_NVS_SEC_HMAC_EFUSE_KEY_ID=2  # BLOCK_KEY2 を HMAC 鍵として使用
```

[注意] `CONFIG_NVS_SEC_HMAC_EFUSE_KEY_ID=2` は HMAC 鍵が `BLOCK_KEY2` に書かれていることを前提とする。

**工程 C: FW 実装変更（可逆・C++変更）**

`sensitiveData.cpp` の初期化部分に以下変更が必要:

```cpp
// 変更前（現在）
// nvs_flash_init() を内部で呼ぶ Preferences.begin()

// 変更後（HMAC方式）
// nvs_sec_cfg_t cfg;
// nvs_flash_generate_keys_v2(&cfg);  または
// nvs_flash_read_security_cfg_v2(&cfg);
// nvs_flash_secure_init_partition("nvs", &cfg);
// その後、通常通り Preferences.begin() で利用可能
```

[重要] Arduino Preferences ライブラリは内部で `nvs_flash_init()` を呼ぶが、`nvs_flash_secure_init_partition()` を先に呼んでおけばその後の `Preferences.begin()` は暗号化された NVS を透過的に利用できる。

**工程 D: eFuse 書込み（不可逆）**

```

```

`HMAC_UP` は「ユーザー方向の HMAC（Upstream）」を意味し、NVS 暗号化鍵導出に使用される。

**工程 E: 検証（最終確認）**

1. NVS に設定を保存後、電源 OFF/ON でデータが正常に復元されることを確認
2. `espefuse summary` で `KEY_PURPOSE_2 = HMAC_UP` を確認
3. FW バイナリを別の未 eFuse 書込み機に焼いた場合、NVS が読めないことを確認（隔離テスト）

[重要] `DIS_DOWNLOAD_MODE`（EF-013）は**焼かない（保持）**。`ENABLE_SECURITY_DOWNLOAD` を代わりに採用。

#### NVS既存データ保全方針（HMAC方式）

[重要] HMAC方式は `nvs` パーティションを移動しないため、**パーティション変更による即時消失は発生しない**。  
ただし、暗号化有効化の初回起動時に初期化不整合が起きる可能性があるため、以下を必須化する。

1. 事前に NVS シークレットバックアップ（JSON + 必要時は生ダンプ）を取得
2. HMAC化後の初回起動で NVS 読書き整合を確認
3. 不整合時は `nvs` を初期化し、バックアップ値を再投入

[厳守] 「保全できるはず」で進めず、毎回バックアップを先行してから不可逆段階へ進む。

#### [原因確定][2026-03-26] セキュア化（FE段階）が**間接的に**起因

**直接原因**: AP モード起動時に `Preferences.cpp: nvs_open failed: NOT_FOUND` が発生し、
`loadRolePasswordsFromPreferences` がスキップされ、AP HTTP サーバーの認証パスワード（`mfg` ロール）が読めない状態で API が動作している。

**根因の因果関係**:
1. `sdkconfig.defaults` で `CONFIG_NVS_ENCRYPTION=y` + `CONFIG_NVS_SEC_HMAC_EFUSE_KEY_ID=2` を有効にしてビルド
2. しかし eFuse `BLOCK_KEY2` には HMAC 鍵が**未投入**（段階2(NVS) は未実施）
3. FW は NVS 暗号化前提で動作するため、HMAC 鍵が eFuse にない状態では NVS の `open` が失敗する
4. AP モードの `Preferences` （role パスワード保存先）も NVS 経由のため `nvs_open failed: NOT_FOUND` となる
5. `loadRolePasswordsFromPreferences skipped` → AP HTTP API の認証設定が不完全 → HTTP 応答不能

**シリアルログ証拠**（`logs/ap-boot-serial-20260326-full.txt`）:
```
[23.3s] [4438][E][Preferences.cpp:50] begin(): nvs_open failed: NOT_FOUND
[23.7s] [WARN] loadRolePasswordsFromPreferences skipped. preferences.begin failed.
[23.7s] [WARN] maintenanceApServer::start success. AP maintenance API enabled.
```

**なぜ STA モードは正常だったか**:
- STA では `sensitiveDataService::loadWifiCredentials failed` 発生後に `sensitiveData.h` マクロ値へフォールバック
- MQTT/NTP/OTA はマクロ値で動作するため影響なし
- AP モードの `loadRolePasswordsFromPreferences` にはマクロ値フォールバックが**存在しない**

**なぜ FE 前は AP が正常だったか**:
- FE 前は `CONFIG_NVS_ENCRYPTION=y` がビルドに含まれて**いなかった**（sdkconfig.defaults に追記したのは FE 段階準備時）
- つまり FE 前の FW は NVS を平文で読み書きしており、`Preferences::begin()` は正常に成功していた

**対策方針**:
- [短期] 段階2(NVS) の HMAC 鍵（EF-009b）を投入すれば NVS 読出しが正常化し、AP HTTP も回復する見込み
- [短期代替] `sdkconfig.defaults` から `CONFIG_NVS_ENCRYPTION=y` 関連を一時的に外してリビルド → OTA で反映すれば NVS 平文読出しで AP 回復（ただし段階2進行時に再ビルド必要）
- [中期] AP モードの `loadRolePasswordsFromPreferences` にもマクロ値フォールバックを追加し、NVS 失敗時でも API が最低限動作するようにする

#### [解決済][2026-03-26] 対策A実施結果

**実施した対策**（対策A + 中期対策の同時実施）:

1. **NVS HMAC鍵投入（EF-009b / 不可逆）**: `espefuse burn_key BLOCK_KEY2 hmac_nvs_key_test.bin HMAC_UP`
2. **FW修正（`secureNvsInit.cpp` 新規追加 / OTA beta.29 反映）**: CONFIG_NVS_ENCRYPTION の有無に関わらず NVS を安全に初期化するフォールバック実装
3. **AP HTTP 全4ロール認証成功を確認**: ping 4/4 OK、health OK、mfg/admin/maintenance/user login OK

**[致命的教訓][2026-03-26] sdkconfig.defaults と eFuse 投入の順序依存**:

今回の問題の本質は **`sdkconfig.defaults` への NVS Encryption 設定追加と、対応する eFuse HMAC鍵投入が同一段階で完了していなかった** ことである。

```
[禁止パターン] CONFIG_NVS_ENCRYPTION=y を含むFWをビルド・OTAした後、
              対応する HMAC鍵（EF-009b）を投入しないまま運用すると
              NVS全面読書き不能 → AP HTTP 認証不能 → セキュア化停止
```

**[厳守] 以下の順序依存ルールを確立する**:

| 順序 | 作業 | タイミング | 理由 |
|---|---|---|---|
| 1 | `sdkconfig.defaults` に NVS Encryption 関連を追加 | eFuse HMAC鍵投入の**直前ビルド** | FWがHMACを前提に動作するため |
| 2 | `secureNvsInit.cpp` のフォールバック実装を含むFWをビルド | 同上 | HMAC鍵不在でもboot停止しない安全策 |
| 3 | OTAまたはシリアルでFW反映 | eFuse投入**前** | HMAC鍵投入後にFWが対応できる状態にする |
| 4 | `espefuse burn_key BLOCK_KEY2 ... HMAC_UP`（EF-009b） | FW反映**後** | FWがHMACを使えるようにしてから投入 |
| 5 | 再起動 → NVS読書き確認 → AP HTTP確認 | 投入**直後** | 投入成功と動作確認を不可分にする |

**[厳守] sdkconfig.defaults 変更とeFuse投入は「同一段階内」で完結させる**。段階間にsdkconfigだけ先行してeFuse未投入の状態を作らない。

**[重要] PlatformIO/Arduino 環境の実態**:
- `sdkconfig.defaults` に `CONFIG_NVS_ENCRYPTION=y` を記載しても、PlatformIO/Arduino ビルドでは **C プリプロセッサマクロ `CONFIG_NVS_ENCRYPTION` が有効にならない** ケースがある。
- 今回の `secureNvsInit.cpp` は `#ifdef CONFIG_NVS_ENCRYPTION` の分岐で plaintext fallback を実装しているが、実際にはプリプロセッサ未定義のため常に plaintext パスが実行される。
- つまり PlatformIO/Arduino 環境では NVS Encryption の「FW側暗号化読書き」は ESP-IDF ネイティブビルドでないと完全には機能しない。
- ただし、eFuse に HMAC鍵を投入すること自体は有効であり、ESP-IDF レベルの NVS ドライバが暗号化を透過的に処理するため、FW明示呼び出し不要で一定の保護は成立する。
- **[将来対応]** ESP-IDF ネイティブビルドへの移行時に、`secureNvsInit` の secure パスが正しく動作することを再検証する。

**[ProductionTool自動化への反映事項]**:
1. ProductionTool は段階2(NVS) の実行前に、FWが `secureNvsInit` フォールバックを含むバージョンであることを `firmwareVersion` チェックで確認すること

#### 6.4a Secure Boot 選択肢整理（2026-03-27確定）

[重要] `ESP32-S3` の Secure Boot は本計画では **Secure Boot v2** を採用する。  
理由: 現行 SoC / ESP-IDF 公式導線 / 既存 `sdkconfig.defaults` / eFuse 計画と整合するため。

| 観点 | 選択肢 | 本計画の採用 | 理由 |
|---|---|---|---|
| Secure Boot 方式 | v1 / v2 | **v2** | `ESP32-S3` での正規導線。現行設定 `CONFIG_SECURE_BOOT=y`, `CONFIG_SECURE_BOOT_V2_ENABLED=y` と一致 |
| 署名方式 | 自動署名 / 外部署名 | **外部署名** | `CONFIG_SECURE_BOOT_BUILD_SIGNED_BINARIES=n`。量産時の署名責務を分離し、署名素材の監査とバックアップを明確化するため |
| 署名鍵運用 | 単一鍵 / 複数鍵ローテーション前提 | **単一鍵** | 1台目確立を優先。`BLOCK_KEY1` に `SECURE_BOOT_DIGEST0` を投入し、未使用 slot は revoke する |
| digest slot | slot0/1/2 を複数有効 | **slot0のみ使用** | 手順を単純化し、読戻し判定を明確にするため |
| 未使用 slot | 保持 / revoke | **revoke1 + revoke2** | 使わない検証経路を閉じて誤運用面積を減らすため |
| aggressive revoke | 有効 / 無効 | **無効** | 1台目で不要な複雑化を避けるため。通常 revoke で十分 |
| 署名対象 | app のみ / bootloader+app | **bootloader + app** | Secure Boot v2 の成立条件として両方を署名対象にする |

[厳守] `burn-key-digest` で投入するのは **秘密鍵そのものではなく公開鍵ダイジェスト** である。  
[厳守] 本番/試験を問わず、`sb_signing_key.pem` は `IoT/LocalServer/data/secure-backups/` 配下へバックアップを保持する。

> **[要確認][反映済み] 段階2(NVS) と 段階3(SB) の実行順序における `RD_DIS` 競合整理について**
>
> **前回項目:**
>
> - 旧版では段階2(SB) 末尾で `EF-015`（`RD_DIS` write-protect）を実施
> - その後に段階2.5(NVS) で `EF-009b`（`BLOCK_KEY2` への `HMAC_UP` 鍵投入）を実施
>
> **詳細内容:**
> 旧版計画では、段階2の最後で `RD_DIS` を write-protect（EF-015）し、その後の段階2.5で HMAC鍵（EF-009b）を `burn-key` していました。
> しかし、`espefuse burn-key` はデフォルトで書き込んだ鍵を read-protect（読出禁止）にするため、内部的に `RD_DIS` eFuse の該当ビットを 1 にしようとします。
> もし段階2で既に `RD_DIS` が write-protect されていると、段階2.5での `espefuse burn-key BLOCK_KEY2 ...` は `RD_DIS` を変更できず **書込みエラー（Failure）** となります。
>
> **変更理由:**
>
> - 前回は「FE鍵とSB鍵」の競合に主眼を置いて順序を整理したが、今回あらためて「NVS HMAC鍵も `burn-key` で read-protect を伴う」点を同じ整理軸で見直した結果、`RD_DIS` の閉鎖タイミングが早すぎることが判明したため。
> - 不可逆工程での順序誤りは、その場で復旧不能な書込み失敗につながるため、元の手順を残したまま [要確認] として明示し、後続作業者が迷わないようにする。
>
> **根拠資料:**
> ESP-IDF Security Features Enablement Workflows (ESP32-S3)
> 「The reason for this particular ordering is that when enabling Secure Boot (SB) v2, it is necessary to keep the SB v2 key readable. To protect the key's readability, the write protection for `RD_DIS` is applied. However, this action poses a challenge when attempting to enable Flash Encryption, as the Flash Encryption (FE) key needs to remain unreadable. This conflict arises because the `RD_DIS` is already write-protected...」
> （※Flash Encryption鍵だけでなく、HMAC鍵など「read-protectが必要なすべての鍵」に対して同じ制約が適用されます）
>
> **出典URL:**
>
> - ESP-IDF Security Features Enablement Workflows（ESP32-S3）  
> `https://docs.espressif.com/projects/esp-idf/en/latest/esp32s3/security/security-features-enablement-workflows.html`
> - esptool / espefuse burn-key（ESP32-S3）  
> `https://docs.espressif.com/projects/esptool/en/latest/esp32s3/espefuse/burn-key-cmd.html`
>
> **旧修正方針（NVS Encryption 採用時）:**
> `EF-015`（`RD_DIS` write-protect）は、FE鍵（EF-001）と HMAC鍵（EF-009b）の両方を `burn-key` して read-protect を完了した **後（すべての鍵投入の最後）** に実行する必要があります。
> **[現行方針 2026-04-29]** 案Aでは HMAC鍵（EF-009b）を焼かないため、本文 §6.4 の実施順序は `FE -> NVS暗号化なし確認（非焼込みゲート） -> SB -> 封鎖` を正とします。

1. 段階4(封鎖): JTAG/DL封鎖
  - JTAG無効化（EF-010, EF-011）
  - `DIS_USB_SERIAL_JTAG`（EF-012）は試験では保留し、最終クローズ時のみ適用可
  - `ENABLE_SECURITY_DOWNLOAD`（EF-014）**※これを最後に焼く**
  - -> 読戻し -> 最終起動確認 -> 証跡確定
  - -> 安定性確認セット（OTA 2回連続 / APモード 10分 / 通常モード 10分）

#### 6.4d 最終形のシリアル方針（2026-03-27整理）

[目標] **シリアルログ出力は残すが、通常のシリアル経由 FW 読書きは不可** とする。  

本計画の最終形は以下の組合せを採用する。

| 項目 | 本計画の採用 | 目的 |
|---|---|---|
| `DIS_PAD_JTAG` | **有効化** | パッド JTAG を完全遮断 |
| `DIS_USB_JTAG` | **有効化** | USB-JTAG 切替を遮断 |
| `DIS_USB_SERIAL_JTAG` | **条件適用** | 試験中は USB シリアルログ出力を残し、最終クローズ時だけ閉鎖可能にする |
| `ENABLE_SECURITY_DOWNLOAD` | **有効化** | ROM DL をセキュアモードへ制限 |
| `DIS_DOWNLOAD_MODE` | **未適用** | 完全封鎖はしない。ログ維持と保守余地を残す |
| `Flash Encryption` | **有効化済み** | フラッシュ平文読出し抑止 |
| `Secure Boot v2` | **段階3で有効化** | 未署名 FW 起動拒否 |
| FW 側危険操作 | **拒否/無効化を確認対象化** | ログは許容しつつ、設定書換え・ダンプ等を封じる |

[重要] この構成により、**通常の平文 FW 書込み、未署名 FW 起動、JTAG 解析、通常 DL モード経由の書換え** は抑止できる見込みである。  
[重要] ただし、**「シリアルログを残したまま、あらゆる意味でのシリアル経由 FW 読書きを絶対不可能にする」ことを eFuse だけで完全保証する構成ではない**。  
理由: `DIS_USB_SERIAL_JTAG` を焼かずに USB シリアルデバイスを残す以上、ROM 側の挙動・セキュアダウンロード条件・FW 側コマンド制御を含めた **組合せ保証** になるため。

[結論][2026-03-27] 本プロジェクトの 007-C 判定で採るべき完了条件は次の 4 点である。
1. シリアルログは取得可能であること
2. 未署名 FW は起動できないこと
3. 通常のシリアル/DL/JTAG 経路で FW 読書きが成立しないこと
4. FW 側の危険コマンド（設定書換え・メモリダンプ等）が拒否または無効化されること

[重要][2026-03-27] 上記 3. は、本番個体では **非破壊に観測できる範囲** で判定する。  
具体的には `esptool --no-stub get_security_info`、`read_mem` 拒否、`read_flash` 拒否、JTAG disabled 表示、シリアル入力コマンド未実装確認の組合せで判定し、`write_flash` 成否を試す破壊的確認は原則行わない。

[実測][2026-03-27] 段階4後の対象機で以下を確認した。
- `python -m esptool --chip esp32s3 --port COM9 --no-stub get_security_info`
  - `Chip is ESP32-S3 in Secure Download Mode`
  - `Secure Boot: Enabled`
  - `Flash Encryption: Enabled`
  - `JTAG: Permanently Disabled`
- `python -m esptool --chip esp32s3 --port COM9 --no-stub read_mem 0x3fce3808`
  - `UnsupportedCommandError: This command (0xa) is not supported in Secure Download Mode`
- `python -m esptool --chip esp32s3 --port COM9 --no-stub read_flash 0x0 0x40 ...`
  - `UnsupportedCommandError: This command (0xe) is not supported in Secure Download Mode`
- `ESP32/src` を検索した結果、`Serial.available` / `Serial.read` / `readStringUntil` 等のシリアル入力処理は存在せず、FW 側のシリアル保守コマンドは未実装

[整理][2026-03-27] これにより、現行構成の `7100` は「USB シリアルポートは残るが、Secure Download Mode で危険な read 系コマンドが拒否され、JTAG は無効、FW 側にもシリアル入力コマンドがない」ことを本番個体で非破壊確認する試験として扱う。

[禁止] 「USB シリアルが見える」ことだけをもって未完成と判定しない。  
[禁止] 逆に「ログが出るから安全」とも判定しない。`007-0010` / `7100` の実測証跡で確定する。

[厳守] 各段階の直後に「読戻し -> 再起動 -> 証跡確定」を完了しない限り次段へ進まない。  
[厳守] `ENABLE_SECURITY_DOWNLOAD` は全eFuse書込み完了後に最後に焼く。以降 `espefuse` での追加書込みが不可になるため。
[厳守] 安定性確認セットで `abort`、再起動ループ、AP到達不能、MQTT再接続失敗、OTA失敗のいずれかが出た場合は即停止し、原因確定まで次段へ進まない。

### 6.4b 試験機不可逆後の利用可能範囲（`006-0011` 完了後の判断用）

- [完了][2026-04-27] `006-0011` は完了済み。以後は本節を「試験機を継続利用するか、`DEV-003` を投入するか」の判断基準として扱う。
- [重要] `006-0011` で不可逆を書いた試験機は「破棄確定」ではない。正常に段階完了すれば、その後も署名済み/暗号化前提の運用試験に継続利用できる。
- [厳守] ただし次の用途は不可または強く制限される。
  - eFuse 再書込みを前提にした試験（再初期化、別鍵への差替え）
  - 未署名FWや平文書込みを前提にした試験
  - JTAG/Download 制約を解除して行う解析
- [推奨] 不可逆後に継続実施する試験は以下を優先する。
  - OTA 更新継続可否（署名/暗号化条件下）

### 6.4c OTA専用最終化の判断条件（`007` 内判断）

- [要確認][2026-03-26] 「最終的に OTA のみ」を採用する場合、以下を満たした後に不可逆設定を判断する。
  1. OTA 連続更新試験（複数回）とロールバック試験が安定していること
  2. 失敗時の復旧手順（工場内限定手順を含む）が文書化されていること
  3. 出荷後保守でシリアル更新経路を使わない運用合意があること
- [重要] OTA 機能そのものは eFuse 未適用でも実現できる。  
ただし「更新経路を OTA に限定する」ことは別論点であり、DL/JTAG/シリアル制約を伴う不可逆設定の採否と一体で決める必要がある。
- [厳守] `007` 内で採否を決める場合も、`4点証跡` と停止条件を満たさない限り反映しない。

### 6.4d 文鎮化時の復旧と再発防止ログ

- [重要][2026-04-27] もし試験機や出荷対象機が文鎮化しても、代替の ESP32 ボードを使って同じ運用状態へ戻せるよう、復旧素材とログの保全点を先に固定する。

- [厳守] 文鎮化が疑われたら、再書込みや復旧コマンドの前に次を採取する。
  1. ESP32 側 `maintenanceApServer` の起動スナップショット
  2. `/api/health` の応答とシリアルログ
  3. `/api/auth/login` の成否と失敗ログ

- [推奨] 復旧結果は `問題点記録書.md` と `試験記録書.md` の両方へ同日転記する。理由: 復旧可否と再発防止を同じ時系列で追えるようにするため。
