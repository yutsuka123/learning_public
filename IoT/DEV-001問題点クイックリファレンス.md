# IoT DEV-001 問題点クイックリファレンス

[最重要] 本書は `DEV-001` の主要問題を短時間で参照できるように整理した要約である。  
理由: 実機作業中に「何が起きたか」「何を禁止するか」を即判断できるようにするため。

[厳守] 詳細の一次情報は `問題点記録書.md` と `試験記録書.md` を正とし、本書はクイック参照に限定する。  
理由: 要約と原本の役割を分離し、記録不整合を防ぐため。

## 1. 事象一覧（即確認）
### 1.1 事象A: 初回停止（NVS初期化系）
- 事象: `secureNvsInit` 失敗時に起動停止へ入りうる状態が発生した。
- 影響: 通常起動できず、AP/STA運用へ進めない。
- 対応: `main.cpp` で失敗時 rollback を明示実装し、復帰可能性を確保。
- 参照: `問題点記録書.md` `#0025`、`試験記録書.md` `7101`

### 1.2 事象B: 復旧失敗（`--force` 後 `invalid header`）
- 事象: secure個体へ平文 `write_flash --force` 復旧を試行後、`invalid header` 連続で起動不可となった。
- 影響: `DEV-001` は解析保全固定へ移行（主系運用から除外）。
- 判断: これは「NVS読取失敗」とは別系統の、書込み/暗号化整合崩れの問題。
- 参照: `問題点記録書.md` `#0024`、`試験記録書.md` `7100`

## 2. 原因整理（誤認防止）
- [重要] `#0025`: `nvs_sec_provider.h` 未提供（現行Arduino系framework）により `secure_final` が fail-closed 停止。
- [重要] `#0024`: `--force` 復旧は secure構成と整合せず、起動イメージ整合を破壊した可能性が高い。
- [厳守] 2つを混同しない（NVS初期化問題と、強制書込み後の起動不能は別件）。

## 3. 禁止事項・再発防止
- [禁止] 本番/保全個体に対する平文 `write_flash --force`。
- [厳守] 先に `esp32s3_secure` と `7099/7101` で安定性確認し、`secure_final` は判定後に適用。
- [厳守] `secure_final` 失敗時は rollback ログ（起動/boot/update partition）を採取して記録。
- [推奨] 復旧用途は `esp32s3_secure_rescue` を使用し、通常運用や出荷判定に混用しない。

## 4. 現在の運用判断
- `DEV-001`: 解析保全固定（再破壊リスク操作を行わない）。
- `DEV-002`: 主系検証機として継続し、`007-C` 条件確認を進める。
- 方式選定: `案B -> 案A -> 案C` で評価（`最終セキュア化方式選定メモ.md` 参照）。
