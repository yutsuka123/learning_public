# IoT ネットワーク運用仕様書

[重要] 本書はESP32/Local/Cloudの名前解決、到達性、常駐起動、公開面最小化を定義する。  
理由: ローカル運用からAWS移行までの通信方式を一貫化し、手戻りを防止するため。

[厳守] デバイス（ESP32）は原則としてインターネットから直接到達可能にしない。  
理由: グローバル公開面を最小化し、DDoS・不正アクセスの攻撃面を削減するため。

[禁止] 「一時的にセキュリティを緩めて後で戻す」運用を恒久化しない。  
理由: 戻し忘れによる事故を防ぐため。

## 1. 適用範囲
- 対象: ESP32-S3、`LocalServer`（TypeScript/Node.js）、`SecretCore`（Rust）、`ProductionTool`（工場内専用）、将来Cloud（AWS想定）
- 対象OS: Windows / macOS
- 対象ネットワーク: ローカルLAN、インターネット経由（将来）

## 1.1 ネットワーク責任範囲
- [厳守] `LocalServer` は通常運用の REST API、Web UI、OTA 配布、MQTT 連携の通信終端を担当する。
- [厳守] `SecretCore` はローカルIPC終端のみを担当し、ネットワーク直公開しない。
- [厳守] `ProductionTool` は工場内専用ネットワークまたは閉域AP経由でのみ利用し、インターネット直公開しない。
- [廃止の方針] `LocalSoft` は `LocalServer` 一体化により新規利用しない。理由: 通信終端と保存終端を `LocalServer` に集約するため。
- [厳守] 詳細な責任境界は `モジュール仕様書.md` を親定義とする。

## 2. 名前解決（DNS）方針
### 2.1 ローカル運用
- [厳守] ローカル専用ゾーンはCoreDNSで管理する（例: `iot.local`）。
- [厳守] DHCP配布DNSをCoreDNSへ向け、端末（Windows/macOS/ESP32）は同一ルールで名前解決する。
- [禁止] 公開ドメイン（インターネットDNS）の上書き運用。
- [推奨] CoreDNSは`forward`で外部DNSへフォールバックし、ローカル名とインターネット名を共存させる。
- [推奨] mDNSは補助用途に限定し、運用本線はCoreDNSへ統一する。
- [重要] MQTT / API / OTA / NTP の各接続先は、`HostName(FQDN)` と `HostIp` の両方を設定として保持する。  
  理由: 名前解決できる環境ではFQDN優先で運用し、障害切り分けや暫定継続時はIP直指定へ即時切替できるようにするため。
- [厳守] `MQTT over TLS` と `HTTPS OTA` の正規接続先名は FQDN を基準に管理する。  
  理由: 証明書検証の正規名を失わず、将来のIP変更でも再書込みを最小化するため。
- [推奨] `NTP` は TLS 名検証要件がないため、障害切り分け時は IP直指定を許容する。  
  理由: 起動時刻同期を早期復旧し、MQTT接続開始条件の切り分けを先に進めやすいため。

### 2.2 AWS移行時
- [重要] AWS IoT Core接続時はAWSが発行するFQDN（ATSエンドポイント）を使用する。
- [厳守] 証明書検証（SAN/CN一致）を有効化し、IP直指定を常用しない。
- [推奨] 事前検証時のみIP接続を許容し、本番ではDNS名運用へ統一する。

### 2.3 WindowsをDNSサーバーとして使う場合のFW要件（2026-03-07追記）
- [重要] 結論: Windows側の受信ファイアウォール設定は必要。  
  理由: ESP32の`hostByName()`はDNSサーバーへ問い合わせるため、受信が閉じていると名前解決が失敗するため。
- [厳守] DNSサーバーPCで `UDP/53` と `TCP/53` の受信を許可する。  
  理由: 通常問い合わせはUDPだが、応答サイズや再送時にTCPへフォールバックするため。
- [厳守] DNSサービスは localhost 限定待受にしない（LAN向けIPで待受する）。  
  理由: localhost待受のみではLAN端末（ESP32含む）から問い合わせできないため。
- [推奨] ルール適用プロファイルは `Private` を基本とし、`Public` は原則許可しない。  
  理由: 必要最小限公開の原則を守るため。
- [禁止] DNS疎通未確認のままTLS/認証起因と断定しない。  
  理由: DNS未解決時はTLSハンドシェイク以前で失敗し、原因誤認を招くため。

#### 2.3.1 Windows受信ルール例（PowerShell管理者）
```powershell
New-NetFirewallRule -DisplayName "DNS-UDP-In-Private" -Direction Inbound -Action Allow -Protocol UDP -LocalPort 53 -Profile Private
New-NetFirewallRule -DisplayName "DNS-TCP-In-Private" -Direction Inbound -Action Allow -Protocol TCP -LocalPort 53 -Profile Private
```

#### 2.3.2 疎通確認手順
1) DNSサーバーIPを明示して確認する。  
`nslookup mqtt.esplab.home.arpa <DNSサーバーIP>`
2) ESP32と同一セグメント端末で 1) が成功することを確認する。  
3) ESP32シリアルログで `hostByName failed` が消えることを確認する。

### 2.4 新ルータ導入後のローカルDNS設計（2026-03-07追記・2026-03-13更新）
- [重要] 新ルータは GW `192.168.1.1` で運用する（2026-03-13 時点で一部導入済み）。  
  理由: IoT用セグメントの見通しを高めるため。旧案 `172.17.1.x` から `192.168.1.x` へ移行。
- [厳守] DNSサーバーPC（CoreDNS）は `192.168.1.100` 固定とする。  
  理由: DHCP割当変動で名前解決基盤が揺れないようにするため。
- [厳守] ESP32等クライアントのDHCP配布レンジは `192.168.1.200` 以降とする。  
  理由: インフラ固定IP帯（GW/DNS/サーバー）と衝突させないため。
- [厳守] `mqtt.esplab.home.arpa` は `192.168.1.100` へ固定割当てする。  
  理由: 現行ローカル管理サーバー兼CoreDNSホストへ名前解決を集約し、FQDN運用を安定化するため。
- [推奨] DHCP配布DNSはプライマリを `192.168.1.100`（CoreDNS）にする。  
  理由: ローカルゾーン解決を常にCoreDNS優先にするため。
- [推奨] セカンダリDNSは空欄またはCoreDNS同一を優先し、外部DNSを直接配布しない。  
  理由: `*.esplab.home.arpa` を外部DNSへ逃がすとNXDOMAINが返る可能性があるため。

### 2.5 新ルータ運用開始後のESP32確認方針（2026-03-08追記・2026-03-13更新）
- [重要] 新ルータ運用（GW `192.168.1.1`、サーバー `192.168.1.100`、DHCP `192.168.1.200+`）を開始した。  
  理由: ローカルFQDN運用を本番相当で確認するため。
- [厳守] ESP32 は `mqtt.esplab.home.arpa` を DNS 名で解決してから接続成功すること。  
  理由: IP直指定やフォールバック接続では、FQDN運用とTLSホスト名前提の確認にならないため。
- [厳守] `SENSITIVE_MQTT_FALLBACK_IP` は空文字で無効化し、DNS未解決時に接続成功してはならない。  
  理由: DNS不備を隠したまま運用開始する事故を防ぐため。
- [推奨] 実機確認では `hostByName(mqtt.esplab.home.arpa)` 成功ログ、`pingBrokerHost success (tcp-probe)`、MQTT接続成功ログを1セットで確認する。  
  理由: 「名前解決成功」「到達性成功」「MQTT接続成功」を切り分けて判断できるため。

#### 2.4.1 CoreDNS運用ディレクトリ
- [重要] CoreDNS設定ファイルの配置先は `C:\coredns\Corefile` とする。
- [厳守] `Corefile` で `mqtt.esplab.home.arpa` と `ntp.esplab.home.arpa` のAレコードを管理する。
- [推奨] ローカル定義以外は `forward` で上位DNSへ委譲する。

## 3. 到達性（グローバルIP要否）方針
### 3.1 結論
- [重要] AWS化してもESP32にグローバルIPは原則不要。
- 理由: ESP32はNAT配下からクラウドへ「アウトバウンド接続」し、クラウド要求は既存MQTTセッション経由で受信できるため。

### 3.2 要求経路
- クラウド -> デバイス要求は「MQTTトピックPublish」で実施する（クラウドからESPへ直接TCP接続しない）。
- デバイス -> クラウドはMQTT over TLS（`8883`）で送信する。
- OTAは「MQTT指令で開始し、デバイスからHTTPSで取得する（pull型）」を採用する。

### 3.3 例外
- [将来対応] どうしても直アクセスが必要な保守要件が出た場合のみ、VPN/踏み台経由を設計し、直接公開は回避する。

## 4. 常駐起動（ログイン前起動）方針
### 4.1 `LocalServer`（TypeScript/Node.js）
- Windows: `Task Scheduler` で「PC起動時」トリガーを登録し、自動起動を有効化する。
- macOS: [将来対応] `launchd` の `LaunchDaemon` を採用する。
- [厳守] アプリプロセスはサービス監視（再起動ポリシー）を設定する。

### 4.1.1 `SecretCore`（Rust）
- Windows: `LocalServer` 起動時にローカルIPC専用プロセスとして起動する。
- macOS: [将来対応] `LocalServer` と同等のローカルIPC前提で起動する。
- [厳守] Named Pipe / Unix Domain Socket 等のローカルIPCのみを使用し、TCP待受を開けない。

### 4.2 [廃止の方針] LocalSoft（Python）
- [新規利用禁止] 新規導入しない。既存環境は `LocalServer` 一体化へ移行する。

## 5. データ保存方針（LocalServer）
- [重要] SQLiteを採用する（単一端末・小規模台数に適合）。
- 保存対象: 受信テレメトリ、コマンド実行結果、監査ログ。
- [厳守] 保存期間を設定可能にする（例: 30/90/365日）。
- [厳守] 「検索」「出力」「削除（論理/物理）」をAPI化し、監査ログに記録する。
- [将来対応] 台数・書込量増大時はTimescaleDB等へ移行を検討する。

## 6. ESP32公開面（ポート）方針
- [厳守] 通常運用時、ESP32はインバウンド待受を原則行わない。
- [厳守] 利用ポートは最小化する。
  - MQTT over TLS: `8883`（アウトバウンド）
  - NTP: `123/UDP`（アウトバウンド）
  - HTTPS OTA: `443`（アウトバウンド）
- [新規利用禁止] 平文HTTPの恒久運用。
- [推奨] APメンテナンスモード時のみ閉域でローカルHTTP/HTTPSを一時有効化し、終了後は必ず停止する。

## 7. DDoS/不正アクセス対策
### 7.1 基本原則
- [厳守] ESP32をインターネットへ直接公開しない（NAT内 + outbound専用）。
- [厳守] MQTTはTLS + 認証必須（将来mTLS拡張）。
- [厳守] 認可（Topic ACL）をデバイス単位で適用する。

### 7.2 サーバー側対策（Local/Cloud共通）
- 接続レート制限（IP単位/ID単位）
- 認証失敗時のバックオフ、一定回数超過で一時遮断
- WAF/リバースプロキシ（HTTPS入口）
- 監査ログ収集とアラート閾値設定（接続数急増、失敗率急増）
- 証明書・鍵ローテーション運用

### 7.3 追加運用対策
- [推奨] Broker管理ポートは管理ネットワークのみに制限する。
- [推奨] Firewallで送信先を必要最小限に絞る（NTP/HTTPS/MQTT先のみ）。
- [推奨] 時刻同期失敗時の挙動（証明書検証失敗時のリトライ制御）を明確化する。

### 7.4 インターネット接続時の最低限対策
- [厳守] ESP32、LocalServer、Broker の通常運用経路以外の待受は停止し、未使用ポートを閉鎖する。
- [厳守] Broker 管理ポート、管理UI、`ProductionTool` 入口、開発用API、`SecretCore` のIPC終端はインターネットへ直公開しない。
- [厳守] Firewall やルータ設定で、送信先と受信元を必要最小限へ制限する。
- [厳守] 認証失敗時のバックオフ、一時遮断、監査ログ記録を有効化する。
- [厳守] Device ごとの Topic ACL と、管理者向け API の認可境界を分離する。
- [禁止] 本プロジェクト要件外の汎用プロキシ、中継、ファイル共有、ポートフォワード機能を同居させない。
- [推奨] 監査ログは接続数急増、認証失敗急増、遮断発生を検知できる粒度で保存し、定期確認する。
- 理由: 侵入後に IoT 機器や管理サーバーが第三者攻撃の踏み台として悪用されることを防ぐため。

## 8. 疑問事項に対する結論（2026-03-07時点）
- Q1. AWS化したらデバイスはグローバルIP必須か  
  - A. 不要（原則）。NAT配下のアウトバウンドMQTT接続で運用する。
  - 補足: ESP32はLAN内のプライベートIPのまま、クラウド宛てに自分から接続を確立する。外部からESP32へ直接入る経路を作らない。
- Q2. どうやってアクセスするか  
  - A. クラウドからMQTTトピックで要求し、デバイスはサブスクライブで受信する。
  - 補足: 「クラウド -> Broker -> デバイス」のメッセージ配信で制御する。クラウドからデバイスへの直接TCP接続は不要。
- Q3. 名前アクセスはどうなるか  
  - A. ローカルはCoreDNS、クラウドはAWS IoTエンドポイントFQDNを使用する。
  - 補足: TLS証明書のホスト名検証（SAN/CN一致）はFQDN前提で運用する。IP直指定は本番で常用しない。
- Q4. Windows/macOSでログインしていなくても自動起動できるか  
  - A. 可能。Windows Service / macOS LaunchDaemonで実現する。
  - 補足: いずれもOSのサービス管理機能で、ユーザーログイン前にプロセス起動と再起動監視ができる。

### 8.1 FQDNとは何か
- FQDN（Fully Qualified Domain Name）は、DNS階層で対象を一意に特定する完全修飾名。
- 例（ローカル）: `mqtt.iot.local`
- 例（AWS）: `xxxxxxxxxxxx-ats.iot.ap-northeast-1.amazonaws.com`
- [重要] 本プロジェクトでは通信先を原則FQDNで管理し、接続先IPはDNS解決結果に委譲する。

### 8.2 FQDNを実現する技術（DNSの構成要素）
- スタブリゾルバ: 端末（ESP32/Windows/macOS）側のDNS問い合わせ機能。
- 再帰リゾルバ: 問い合わせを代行するDNSサーバー（CoreDNSや上位DNS）。
- 権威DNS: 対象ドメインの正解レコード（A/AAAA/CNAMEなど）を保持するサーバー。
- キャッシュ（TTL）: 同一問い合わせを高速化し、不要な再問い合わせを抑制する仕組み。

### 8.3 FQDN解決からTLS確立までの原理
- 手順1: クライアントはFQDNをDNSへ問い合わせ、IPを取得する。
- 手順2: 取得したIPへTCP接続し、TLSハンドシェイクを開始する。
- 手順3: サーバー証明書のSAN/CNが「接続に使ったFQDN」と一致するか検証する。
- 手順4: 検証成功時のみMQTT接続を継続し、失敗時は接続を中断する。
- [厳守] ホスト名検証を無効化しない。

### 8.4 なぜIP直指定ではなくFQDNを使うか
- AWS側は負荷分散や冗長化で接続先IPが変動し得るため、固定IP前提は運用破綻しやすい。
- FQDN運用ならDNSが最新の到達先へ誘導できる。
- TLSの正当性確認（証明書名一致）を安全に成立させやすい。

### 8.5 本プロジェクトでの適用ルール
- ローカル: CoreDNSで`iot.local`ゾーンを管理し、外部ドメインは`forward`で上位DNSへ委譲する。
- クラウド: AWS IoT CoreのATSエンドポイントFQDNを使用する。
- [厳守] 設定値は `mqttHost=<FQDN>` を基本とし、`mqttIp=<固定IP>` は検証用途のみに限定する。
- [禁止] 本番運用で証明書検証回避（CN/SAN不一致許容）を行わない。

### 8.6 関係図（ローカル / クラウド / インターネット / FQDN）
[重要] クラウド化しても、ローカルWebサーバーをインターネットへ直接公開することは原則不要。  
理由: 攻撃面を最小化し、運用をクラウド公開面へ集約するため。

```text
[インターネット利用者]
        |
        | HTTPS(443)
        v
+--------------------------------------------------+
| インターネット公開領域（クラウド）               |
|  - Web/API FQDN: app.example.com                |
|  - MQTT   FQDN: iot.example.com:8883            |
|  - OTA配布FQDN: ota.example.com:443             |
|  - または AWS IoT ATS Endpoint FQDN             |
+--------------------------------------------------+
                 ^                        |
                 | MQTT over TLS(8883)   | Publish/Subscribe + OTA(HTTPS:443)
                 |                        v
        +--------------------------------------------------+
        | ローカルLAN（非公開）                           |
        |  ESP32 (private IP)  ----MQTT接続(常時)-------> |
        |     outbound only                               |
        |  ESP32 (OTA時) --------HTTPS取得(都度)-------> |
        |                                                  |
        |  LocalServer (private IP)                       |
        |  CoreDNS (local FQDN: *.iot.local)              |
        +--------------------------------------------------+
```

- ローカルFQDN（例: `mqtt.iot.local`）はCoreDNSが解決し、LAN内のみで有効。
- クラウドFQDN（例: `iot.example.com`、`ota.example.com`、`<ats-endpoint>.amazonaws.com`）は公開DNSが解決。
- ESP32は原則「外向き接続のみ」で運用し、インターネットからの直接到達経路を作らない。
- [例外方針][将来対応] どうしても外部からローカル画面へ入る要件がある場合、VPNまたはゼロトラストトンネルを使用し、ポート直公開を避ける。

### 8.7 OTA通信の詳細（MQTT指令 + HTTPS配布）
- [厳守] OTA開始トリガーはMQTTコマンドで受信する（例: `ota/start`）。
- [厳守] ESP32はOTAバイナリをHTTPS（`443`）で自分から取得する（pull型）。
- [厳守] 取得先URLはFQDNで指定し、TLS証明書のホスト名検証を有効化する。
- [推奨] OTA手順は「manifest取得 -> 署名/ハッシュ検証 -> firmware取得 -> 書込 -> 再起動 -> 結果通知」とする。
- [厳守] 失敗時は旧ファームへロールバックし、結果をMQTTで通知する。

```text
1) Cloud/Local管理系 -> MQTT Publish (OTA開始指令)
2) ESP32 -> MQTT Subscribe で指令受信
3) ESP32 -> HTTPS GET https://ota.example.com/manifest.json
4) ESP32 -> manifestの署名/ハッシュを検証
5) ESP32 -> HTTPS GET https://ota.example.com/firmware.bin
6) ESP32 -> 書き込み/検証/再起動
7) ESP32 -> MQTT Publish (成功/失敗ステータス)
```
