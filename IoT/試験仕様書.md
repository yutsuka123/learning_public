# IoT 試験仕様書

[重要] 本書は LocalServer 初期実装の疎通・機能試験仕様を定義する。  
理由: 実装後の合否判定を再現可能にするため。

## 1. 試験方針
- 試験番号は4桁（上1桁: 分類、下3桁: 連番）で管理する。
- 7000番台をネットワーク/サーバー系試験として利用する。
- [厳守] 実施結果は `試験記録書.md` に1対1で記録する。

## 2. 試験項目
- 7001: LocalServer 起動試験
  - 手順: `cd IoT/LocalServer && npm install && npm run build && npm run start`
  - 期待結果: `HTTP started` が出力され、`/api/health` が `status=ok` を返す。
- 7002: MQTT 接続試験
  - 手順: `cd IoT/LocalServer && npm run test:connect`
  - 期待結果: MQTTへ接続し `esp32lab/call/status/all` publish成功を出力する。
- 7003: Web GUI 一覧表示試験
  - 手順: `http://localhost:3100/` を開き、`/api/devices` 応答確認。
  - 期待結果: テーブル描画ができ、操作ボタンが押下可能。
- 7004: statusコマンド送信試験
  - 手順: `POST /api/commands/status` に `{"targetNames":"all"}` を送信。
  - 期待結果: `result=OK` が返り、MQTT publishされる。
- 7005: OTAコマンド送信試験
  - 手順: `POST /api/commands/ota` に `{"targetNames":"all"}` を送信。
  - 期待結果: `result=OK` が返り、`otaStart` publishされる。
- 7006: OTA配布エンドポイント試験
  - 手順: `/ota/manifest.json` と `/ota/firmware.bin` を取得。
  - 期待結果: manifest取得成功。firmware未配置時は404を返す。
- 7007: OTA進捗通知試験（デバイス別）
  - 手順: OTA実行中に `esp32lab/notice/otaProgress/+` を購読し、LocalServer画面を確認。
  - 期待結果: デバイス別に進捗率（例: 30%）が表示される。
- 7008: OTA同一進捗率リトライ試験
  - 手順: 通信を劣化させ、同一進捗率で停止する状況を作る。
  - 期待結果: 5秒間隔で最大3回まで同一進捗率リトライを行う。
- 7009: OTA最初から再試行試験
  - 手順: 7008で回復不可条件を維持する。
  - 期待結果: 最初から再試行を最大3回実施し、上限超過でNG通知する。
- 7010: SHA256一致試験
  - 手順: 正常な `firmware.bin` と一致するSHA256を配布指令へ設定する。
  - 期待結果: SHA256一致後のみ更新確定し、次回起動先が更新領域へ切替わる。
- 7011: SHA256不一致試験（異常系）
  - 手順: 意図的に誤SHA256を `otaStart` に設定する。
  - 期待結果: `shaMismatch` としてNG終了し、更新確定しない。
- 7012: OTA再認証試験
  - 手順: Web UI または API 経由で OTA実行前に管理者パスワードを再入力する。
  - 期待結果: 再認証成功後のみ OTA開始操作が許可される。
- 7013: OTA再認証セッション有効期限試験
  - 手順: 7012 成功後、同一ブラウザセッション内で 3時間未満と 3時間超過後の双方で OTA開始可否を確認する。
  - 期待結果: 3時間以内は再入力不要、3時間超過後は再認証が再度要求される。
- 7014: OTA HTTPS証明書検証試験
  - 手順: 正常証明書と異常証明書（ホスト名不一致、信頼外CA等）で OTA取得を試行する。
  - 期待結果: 正常証明書のみ取得成功し、異常証明書は拒否される。
- 7015: OTAデュアルパーティション交互切替試験
  - 手順: 連続2回以上 OTA を実施し、各回の書込み先パーティションを確認する。
  - 期待結果: 書込み先が 0 / 1 の2面で交互に切り替わる。
- 7016: メーカー署名済み配布物実行試験
  - 手順: メーカー署名済み `firmware.bin` を LocalServer へ配置し、顧客環境相当の LocalServer から OTA を実行する。
  - 期待結果: 署名済み配布物のみ OTA が開始・完了できる。
- 7017: 未署名配布物拒否試験
  - 手順: 未署名または署名情報欠落の `firmware.bin` を用意し、OTA 実行を試みる。
  - 期待結果: LocalServer または ESP32 側で実行拒否し、更新処理へ進まない。
- 7018: 改ざん配布物拒否試験
  - 手順: 署名済み配布物の一部バイトを改変した `firmware.bin` を用意し、OTA 実行を試みる。
  - 期待結果: 署名検証または整合性検証で NG となり、更新確定しない。
- 7019: AP 共通トップ画面認証試験
  - 手順: AP 接続後に共通トップ画面へアクセスし、未認証、誤パスワード、正パスワードの各条件で表示可否を確認する。
  - 期待結果: 正しい共通パスワード成功時のみ共通トップ画面が表示される。
- 7020: Production 追加認証試験
  - 手順: 共通トップ画面から Production 画面へ遷移し、通常管理者パスワード、誤メーカーモード専用パスワード、正しいメーカーモード専用パスワードの各条件で遷移可否を確認する。
  - 期待結果: Production 画面は追加認証成功時のみ表示され、通常管理者パスワードでは代替できない。
- 7021: Production 画面段階実行試験
  - 手順: Production 画面を起動し、事前確認、対象機確認、実行内容確認、最終確認、段階実行、完了判定の各画面遷移を確認する。
  - 期待結果: 各段階で確認操作が要求され、最終確認完了前には eFuse 処理が開始されない。
- 7022: eFuse 書込み中断後の再判定試験
  - 手順: Production 側の eFuse 処理中に電源断、通信断、内部例外を模擬し、再起動後に再判定フローを確認する。
  - 期待結果: eFuse 読戻し結果から残工程または隔離判定へ進み、通常顧客モードの継続処理へ戻らない。
- 7023: eFuse 専用処理タスクのスタック余裕試験
  - 手順: eFuse 専用処理タスクについて、平常時と最悪条件時の `uxTaskGetStackHighWaterMark` 相当値、空きヒープ、開始可否判定を確認する。
  - 期待結果: 定義した安全閾値以上のスタック余裕があり、閾値未達またはヒープ不足時は eFuse 開始を拒否する。
- 7024: eFuse 本番前リハーサル完了確認試験
  - 手順: 本番対象の手順・ファームウェア系統・鍵IDに対応する試験機リハーサル記録の有無を確認し、未完了状態で本番開始を試みる。
  - 期待結果: 対応するリハーサル記録が存在する場合のみ本番開始可能とし、未完了時は開始拒否する。
- 7025: OTA途中失敗時の旧パーティション継続起動試験
  - 手順: OTA の途中失敗、整合性不一致、未確定起動失敗の各条件を作り、再起動後の起動パーティションと稼働版数を確認する。
  - 期待結果: 旧パーティションから継続起動し、直前まで正常稼働していた版数で運用継続できる。
- 7026: eFuse 最終化後の OTA バージョンアップ試験
  - 手順: eFuse / Secure Boot / Flash Encryption 有効化後の個体に対し、証明書検証有効のまま署名済み `firmware.bin` を OTA 配布し、版数更新を確認する。
  - 期待結果: eFuse 最終化後も OTA による更新が成功し、更新後版数で正常起動する。
- 7027: ESP32 設定永続保存試験
  - 手順: Wi-Fi / MQTT / HTTPS / 時刻同期 / その他設定を更新し、再起動後に NVS から復元されることを確認する。
  - 期待結果: 設定値が再起動後も保持され、主要運用設定の主保存先が NVS であることを確認できる。
- 7028: APメンテナンスモード設定修正試験
  - 手順: Wi-Fi AP または MQTT ブローカーの接続先を意図的に変更し、ESP32 をメンテナンス用 Wi-Fi AP モードへ移行して Web UI から設定修正を行う。
  - 期待結果: メンテナンスモード経由で設定修正・保存でき、通常運用へ復帰できる。
- 7029: 新ルータ運用開始後の DNS / MQTT 起動前確認試験
  - 手順: `WiFi AP=172.17.1.1` `DNS=172.17.1.100` 環境で ESP32 を書き込み起動し、`mqtt.esplab.home.arpa` と `ntp.esplab.home.arpa` の名前解決、NTP 同期、MQTT 接続開始条件をログで確認する。
  - 期待結果: `hostByName()` が DNS 名を解決し、NTP 同期後に MQTT 接続処理へ進める。`SENSITIVE_MQTT_FALLBACK_IP` に依存しない。
- 7030: 通常運用FWのシリアル出力制約試験
  - 手順: 量産通常FWで起動し、シリアル高詳細ログ、危険な保守コマンド、秘密値出力の有無を確認する。
  - 期待結果: 通常運用に不要な詳細シリアル出力や危険な保守コマンドは停止または最小化され、秘密値が出力されない。
- 7031: 診断用FWの一時シリアル有効化試験
  - 手順: メーカー管理下の署名済み診断用FWを書き込み、一時的にシリアル出力で障害解析できることと、通常FWへ戻した後に再び制約されることを確認する。
  - 期待結果: 診断時のみ必要なシリアル出力が可能で、通常FWへ戻すと恒久的な開放状態が残らない。
- 7032: 未使用ポート閉鎖確認試験
  - 手順: ESP32、LocalServer、Broker の稼働状態で利用予定外の待受ポートを列挙し、閉鎖状態を確認する。
  - 期待結果: 設計上必要なポート以外は待受しておらず、開発用・管理用ポートが本番相当構成で露出していない。
- 7033: 認証失敗時のバックオフ・一時遮断試験
  - 手順: MQTT 認証や管理系 API 認証を連続失敗させ、バックオフ、一定回数超過時の一時遮断、監査ログ記録を確認する。
  - 期待結果: 連続失敗に応じて抑止動作が入り、総当たり的な試行がそのまま継続できない。
- 7034: インターネット接続時の踏み台化防止設定確認試験
  - 手順: 本番相当ネットワークで、管理ポートの非公開、不要待受停止、汎用中継機能無効、送信先制限、監査ログ記録の有無を確認する。
  - 期待結果: 本プロジェクト要件外の中継機能が有効になっておらず、不要な公開面が残っていない。
- 7035: `createPairingBundle` 必須設定事前検証試験
  - 手順: Wi-Fi / MQTT / OTA / 認証情報のいずれかを欠落させた入力で bundle 作成を要求し、TS 側の事前検証結果を確認する。
  - 期待結果: `SecretCore` 呼出し前にエラーとなり、bundle 生成が開始されない。NTP 欠落のみは許容される。
- 7036: `createPairingBundle` AP 初回投入試験
  - 手順: AP モード接続した未ペアリング個体へ `public_id`、`k-device`、`keyVersion`、Wi-Fi / MQTT / OTA / 認証情報を含む bundle を投入する。
  - 期待結果: ESP32 が署名検証、ECDH セッション復号、NVS 保存を行い、通常運用へ移行する。
- 7037: `k-user` 再発行後の current/previous 2 スロット試験
  - 手順: `k-user` 再発行後に 1 台を再ペアリングし、ESP32 側の `currentKeyVersion/currentKDevice` と `previousKeyVersion/previousKDevice` の更新状態を確認する。
  - 期待結果: 再ペアリング直後に `previousKeyState=grace` となり、新旧 2 スロットが正しく更新される。
- 7038: 48h / 240h 稼働時間による previous key 状態遷移試験
  - 手順: 並存期間中に runtime 積算を進め、48h 到達時と 240h 到達時の状態遷移と NVS 更新タイミングを確認する。
  - 期待結果: 48h で `expired-retained` へ遷移して旧鍵認証を拒否し、240h 後に previous key を自動削除し、結果は監査ログのみに残る。
- 7039: `runPairingSession()` workflow 状態遷移試験
  - 手順: AP モード初回投入を実施し、`workflowId` に対する状態遷移を取得する。
  - 期待結果: `queued` -> `running` -> `waiting_device` -> `verifying` -> `completed` の順で遷移し、TS 側は進捗と結果のみを表示する。
- 7040: `runKeyRotationSession()` 完了判定試験
  - 手順: `k-user` 再発行後の再ペアリングを workflow 経由で実施し、個体単位の完了判定を確認する。
  - 期待結果: `SecretCore` が再ペアリング開始から完了判定まで実行し、TS 側は `completed` または `failed` の結果のみを扱う。
- 7041: `runProductionSecureFlow()` 段階実行試験
  - 手順: Production 高リスク処理を workflow 経由で開始し、段階実行、読戻し検証、中断後再判定を確認する。
  - 期待結果: `SecretCore` が段階実行と完了判定を担当し、TS 側は進捗状態と結果のみを表示する。
- 7042: `runSignedOtaCommand()` 署名付き OTA 開始試験
  - 手順: OTA 開始を workflow 経由で実施し、署名付き開始指示、送達、検証、完了判定を確認する。
  - 期待結果: `SecretCore` が OTA 開始ワークフローを実行し、TS 側は中間秘密を扱わず進捗と結果のみを扱う。
- 7043: workflow 異常系 `failed` 遷移試験
  - 手順: AP 未接続、署名検証失敗、NVS 保存失敗などの異常条件を作り、workflow の失敗遷移を確認する。
  - 期待結果: `queued` / `running` / `waiting_device` / `verifying` のいずれかから `failed` へ遷移し、TS 側は内部手順を再構成せずエラー表示のみを行う。
- 8001: 異常系（MQTT認証失敗）
  - 手順: `.env` の `MQTT_PASSWORD` を誤値にして起動。
  - 期待結果: 接続失敗ログが出力され、プロセスが異常終了しない。
