# IoT ストレージ仕様書

[重要] 本書は ESP32-S3 の `NVS` と `LittleFS` の保存区分、更新方式、ログ保持方式を定義する。  
理由: 設定値、機密、ファイル資産、運用ログの保存先が文書ごとにぶれないようにするため。

## 1. 基本方針
- [厳守] 主要運用設定の主保存先は `NVS` とする。
- [厳守] ファイル性を持つ資産の保存先は `LittleFS` とする。
- [禁止] 機密を含む主要接続設定の主保存先を、平文の `LittleFS` 設定ファイルへ戻さない。
- [推奨] 実体ファイルは `LittleFS`、版管理や参照状態は `NVS` に持つハイブリッド構成を採用する。

## 2. 保存領域の責務
### 2.1 `NVS`
- [厳守] 以下を `NVS` に保存する。
  - 鍵情報全般
  - Wi-Fi 接続設定
  - MQTT 接続設定
  - HTTP / HTTPS 接続先設定
  - NTP 接続設定
  - ユーザー名 / パスワード / トークン等の認証情報
  - 証明書の版数、SHA-256、有効フラグ、最終更新日時
  - ログ保持設定、起動回数、時刻同期状態
- [理由] 少量かつ参照頻度の高い設定値を一貫した形式で管理しやすいため。

### 2.2 `LittleFS`
- [厳守] 以下を `LittleFS` に保存する。
  - 画像ファイル
  - MQTT / HTTPS 用証明書ファイルおよび秘密鍵ファイル
  - ファイルログ
- [厳守] 保存先ディレクトリは少なくとも以下に分離する。
  - `/images`
  - `/certs`
  - `/logs`
- [禁止] 画像更新や証明書更新時に `/logs` を巻き込んで全消去しない。

## 3. 証明書保存方針
- [重要] 現状の FW 埋め込み証明書は、更新容易性を優先して `LittleFS` の `/certs` へ移行する。
- [厳守] 対象は CA 証明書、クライアント証明書、秘密鍵ファイルを区別せず、必要なものを `/certs` に格納できるようにする。
- [厳守] 証明書実体の主保存先は `LittleFS` とし、`NVS` には以下のメタ情報のみ保持する。
  - アクティブ証明書識別子
  - SHA-256
  - 版数
  - 最終更新日時
- [禁止] 証明書実体や秘密鍵実体を、移行完了後も恒久的に FW へハードコードし続けない。
- [推奨] ローカル更新経路は `MQTT` を第一候補、`AP` モードを第二候補とする。

## 4. 画像一斉更新方針
### 4.1 通信経路
- [厳守] 第一候補は `MQTT` とする。
- [推奨] 現地保守時は AP モードの HTTP アップロードでも同等の更新意味論を提供する。
- [厳守] 大容量ファイル転送中も、更新対象外ファイルと `/logs` は保持する。

### 4.2 更新単位
- [厳守] 更新単位はファイル単位とする。
- [厳守] `LittleFS` 全面書換ではなく、差分更新とする。
- [厳守] 操作種別として少なくとも以下を提供する。
  - 追加
  - 差替
  - 削除
  - 全削除
- [厳守] 更新対象ディレクトリは `/images` を既定とする。

### 4.3 整合性確認
- [厳守] 更新対象ファイルごとに `SHA-256` を必須とする。
- [厳守] ファイル受信完了後に ESP32 側で `SHA-256` を再計算し、一致時のみ反映する。
- [禁止] ハッシュ未一致時に新ファイルへ切り替えない。

### 4.4 失敗時方針
- [厳守] 失敗時は旧ファイル維持とする。
- [厳守] 書込みは `tmp` ファイルへ行い、検証成功後に `rename` で本番ファイルへ切り替える。
- [禁止] 途中失敗時に旧ファイルを先に削除しない。

## 5. `MQTT` 一斉更新コマンド方針
- [重要] 画像および証明書更新には、共通の `fileSync` 系コマンドを採用する。
- [厳守] `fileSync` 系コマンドは少なくとも以下の段階を持つ。
  - `fileSyncPlan`: 対象領域、削除方針、ファイル一覧、SHA-256 を提示する
  - `fileSyncChunk`: 必要ファイルのみをチャンク転送する
  - `fileSyncCommit`: 検証成功ファイルの反映と削除適用を行う
- [厳守] `targetArea` は少なくとも `images` と `certs` を許容する。
- [厳守] ESP32 は `fileSyncPlan` 受信時に既存ファイルの SHA-256 を比較し、転送が必要なファイルだけを要求する。
- [推奨] `deleteMode` は `none` / `listed` / `all` を許容する。
- [厳守] 画像更新系コマンド（`fileSync*`）は OTA と同等に HMAC/署名検証を必須とする。

### 5.1 ZIP一括更新（将来対応）
- [重要] 複数画像をZIP化して一括反映する `imagePackageApply` 方式を将来対応として許容する。
- [厳守] MQTT コマンドをトリガに、ESP32 は HTTPS で ZIP を取得する。
- [厳守] ZIP の `SHA-256` 一致確認後にのみ展開を開始する。
- [厳守] 展開先ディレクトリは `destinationDir` で指定し、`/images` 配下に限定する。
- [厳守] 同名ファイルは `overwrite=true` の場合のみ上書きする。
- [禁止] `/logs` への展開・上書きを許可しない。
- [推奨] OTA同様に `download -> verify -> stage(tmp) -> apply(rename)` の段階適用を行う。

## 6. AP モード更新方針
- [厳守] AP モード時も `/images` と `/certs` に対して局所更新できるようにする。
- [厳守] AP モード更新でも `SHA-256` 検証、`tmp` 書込み、成功後 `rename` を適用する。
- [厳守] AP モード更新の認証は、共通トップ画面認証後にのみ許可する。
- [推奨] 証明書更新は AP モードからも実行可能とし、現地復旧性を確保する。

## 7. ログ保存方針
### 7.1 基本方針
- [厳守] ファイルログはシリアル出力とは別経路・別関数で実装する。
- [厳守] ファイルログは重要ログのみを必要に応じて ON にできるようにする。
- [推奨] 初期状態は常時高詳細ログを OFF とし、試験後は必要時のみ有効化する。

### 7.2 保存先
- [厳守] 時刻同期済みの場合のパス形式は以下とする。
  - `/logs/<年>/<月日>/<ファイル名>.log`
  - 例: `/logs/2026/0309/202603091103001-1.log`
- [厳守] 時刻同期できない場合は以下とする。
  - `/logs/<起動回数>/00001.log`
- [重要] `<ファイル名>` は `年月日時分秒ms-連番` 形式を基本とする。

### 7.3 ローテーション
- [厳守] 再起動時に新しいログファイルへ切り替える。
- [厳守] 1ファイルあたり 1KB 到達時に新ファイルへ切り替える。
- [推奨] 連番は同一日・同一路径内で採番する。

### 7.4 削除条件
- [厳守] 時刻同期が取れている場合は 30 日超過ログを削除対象とする。
- [厳守] ログ総量が 100KB に達した場合は、もっとも古いログから順に削除する。
- [推奨] 30 日超過削除と総量超過削除の両方を適用し、どちらか早い条件で整理する。

## 8. ディレクトリ構成
- `/images`
  - 画面表示用画像、アイコン、静的表示資産
- `/certs`
  - `mqtt-ca.pem`、`mqtt-client.crt`、`mqtt-client.key` 等の証明書/鍵ファイル
- `/logs`
  - 日付または起動回数ベースのログファイル

## 9. 更新時の禁止事項
- [禁止] 画像更新のたびに `LittleFS` 全面初期化を行わない。
- [禁止] ログ保持領域を画像更新の副作用で消去しない。
- [禁止] 証明書の更新完了前に旧証明書を破棄しない。
- [禁止] 機密を含む主要接続設定を `LittleFS` の平文 JSON 主保存へ戻さない。
