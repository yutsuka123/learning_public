# 鍵管理・初期セットアップ 実装たたき台

[重要] 本書は `鍵管理および初期セットアップ設計仕様書.md` を実装へ落とし込むためのドラフトである。  
理由: `TPM + wrapped_secret + HKDF` 方式の実装順序と責務分担を先に固定し、手戻りを防ぐため。

[厳守] 本書には鍵実値・秘密情報を記載しない。  
理由: 公開リポジトリ運用のため。

## 1. スコープ（第1版）
- 対象:
  - `S_random` 生成
  - TPM による `wrapped_secret` 生成・保存
  - `wrapped_secret` 復号による `k-user` 再生成
  - `k-device` 導出
  - `createPairingBundle` 生成
  - Wi-Fi更新での暗号 payload 生成/受信準備
  - 同一PC再インストール時の `wrapped_secret` / `device_db` 復元
- 対象外（次版）:
  - TPM 非搭載環境向け代替方式
  - macOS 向け同等ハードウェア拘束方式
  - クラウドKMS連携

## 1.1 責任範囲の親定義
- [厳守] `Production` / `LocalServer` / `SecretCore` / `ESP32` / `LocalSoft` の責任境界は `モジュール仕様書.md` を親定義とする。
- [厳守] 本書はそのうち `LocalServer` と `SecretCore` の実装たたき台を主対象とし、`Production` の eFuse 実行主体や `LocalSoft` の保存主体を混在させない。

## 1.2 高リスクワークフロー方針
- [厳守] TS は高リスク処理の開始要求、入力検証、進捗表示、結果表示のみを担当する。
- [厳守] `SecretCore` は対ESP32通信開始から完了判定までを Rust 主導ワークフローとして実行する。
- [厳守] ワークフロー名は `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` を基本とする。
- [厳守] 進捗状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` を正規値とする。

## 2. 実装フェーズ案
### Phase 1: TPM 初期化基盤
- セットアップ処理で `S_random` を CSPRNG 生成する。
- TPM で `S_random` をラップし、`wrapped_secret` を保存する。
- `wrapped_secret` 保存先と ACL を固定する。
- 失敗時エラー設計:
  - TPM未検出
  - TPMラップ失敗
  - 保存先権限不足
  - 既存 `wrapped_secret` との競合

### Phase 2: LocalServer 起動時鍵導出
- `wrapped_secret` を読込む。
- TPM で `S_random` を復号する。
- HKDF で `k-user` を生成する。
- `device_id`（base MAC）から `k-device` を導出する。
- `k-user` / `k-device` はメモリ上のみ保持し、使用後はゼロ化する。

### Phase 3: 初回ペアリング・暗号通信
- デバイスAPモード起動（物理操作必須）
- `LocalServer` で必須入力（Wi-Fi / MQTT / OTA / 認証情報）を事前検証
- `SecretCore` で `runPairingSession()` を開始
- `createPairingBundle` は workflow 内部で生成
- AP モード ECDH と固定公開鍵検証前提で bundle を投入し、完了判定まで実行
- MQTT / OTA / Wi-Fi設定更新で `k-device` を利用

### Phase 4: バックアップ/復元
- `wrapped_secret` と `device_db` をバックアップ対象とする。
- 再インストール時に両ファイルを復元する。
- 同一PC + 同一TPM でのみ復旧できることを確認する。
- [禁止] 平文 `k-user` をバックアップ対象にしない。

### Phase 5: 障害時運用
- TPM初期化時の再発行フロー
- マザーボード交換時の再登録フロー
- PC故障時の全デバイス再登録フロー

## 3. モジュール案（PC側）
- `setup-core`:
  - `S_random` 生成、TPMラップ、`wrapped_secret` 配置
- `secret-core`:
  - TPM復号、`k-user` 生成、`k-device` 導出、署名/HMAC、workflow 実行、監査ログ
- `device-registry`:
  - `device_db` 管理、`base_mac` / `public_id` / 状態情報保持
- `local-server-ui`:
  - AP 共通トップ画面、入力検証、進行状態表示、監査表示
- `backup-manager`:
  - `wrapped_secret` / `device_db` のエクスポート・インポート
- `support-package`:
  - 鍵を含まない診断情報パッケージ生成

## 4. モジュール案（ESP側）
- `certification.cpp`:
  - `device_id` 取扱い、受信メッセージ検証
- `filesystem.cpp`:
  - NVS / LittleFS 保存処理
- `wifi.cpp`:
  - APモード/通常モード切替
- `mqtt.cpp`:
  - update / confirm 受信、暗号payload処理
- `input.cpp`:
  - 物理ボタン長押し検知（AP復旧）

## 6. APIたたき台
### 6.1 セットアップ API（PC内）
- `initializeKeyHierarchy() -> { created: boolean, keySchemaVersion: string }`
- `isKeyHierarchyInitialized() -> boolean`
- `rewrapSecret() -> { rotated: boolean }`

### 6.2 SecretCore API
- `unwrapSecretToRuntime() -> boolean`
- `deriveUserKey() -> runtimeOnly`
- `deriveDeviceKey(baseMac: string) -> runtimeOnly`
- `runPairingSession(targetDeviceId, sessionId, keyVersion, requestedSettings) -> { workflowId, state }`
- `runKeyRotationSession(targetDeviceId, sessionId, keyVersion, requestedSettings) -> { workflowId, state }`
- `runProductionSecureFlow(deviceId, runId, productionSettings) -> { workflowId, state }`
- `runSignedOtaCommand(targetDeviceId, firmwareVersion, manifestUrl, firmwareUrl, sha256) -> { workflowId, state }`
- `getWorkflowStatus(workflowId) -> { state, result, errorSummary }`
- [厳守] `createPairingBundle(...)` は `runPairingSession()` 内部 helper とし、`LocalServer` 公開 API にしない。

### 6.3 LocalServer REST API
- `POST /api/workflows/pairing/start -> runPairingSession(...)`
- `POST /api/workflows/key-rotation/start -> runKeyRotationSession(...)`
- `POST /api/workflows/production/start -> runProductionSecureFlow(...)`
- `POST /api/workflows/signed-ota/start -> runSignedOtaCommand(...)`
- `GET /api/workflows/{workflowId} -> getWorkflowStatus(...)`
- [厳守] REST 応答は `workflowId`、`state`、`result`、`errorSummary` に限定し、raw key や中間秘密を返さない。
- [禁止] `POST /api/pairing/bundle` のような内部 helper 直公開 API を作成しない。

### 6.4 バックアップ API
- `exportWrappedSecretBackup() -> file`
- `exportDeviceDbBackup() -> file`
- `restoreWrappedSecretBackup(file) -> boolean`
- `restoreDeviceDbBackup(file) -> boolean`

## 7. 試験観点たたき台
- 鍵生成:
  - `S_random` が毎回異なること
  - `wrapped_secret` だけが保存され、平文秘密が残らないこと
- 起動時導出:
  - 同一PCで `k-user` が再現できること
  - `k-device` が同一 `base_mac` から再現できること
- workflow 実行:
  - `runPairingSession()` が必須設定不足で前段検証停止すること
  - `runKeyRotationSession()` が個体単位で完了判定できること
  - `runProductionSecureFlow()` が段階実行と再判定を扱えること
  - `runSignedOtaCommand()` が署名付き開始指示と完了判定を扱えること
  - raw `k-device` や `k-pairing-session` が TS 側へ返らないこと
  - 進捗状態が `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` で遷移すること
- 別PC移行防止:
  - `wrapped_secret` 単体を別PCへコピーしても復号失敗すること
- 再インストール復旧:
  - 同一PCで `wrapped_secret` + `device_db` 復元後に通信継続できること
  - 別PCで `k-user-backup.enc` + パスワード + `device_db` 復元後に通信継続できること
- 障害時:
  - TPM初期化後は既存 `wrapped_secret` を復号できないこと
  - `k-user-backup.enc` 経路で復旧できること
  - 復旧不能時は再発行後に再登録が必要になること

## 8. リスクと対策たたき台
- リスク: TPM未搭載/無効化PCで動作しない
  - 対策: `TPM由来方式` と `暗号化ファイル + パスワード方式` を選択可能とし、導入時に選択できるようにする
- リスク: `wrapped_secret` 破損で復旧不能
  - 対策: `k-user-backup.enc` による復旧経路を併用し、バックアップ運用を導入手順書へ追記する
- リスク: 同一PC侵害時のメモリ窃取
  - 対策: `k-user` / `k-device` 保持時間最短化、zeroize、監査ログ強化
- リスク: device DB 欠損
  - 対策: `device_db` のバックアップ必須化と整合性チェック実装

## 9. 先行実装順（提案）
1. `secret-core` に TPM 初期化と `wrapped_secret` 生成を実装する  
2. LocalServer 起動時の `wrapped_secret` 読込 + `k-user` 導出を実装する  
3. `deriveDeviceKey(baseMac)` を実装する  
4. `device_db` と `public_id` の管理を実装する  
5. `runPairingSession()` を実装する  
6. `runSignedOtaCommand()` を実装する  
7. `runKeyRotationSession()` / `runProductionSecureFlow()` を実装する  
8. 同一PC再インストール復旧試験を実施する
