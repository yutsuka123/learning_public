# 鍵管理・初期セットアップ 実装たたき台

[重要] 本書は `鍵管理および初期セットアップ設計仕様書.md` を実装へ落とし込むためのドラフトである。  
理由: `TPM + wrapped_secret + HKDF` 方式の実装順序と責務分担を先に固定し、手戻りを防ぐため。

[厳守] 本書には鍵実値・秘密情報を記載しない。  
理由: 公開リポジトリ運用のため。

## 1. スコープ（第1版）
- 対象:
  - `S_random` 生成
  - TPM による `wrapped_secret` 生成・保存
  - `wrapped_secret` 復号による `k-user` 再生成
  - `k-device` 導出
  - Wi-Fi更新での暗号 payload 生成/受信準備
  - 同一PC再インストール時の `wrapped_secret` / `device_db` 復元
- 対象外（次版）:
  - TPM 非搭載環境向け代替方式
  - macOS 向け同等ハードウェア拘束方式
  - クラウドKMS連携

## 2. 実装フェーズ案
### Phase 1: TPM 初期化基盤
- セットアップ処理で `S_random` を CSPRNG 生成する。
- TPM で `S_random` をラップし、`wrapped_secret` を保存する。
- `wrapped_secret` 保存先と ACL を固定する。
- 失敗時エラー設計:
  - TPM未検出
  - TPMラップ失敗
  - 保存先権限不足
  - 既存 `wrapped_secret` との競合

### Phase 2: LocalServer 起動時鍵導出
- `wrapped_secret` を読込む。
- TPM で `S_random` を復号する。
- HKDF で `k-user` を生成する。
- `device_id`（base MAC）から `k-device` を導出する。
- `k-user` / `k-device` はメモリ上のみ保持し、使用後はゼロ化する。

### Phase 3: 初回ペアリング・暗号通信
- デバイスAPモード起動（物理操作必須）
- PCツールから `base_mac` を取得
- `k-device` を生成
- ペアリング仕様に従って必要な登録処理を実施
- MQTT / OTA / Wi-Fi設定更新で `k-device` を利用

### Phase 4: バックアップ/復元
- `wrapped_secret` と `device_db` をバックアップ対象とする。
- 再インストール時に両ファイルを復元する。
- 同一PC + 同一TPM でのみ復旧できることを確認する。
- [禁止] 平文 `k-user` をバックアップ対象にしない。

### Phase 5: 障害時運用
- TPM初期化時の再発行フロー
- マザーボード交換時の再登録フロー
- PC故障時の全デバイス再登録フロー

## 3. モジュール案（PC側）
- `setup-core`:
  - `S_random` 生成、TPMラップ、`wrapped_secret` 配置
- `secret-core`:
  - TPM復号、`k-user` 生成、`k-device` 導出、署名/HMAC、監査ログ
- `device-registry`:
  - `device_db` 管理、`base_mac` / `public_id` / 状態情報保持
- `backup-manager`:
  - `wrapped_secret` / `device_db` のエクスポート・インポート
- `support-package`:
  - 鍵を含まない診断情報パッケージ生成

## 4. モジュール案（ESP側）
- `certification.cpp`:
  - `device_id` 取扱い、受信メッセージ検証
- `filesystem.cpp`:
  - NVS / LittleFS 保存処理
- `wifi.cpp`:
  - APモード/通常モード切替
- `mqtt.cpp`:
  - update / confirm 受信、暗号payload処理
- `input.cpp`:
  - 物理ボタン長押し検知（AP復旧）

## 7. 試験観点たたき台
- 鍵生成:
  - `S_random` が毎回異なること
  - `wrapped_secret` だけが保存され、平文秘密が残らないこと
- 起動時導出:
  - 同一PCで `k-user` が再現できること
  - `k-device` が同一 `base_mac` から再現できること
- 別PC移行防止:
  - `wrapped_secret` を別PCへコピーしても復号失敗すること
- 再インストール復旧:
  - 同一PCで `wrapped_secret` + `device_db` 復元後に通信継続できること
- 障害時:
  - TPM初期化後は既存 `wrapped_secret` を復号できないこと
  - 再発行後に再登録が必要になること

## 8. リスクと対策たたき台
- リスク: TPM未搭載/無効化PCで動作しない
  - 対策: セットアップ前提条件として TPM 2.0 必須を明示する
- リスク: `wrapped_secret` 破損で復旧不能
  - 対策: バックアップ対象へ明示し、導入手順書へ退避手順を追記する
- リスク: 同一PC侵害時のメモリ窃取
  - 対策: `k-user` / `k-device` 保持時間最短化、zeroize、監査ログ強化
- リスク: device DB 欠損
  - 対策: `device_db` のバックアップ必須化と整合性チェック実装

## 9. 先行実装順（提案）
1. `secret-core` に TPM 初期化と `wrapped_secret` 生成を実装する  
2. LocalServer 起動時の `wrapped_secret` 読込 + `k-user` 導出を実装する  
3. `deriveDeviceKey(baseMac)` を実装する  
4. `device_db` と `public_id` の管理を実装する  
5. Wi-Fi更新 / OTA / 署名処理を `k-device` 利用へ接続する  
6. 同一PC再インストール復旧試験を実施する
