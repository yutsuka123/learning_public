# IoT モジュール仕様書

[重要] 本書は `IoT` 配下モジュールの責務を定義する。  
理由: 実装担当の境界を明確化し、修正影響範囲を把握しやすくするため。

## 1. 想定フォルダ構成
- `IoT/Production/`  
  メーカー専用の製造・初回書込み・初回セキュア化・eFuse 実行ツール群
- `IoT/ESP32/`  
  ESP32-S3向けファームウェア（MQTT通信、TLS認証、セキュリティ機能）
- `IoT/ESP32/header/`  
  ESP専用ヘッダ群（タスクIF、`define.h`、`mqttProtocol.h`、`httpProtocol.h` など）
- `IoT/ESP32/src/`  
  ESP実装本体（各タスク実装、`mqttProtocol.cpp`、`httpProtocol.cpp` など）
- `IoT/shared/include/`  
  ESP/Local/Cloudで共有する公開定義（`common.h`）
- `IoT/LocalServer/`  
  ローカルHTTPサーバー実装、ローカル向けAPI提供、Web GUI、OTA配布
- `IoT/LocalServer秘密処理別層仕様書.md`  
  `SecretCore`（Rust）と TypeScript 本体の責務境界・IPC・秘密処理仕様
- `IoT/LocalSoft/`  
  ESP32とMQTT通信し、状態監視・命令送信・履歴保存（SQLite）を行うローカル運用ソフト
- `IoT/LocalSoftForCloud/`  
  クラウド連携モジュール（設定/指令情報の双方向同期）
- `IoT/CloudServer/`  
  [将来対応] クラウド側サーバー実装、デプロイ構成、運用関連ファイル
- `IoT/docs/`  
  [将来対応] 追加設計書・試験証跡（必要時）

### 2.1 補足
- [厳守] `IoT/Production` は `IoT/LocalServer` の通常運用 UI と混在させない。
- [厳守] `SecretCore` は TypeScript から分離し、raw key を返却しない。
- [厳守] 高リスク処理は `SecretCore` が開始から完了判定まで責任を持ち、`LocalServer` は逐次オーケストレーションを行わない。
- [厳守] `ESP32` は `k-device`、Wi-Fi、MQTT、OTA、認証情報を NVS に保持する。
- [厳守] `ESP32` は `current/previous` の 2 スロットで鍵状態を管理する。
- [厳守] `ESP32` は AP 共通トップ画面認証後に状態表示し、Production 操作は追加認証で分離する。

### 2.1.1 ワークフロー進捗状態
- [厳守] 高リスクワークフローの進捗状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` を正規値とする。
- [厳守] `LocalServer` は上記進捗状態と結果のみを表示し、中間秘密や署名素材は受け取らない。

## 2.2 責任範囲の明確化
- [厳守] `IoT/Production` の責任範囲は、製造、初回書込み、初回セキュア化、eFuse 最終有効化、製造ログ保存までとする。
- [禁止] `IoT/Production` に通常運用 UI、通常監視、通常 OTA 配布、テレメトリ保存の責務を混在させない。
- [厳守] `IoT/LocalServer` の責任範囲は、通常運用 UI、REST API、状態表示、OTA 実行管理、AP セッション管理、再ペアリング進行管理、監査表示までとする。
- [厳守] `IoT/LocalServer` は高リスクワークフローの開始要求、入力検証、進捗表示、最終結果表示のみを担当する。
- [禁止] `IoT/LocalServer` が raw `k-user`、raw `k-device`、製造用秘密鍵、eFuse 実値を保持しない。
- [禁止] `IoT/LocalServer` が `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` の手順を TS 側で逐次制御しない。
- [厳守] `SecretCore` の責任範囲は、TPM 拘束秘密管理、`k-user` 再生成、`k-device` 導出、署名、暗号化、高リスクワークフロー実行、監査記録までとする。
- [禁止] `SecretCore` に通常 UI、MQTT 監視表示、SQLite 保存を持たせないが、高リスクワークフロー中の対ESP32通信完了判定は担当してよい。
- [厳守] `IoT/ESP32` の責任範囲は、デバイス側 AP モード、bundle 検証/復号、NVS 保存、current/previous 鍵運用、MQTT/OTA 実行、状態通知までとする。
- [禁止] `IoT/ESP32` に製造用鍵の長期保持、未検証 bundle 受理、無認証高権限操作を持たせない。
- [厳守] `IoT/LocalSoft` の責任範囲は、MQTT 通知の保存、履歴参照、保持期間管理、監査ログの補助保存までとする。
- [禁止] `IoT/LocalSoft` が鍵導出、署名、pairing、eFuse 実行主体にならない。
- [厳守] 文書上の責任境界の親定義は本書とし、他仕様書で役割を記載する場合は本書と矛盾させない。

## 2.1 補足方針（新規4フォルダ）
- [重要] `readme.txt` は各フォルダの入口文書として保持する。  
  理由: Git管理時に空フォルダ消失を防ぎ、役割を即座に把握できるため。
- [厳守] 実装を追加する際は、同一フォルダの `readme.txt` に「更新日時」と「変更理由」を追記する。  
  理由: 仕様変更の経緯をフォルダ単位で追跡するため。
- [禁止] 機密情報（ID/パスワード/秘密鍵）を `readme.txt` に記載しない。  
  理由: 誤コミットによる漏えい防止のため。

## 3. 現行実装対象
- [厳守] 第3段階のみを実施対象とする。
- [廃止の方針] 第1段階・第2段階（特にOTA関連）は現行スコープから除外する。

## 4. セキュリティ境界
- [厳守] 認証情報は `IoT/` 配下に保存しない（必要時はローカル安全領域参照）。
- [推奨] 認証ロジックは `authLogic` 等の非機密モジュールとして分離実装する。
- [厳守] 現行の鍵保護は TPM 前提とし、保存対象は `wrapped_secret` のみとする。
- [厳守] pairing bundle は `SecretCore` の `runPairingSession()` 内部の `createPairingBundle` でのみ生成し、TS へ raw `k-device` や ECDH 共有秘密を返さない。
- [厳守] `public_id` は公開識別子として扱うが、Wi-Fi ステーション名や通常 topic に base MAC を直接使用しない。
- [厳守] `k-iot-flash-encryption` は個体ごとに初回セキュア化時へ設定し、通常 OTA 運用では再投入しない。
