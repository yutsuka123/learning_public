# IoT モジュール仕様書

[重要] 本書は `IoT` 配下モジュールの責務を定義する。  
理由: 実装担当の境界を明確化し、修正影響範囲を把握しやすくするため。

## 1. 想定フォルダ構成
- `IoT/ProductionTool/`  
  `LocalServer` と分離したメーカー専用ソフトウェア `ProductionTool` の実装格納先。製造・初回書込み・初回セキュア化・eFuse 実行を担当する。
- `IoT/ESP32/`  
  ESP32-S3向けファームウェア（MQTT通信、TLS認証、セキュリティ機能）
- `IoT/ESP32/header/`  
  ESP専用ヘッダ群（タスクIF、`define.h`、`mqttProtocol.h`、`httpProtocol.h` など）
- `IoT/ESP32/src/`  
  ESP実装本体（各タスク実装、`mqttProtocol.cpp`、`httpProtocol.cpp` など）
- `IoT/shared/include/`  
  ESP/Local/Cloudで共有する公開定義（`common.h`）
- `IoT/LocalServer/`  
  ローカルHTTPサーバー実装、ローカル向けAPI提供、Web GUI、OTA配布、SQLite保存、平文出力
- `IoT/LocalServer秘密処理別層仕様書.md`  
  `SecretCore`（Rust）と TypeScript 本体の責務境界・IPC・秘密処理仕様
- `IoT/LocalSoft/`  
  [廃止の方針] 旧ローカル運用ソフト。新規利用しない（`LocalServer` へ統合）
- `IoT/LocalSoftForCloud/`  
  [廃止の方針] 旧クラウド連携分離モジュール。将来は `LocalServer` のクラウド連携モードへ統合
- `IoT/CloudServer/`  
  [将来対応] クラウド側サーバー実装、デプロイ構成、運用関連ファイル
- `IoT/docs/`  
  [将来対応] 追加設計書・試験証跡（必要時）

### 2.1 補足
- [厳守] `IoT/ProductionTool` は `IoT/LocalServer` の通常運用 UI と混在させない。
- [厳守][2026-03-16] `IoT/ProductionTool` は、基本機能安定化フェーズ（`004-0008`〜`004-0010`）では「起動、追加認証、対象機確認、dry-run、監査ログ」までを責務とし、eFuse / Secure Boot / Flash Encryption の不可逆処理本体は後続フェーズへ分離する。
- [厳守][2026-03-16] Rust 共通化は `LocalServer` 側で整備した `SecretCore` 共通部や関連ライブラリを `ProductionTool` でも再利用する意味であり、ソフトウェア自体を統合する意味ではない。
- [厳守] `SecretCore` は TypeScript から分離し、raw key を返却しない。
- [厳守] 高リスク処理は `SecretCore` が開始から完了判定まで責任を持ち、`LocalServer` は逐次オーケストレーションを行わない。
- [重要] 通信Rust化の移行期間中、`LocalServer` は具体通信実装ではなく `deviceTransport` IF に依存し、TS 実装と将来の Rust 実装を差し替え可能にする。
- [進捗][2026-03-15] Stage1 として MQTT command publish は `SecretCore` が担当し、TS 側は subscribe/受信解析を継続する。
- [進捗][2026-03-15] Stage2 として MQTT notice subscribe は `SecretCore` が担当し、TS 側は drain 後の復号・正規化・状態反映を継続する。
- [厳守] `ESP32` は `k-device`、Wi-Fi、MQTT、OTA、認証情報を NVS に保持する。
- [厳守] `ESP32` は画像、証明書、ログを `LittleFS` の `/images` `/certs` `/logs` に分離保持する。
- [厳守] `ESP32` は `current/previous` の 2 スロットで鍵状態を管理する。
- [重要] `LocalServer` は `1 Server - 複数 Broker` を考慮したメタ情報を保持してよいが、当面の推奨運用は `1 Server - 1 Broker` とする。
- [重要] `ESP32` は将来の複数 `Server` 同時接続を考慮し、`serverId` 単位で `brokerId`、`wifiApId`、認証情報、`k-device` を区別できる構造を前提とする。
- [厳守] `ESP32` は AP 共通トップ画面認証後に状態表示し、ProductionTool 操作は追加認証で分離する。
- [厳守] `ESP32` の `LittleFS` 更新は `SHA-256` 一致確認後のみ反映し、失敗時は旧ファイルを維持する。

### 2.1.1 ワークフロー進捗状態
- [厳守] 高リスクワークフローの進捗状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` を正規値とする。
- [厳守] `LocalServer` は上記進捗状態と結果のみを表示し、中間秘密や署名素材は受け取らない。

## 2.2 責任範囲の明確化
- [厳守] `IoT/ProductionTool` の責任範囲は、製造、初回書込み、初回セキュア化、eFuse 最終有効化、製造ログ保存までとする。
- [厳守][2026-03-16] ただし `004-0008`〜`004-0010` の段階では、`IoT/ProductionTool` の責務を「起動、追加認証、対象機確認、実行内容確認、dry-run、監査ログ、安全停止」までに限定してよい。
- [重要][2026-03-16] `IoT/ProductionTool` はこの段階で `SecretCore` 共通部を再利用してよいが、戻り値は `workflowId`、`state`、`result`、`errorSummary`、監査表示向け最小情報に限定する。
- [禁止] `IoT/ProductionTool` に通常運用 UI、通常監視、通常 OTA 配布、テレメトリ保存の責務を混在させない。
- [禁止][2026-03-16] `IoT/ProductionTool` の基本機能フェーズで eFuse 実値、製造用秘密鍵、raw `k-device`、中間秘密を UI やログへ露出しない。
- [厳守] `IoT/LocalServer` の責任範囲は、通常運用 UI、REST API、状態表示、OTA 実行管理、AP セッション管理、再ペアリング進行管理、監査表示までとする。
- [厳守] `IoT/LocalServer` は高リスクワークフローの開始要求、入力検証、進捗表示、最終結果表示のみを担当する。
- [重要] `IoT/LocalServer` は秘密を含まないサポートパッケージの生成を担当してよい。
- [厳守] サポートパッケージ生成は秘密値の抽出機能ではなく、診断用メタ情報とログ要約の収集に限定する。
- [禁止] `IoT/LocalServer` が raw `k-user`、raw `k-device`、製造用秘密鍵、eFuse 実値を保持しない。
- [禁止] `IoT/LocalServer` が `runPairingSession()`、`runKeyRotationSession()`、`runProductionSecureFlow()`、`runSignedOtaCommand()` の手順を TS 側で逐次制御しない。
- [厳守] `SecretCore` の責任範囲は、TPM 拘束秘密管理、`k-user` 再生成、`k-device` 導出、署名、暗号化、高リスクワークフロー実行、監査記録までとする。
- [禁止] `SecretCore` に通常 UI、MQTT 監視表示、SQLite 保存を持たせないが、高リスクワークフロー中の対ESP32通信完了判定は担当してよい。
- [厳守] `IoT/ESP32` の責任範囲は、デバイス側 AP モード、bundle 検証/復号、NVS 保存、current/previous 鍵運用、MQTT/OTA 実行、状態通知までとする。
- [厳守] `IoT/ESP32` の責任範囲には、`LittleFS` の `/images` `/certs` `/logs` 管理、証明書読込、画像/証明書の局所更新、ログローテーションを含める。
- [禁止] `IoT/ESP32` に製造用鍵の長期保持、未検証 bundle 受理、無認証高権限操作を持たせない。
- [禁止] `IoT/ESP32` に画像更新のたびの `LittleFS` 全面書換や、ログ巻き込み削除を持たせない。
- [廃止の方針] `IoT/LocalSoft` は `LocalServer` 一体化方針により新規利用しない。理由: 運用ソフトを単一化して保守性と導入性を向上するため。
- [新規利用禁止] `IoT/LocalSoft` / `IoT/LocalSoftForCloud` への新規機能追加を禁止する。理由: 機能分散による仕様乖離と二重保守を防ぐため。
- [厳守] 文書上の責任境界の親定義は本書とし、他仕様書で役割を記載する場合は本書と矛盾させない。

## 2.1 補足方針（新規4フォルダ）
- [重要] `readme.txt` は各フォルダの入口文書として保持する。  
  理由: Git管理時に空フォルダ消失を防ぎ、役割を即座に把握できるため。
- [厳守] 実装を追加する際は、同一フォルダの `readme.txt` に「更新日時」と「変更理由」を追記する。  
  理由: 仕様変更の経緯をフォルダ単位で追跡するため。
- [禁止] 機密情報（ID/パスワード/秘密鍵）を `readme.txt` に記載しない。  
  理由: 誤コミットによる漏えい防止のため。

## 3. 現行実装対象
- [厳守] 第3段階のみを実施対象とする。
- [廃止の方針] 第1段階・第2段階（特にOTA関連）は現行スコープから除外する。

## 4. セキュリティ境界
- [厳守] 認証情報は `IoT/` 配下に保存しない（必要時はローカル安全領域参照）。
- [推奨] 認証ロジックは `authLogic` 等の非機密モジュールとして分離実装する。
- [厳守] 鍵保護は `TPM由来方式` または `暗号化ファイル + パスワード方式` を許容し、復号処理は `SecretCore` 内でのみ実行する。
- [厳守] pairing bundle は `SecretCore` の `runPairingSession()` 内部の `createPairingBundle` でのみ生成し、TS へ raw `k-device` や ECDH 共有秘密を返さない。
- [厳守] `public_id` は公開識別子として扱うが、Wi-Fi ステーション名や通常 topic に base MAC を直接使用しない。
- [厳守] `k-iot-flash-encryption` は個体ごとに初回セキュア化時へ設定し、通常 OTA 運用では再投入しない。
