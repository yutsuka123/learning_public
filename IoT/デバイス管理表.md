# IoT デバイス管理表

[重要] 本書は実機ごとの識別情報、暗号化状態、`eFuse` 状態、試験利用履歴を管理する。  
理由: 1台目試験機と量産対象の取り違えを防ぎ、不可逆処理の適用履歴を後追い監査できるようにするため。

[厳守] `eFuse`、鍵、FW版、試験番号、判定履歴を更新した場合は、本書と `試験記録書.md` を同日更新する。  
理由: 実機状態と試験証跡の不整合を防ぐため。

[禁止] 秘密鍵実体、認証パスワード実値、raw secret を記載しない。  
理由: 管理表から機密漏えいを起こさないため。

## 1. 管理ルール
- [厳守] 管理番号は `DEV-xxx` の連番とし、今回試験に使用した個体を `DEV-001` とする。
- [厳守] `MAC` は可能な範囲で「チップ実測値」と「運用表示値」の両方を残す。
- [重要] `NVS` は「設計意図」と「実運用パターン」を分けて記載する。
- [重要] 出荷可否は `007-C` のレビューゲート完了後に確定する。

## 2. 項目定義
- 管理番号: 本プロジェクト内での固定識別子。
- 用途区分: 試験機、本番1台目、量産候補、隔離個体などの区分。
- チップ実測MAC: `espefuse summary` などの低レイヤ確認値。
- 運用表示識別子: `deviceName`、`public_id`、ラベル表示値などの運用上の識別子。
- セキュア状態: `Flash Encryption`、`NVS(HMAC)`, `Secure Boot v2`, `JTAG封鎖`, `Secure Download` の適用状態。
- NVS実行パターン: `patternA` / `patternC` のどちらで動作しているか。
- 最新判定: `007-B` 完了、`007-C` レビュー中、出荷可、隔離 など。

## 4. `DEV-001` 詳細メモ
- [重要][2026-03-27] 本個体は `FE -> NVS(HMAC) -> SB -> 封鎖` の順で不可逆工程を実施済み。
- [重要][2026-03-27] `DIS_USB_SERIAL_JTAG` と `DIS_DOWNLOAD_MODE` は未適用方針とし、シリアルログを維持している。
- [重要][2026-03-27] `get_security_info` では `Secure Boot: Enabled`、`Flash Encryption: Enabled`、`JTAG: Permanently Disabled`、`Secure Download Mode` を確認済み。
- [重要][2026-03-27] `read_mem` / `read_flash` は `Secure Download Mode` で拒否されることを確認済み。
- [要確認][2026-03-27] `NVS` は現状 `patternA` 動作である。設計意図の `patternC` と開発時の安全性を両立する方式は `008` で継続検討する。

## 5. 更新手順
- `eFuse` 更新時: `espefuse summary` before/after を採取し、本書の状態列を更新する。
- OTA更新時: `FW版`、`備考`、関連試験番号を更新する。
- レビューゲート時: `最新判定` を更新し、必要なら `問題点記録書.md` と相互参照を追記する。

## 6. 変更履歴
