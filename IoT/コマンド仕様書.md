# IoT コマンド仕様書

[重要] 本書は開発・試験で使用するコマンドを定義する。  
理由: 手順の揺れによる再現不可を防ぐため。

## 0. コマンド責任範囲
- [厳守] `LocalServer` 用コマンドは通常運用、`Production` 用コマンドは工場内専用運用として分離する。
- [厳守] `SecretCore` は `LocalServer` 配下のローカルIPC前提で利用し、TCP/HTTP で直接起動・疎通確認する運用を行わない。
- [厳守] 詳細な責任境界は `モジュール仕様書.md` を親定義とする。

## 1. Local MQTT（mosquitto）
- ブローカ起動:
  - `mosquitto -c mosquitto.conf -v`
- Subscribe確認:
  - `mosquitto_sub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/+/status/#" -v`
- Publish確認:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/esp32s3-001/cmd/led" -m "{\"requestId\":\"req-001\",\"ledState\":\"on\"}"`

## 2. PlatformIO（ESP32-S3）
- プロジェクト初期化:
  - `pio project init --board esp32-s3-devkitc-1`
- ビルド:
  - `pio run`
- シリアル監視:
  - `pio device monitor`
- 書込み:
  - `pio run -t upload`

## 3. セキュリティ有効化時の注意
- [厳守] eFuse書込み系コマンドは事前レビュー必須。
- [禁止] 秘密鍵パスを履歴に残す実行方法（平文引数）を使用しない。
- [厳守] MQTT検証はTLS接続コマンドで実施し、平文ポート1883を使わない。
- [禁止] `--insecure` 相当の検証スキップを恒久運用しない。

## 4. LocalServer（TypeScript/Node.js）
- 依存導入:
  - `cd IoT/LocalServer`
  - `npm install`
- 開発起動:
  - `npm run dev`
- 本番ビルド:
  - `npm run build`
- 本番起動:
  - `npm run start`
- MQTT疎通試験:
  - `npm run test:connect`

## 5. LocalServer 自動起動（Task Scheduler）
- タスク登録（管理者PowerShell）:
  - `cd IoT/LocalServer/scripts`
  - `powershell -ExecutionPolicy Bypass -File .\install-task-scheduler.ps1`
- タスク確認:
  - `Get-ScheduledTask -TaskName "IoT_LocalServer_AutoStart"`
- 手動実行:
  - `Start-ScheduledTask -TaskName "IoT_LocalServer_AutoStart"`

## 6. LocalServer OTA配布（HTTPS）
- [重要] 2026-03-07 追加: OTA方式①では `LocalServer` が `4443/TCP` で `firmware.bin` を配布する。  
  理由: ESP32 が `mqtt.esplab.home.arpa:4443` へ HTTPS 接続して OTA を実施するため。
- OTA配布サーバー起動:
  - `cd IoT/LocalServer`
  - `npm run build`
  - `node dist/server.js`
- OTA配布確認:
  - `curl.exe -k https://127.0.0.1:4443/ota/manifest.json`
  - `curl.exe -k https://127.0.0.1:4443/ota/firmware.bin -o NUL`
- サーバー証明書確認（ホスト名付き）:
  - `openssl s_client -connect mqtt.esplab.home.arpa:4443 -servername mqtt.esplab.home.arpa`
- CA指定での検証確認:
  - `openssl s_client -connect mqtt.esplab.home.arpa:4443 -servername mqtt.esplab.home.arpa -CAfile IoT/ESP32/src/MQTT/ca.crt`
- Windows Firewall 受信許可（管理者PowerShell）:
  - `netsh advfirewall firewall add rule name="IoT LocalServer OTA 4443" dir=in action=allow protocol=TCP localport=4443`
  - `Get-NetFirewallRule -DisplayName "IoT LocalServer OTA 4443"`
- 到達確認:
  - `Test-NetConnection -ComputerName mqtt.esplab.home.arpa -Port 4443`
- DNS確認:
  - `nslookup mqtt.esplab.home.arpa 172.17.1.100`
- [厳守] OTA実機試験前に `4443/TCP` の受信許可と `OTA HTTPS started. httpsPort=4443` の両方を確認する。  
  理由: Node起動済でも Windows 側で受信拒否されると ESP32 では `connection refused` になるため。
- [厳守] 本番運用では `curl.exe -k` による検証スキップ確認だけで合格としない。  
  理由: 実運用では証明書検証有効が前提であるため。

## 7. ESP32 beta版数更新（書込み前運用）
- [重要] 2026-03-07 追加: 当面の間、ESP32へ書き込むたびに `beta.x` の `x` をインクリメントして、書換有無を識別しやすくする。  
  理由: 実機確認時に「どの書込みが稼働中か」を画面・MQTTログで即座に判別するため。
- 自動更新（`x` を +1）:
  - `cd IoT/LocalServer/scripts`
  - `powershell -ExecutionPolicy Bypass -File .\bump-esp32-beta-version.ps1`
- 版数を明示指定して更新:
  - `powershell -ExecutionPolicy Bypass -File .\bump-esp32-beta-version.ps1 -NextVersion "1.1.0-beta.10"`
- 変更内容の確認のみ（DryRun）:
  - `powershell -ExecutionPolicy Bypass -File .\bump-esp32-beta-version.ps1 -DryRun`
- [厳守] 本スクリプト実行後に `IoT/ESP32/header/version.h` と `IoT/LocalServer/.env` の両方が同一版数になっていることを確認する。  
  理由: OTA指令版数とESP32自己申告版数の不整合を防ぐため。

## 8. LocalServer 設定画面/API
- 設定画面:
  - `http://127.0.0.1:3100/settings.html`
- 設定取得:
  - `curl.exe -s http://127.0.0.1:3100/api/settings`
- 設定更新（例: タイムゾーンとOTA版数）:
  - `powershell -Command "$body=@{timeZone='Asia/Tokyo';otaFirmwareVersion='1.1.0-beta.5'}|ConvertTo-Json; Invoke-RestMethod -Method Put -Uri 'http://127.0.0.1:3100/api/settings' -ContentType 'application/json' -Body $body"`
- ファームウェアアップロード:
  - `curl.exe -X POST -F "firmware=@C:/mydata/project/myproject/learning_public/IoT/ESP32/.pio/build/esp32s3_secure/firmware.bin" http://127.0.0.1:3100/api/settings/firmware-upload`
- [重要] 時刻表示は設定した `timeZone` を基準に `OTA進捗`/`最終更新日時` へ反映される。  
  理由: サーバー運用拠点の現地時刻で状態判断しやすくするため。

## 9. OTA再認証（実装後に使用）
- [重要] OTA開始前に、Web UI で管理者パスワード再入力を要求する。  
  理由: OTA は高リスク操作であり、誤操作・離席時操作・なりすまし実行を抑止するため。
- 再認証API確認（例）:
  - `curl.exe -X POST http://127.0.0.1:3100/api/auth/ota/reauth -H "Content-Type: application/json" -d "{\"password\":\"<local-password>\"}"`
- セッション有効確認（例）:
  - `curl.exe -i http://127.0.0.1:3100/api/auth/ota/session`
- [厳守] 再認証成功後の有効期間は 3時間以内とし、ブラウザセッション終了後は再認証を要求する。
- [推奨] Cookie 属性は `HttpOnly` `Secure` `SameSite=Strict` を使用する。

## 10. AP 共通トップ画面と Production 追加認証（実装後に使用）
- [重要] AP 名は `AP-esp32lab-<MAC(no colon)>` とし、顧客向けメンテナンスと Production で共通化する。  
  理由: 接続方法を単純化しつつ、認証・画面・権限で分離するため。
- 共通トップ画面表示確認（実装後に使用）:
  - `curl.exe -i http://192.168.4.1/`
- 共通トップ画面認証確認（実装後に使用）:
  - `curl.exe -X POST http://192.168.4.1/api/auth/ap-top -H "Content-Type: application/json" -d "{\"password\":\"<ap-common-password>\"}"`
- Production 追加認証確認（実装後に使用）:
  - `curl.exe -X POST http://192.168.4.1/api/auth/production -H "Content-Type: application/json" -d "{\"password\":\"<manufacturer-mode-password>\"}"`
- [厳守] 共通トップ画面パスワードと Production 用追加認証パスワードを同一にしない。
- [厳守] 共通トップ画面用パスワード、メーカーモード専用パスワードを PowerShell 履歴や共有ログへ平文で残さない。

## 10.1 AP一括メンテナンス（実装後に使用）
- [重要] 指定USB Wi-Fiで `AP-esp32lab-*` を探索し、1台ずつ接続して処理する。
- AP探索（Windows例）:
  - `netsh wlan show networks mode=bssid`
- ログイン（例）:
  - `curl.exe -X POST http://192.168.4.1/api/auth/ap-top -H "Content-Type: application/json" -d "{\"password\":\"<ap-common-password>\"}"`
- 設定反映（例）:
  - `curl.exe -X PUT http://192.168.4.1/api/settings/network -H "Content-Type: application/json" -d "{\"wifi\":{...},\"mqtt\":{...},\"https\":{...}}"`
- 再起動要求（例）:
  - `curl.exe -X POST http://192.168.4.1/api/device/reboot`
- [厳守] 成功判定はAP切断ではなく、再起動後の `status` topic 応答（`fwVersion` / `configVersion`）で行う。

## 11. `runPairingSession()` と AP 初回投入（実装後に使用）
- [重要] `requestedSettings` の必須項目（Wi-Fi / MQTT / OTA / 認証情報）は TS 側で事前検証し、不足時は `SecretCore` を呼び出さない。  
  理由: bundle 作成 API を用途固定で安全に保つため。
- workflow 開始要求（実装後に使用）:
  - `curl.exe -X POST http://127.0.0.1:3100/api/workflows/pairing/start -H "Content-Type: application/json" -d "{\"targetDeviceId\":\"<deviceId>\",\"sessionId\":\"<sessionId>\",\"keyVersion\":\"<keyVersion>\",\"requestedSettings\":{...}}"`
- workflow 状態確認（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3100/api/workflows/<workflowId>`
- AP 初回投入状態確認（実装後に使用）:
  - `curl.exe -s http://192.168.4.1/api/pairing/state`
- [厳守] `runPairingSession()` の入力には少なくとも `targetDeviceId` `sessionId` `keyVersion` `requestedSettings` を含める。
- [厳守] `LocalServer` は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` の進捗状態と結果のみを扱う。
- [厳守] TS へ raw `k-device`、ECDH 共有秘密、`k-pairing-session` を返さない。

## 12. `Production` ツールでの eFuse 実行前確認
- [厳守] `Production` は `LocalServer` と分離した別ツール群として運用する。
- [厳守] 本番個体へ適用する前に、同一ファームウェア系統・同一手順・同一鍵IDに対する試験機リハーサル記録を確認する。
- リハーサル記録確認（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3200/api/production/rehearsals/<rehearsalId>`
- [厳守] eFuse 開始前に、対象機識別、電源安定、版数、鍵ID、未セキュア化状態、スタック余裕、空きヒープ、停止対象タスク状態を確認する。
- 開始前チェック取得（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3200/api/production/devices/<deviceName>/precheck/resources`
- [推奨] ESP32 側では `uxTaskGetStackHighWaterMark` 相当で eFuse 専用タスクのスタック高水位を取得し、開始可否判定に用いる。
- [禁止] スタック余裕未確認、ヒープ不足、通常通信タスク未停止の状態で eFuse 処理を開始しない。

## 13. `Production` ツールでの eFuse 段階実行と再判定
- 段階実行開始（実装後に使用）:
  - `curl.exe -X POST http://127.0.0.1:3200/api/production/secure-run/start -H "Content-Type: application/json" -d "{\"deviceName\":\"<deviceName>\",\"runId\":\"<runId>\"}"`
- 段階状態確認（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3200/api/production/secure-run/<runId>/status`
- 中断後の再判定取得（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3200/api/production/secure-run/<runId>/recovery`
- [厳守] 中断後は eFuse 読戻し結果を確認し、残工程継続または隔離判定を行う。
- [禁止] 中断後に状態未確認のまま一律で再実行しない。

## 14. `k-user` 再発行後の current/previous key 運用（実装後に使用）
- key rotation workflow 開始（実装後に使用）:
  - `curl.exe -X POST http://127.0.0.1:3100/api/workflows/key-rotation/start -H "Content-Type: application/json" -d "{\"targetDeviceId\":\"<deviceId>\",\"sessionId\":\"<sessionId>\",\"keyVersion\":\"<nextKeyVersion>\",\"requestedSettings\":{...}}"`
- workflow 状態確認（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3100/api/workflows/<workflowId>`
- current/previous 状態確認（実装後に使用）:
  - `curl.exe -s http://192.168.4.1/api/keys/state`
- [厳守] 稼働時間 48h 到達で `previousKeyState=expired-retained` とし、旧鍵認証を拒否する。
- [厳守] さらに稼働時間 240h 保持後に previous key を自動削除する。
- [厳守] previous key 自動削除結果は監査ログのみに残し、UI へは表示しない。

## 15. Rust 主導 workflow 状態確認（実装後に使用）
- [厳守] 高リスク workflow は `SecretCore` が完了判定まで実行し、TS 側は進捗と結果のみを表示する。
- progress 確認（実装後に使用）:
  - `curl.exe -s http://127.0.0.1:3100/api/workflows/<workflowId>`
- [厳守] 正規 progress は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` とする。

## 16. `LittleFS` 画像・証明書更新（実装後に使用）
- [重要] 画像一斉更新の第一経路は `MQTT` とし、`fileSync` 系コマンドで局所更新する。  
  理由: 全面書換を避け、`/logs` を保持したまま差分更新したいため。
- `fileSyncPlan` 送信例（画像）:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "esp32lab/call/fileSyncPlan/<deviceName>" -m "{\"v\":1,\"DstID\":\"<deviceId>\",\"SrcID\":\"server-001\",\"id\":\"req-filesync-001\",\"ts\":\"2026-03-09T11:30:00.000Z\",\"op\":\"call\",\"sub\":\"fileSyncPlan\",\"args\":{\"sessionId\":\"filesync-001\",\"targetArea\":\"images\",\"basePath\":\"/images\",\"deleteMode\":\"listed\",\"files\":[{\"path\":\"/images/top/banner.png\",\"size\":10240,\"sha256\":\"<sha256>\",\"action\":\"upsert\"}]}}"`
- `fileSyncChunk` 送信例（画像チャンク）:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "esp32lab/call/fileSyncChunk/<deviceName>" -m "{\"v\":1,\"DstID\":\"<deviceId>\",\"SrcID\":\"server-001\",\"id\":\"req-filesync-002\",\"ts\":\"2026-03-09T11:30:05.000Z\",\"op\":\"call\",\"sub\":\"fileSyncChunk\",\"args\":{\"sessionId\":\"filesync-001\",\"targetArea\":\"images\",\"path\":\"/images/top/banner.png\",\"chunkIndex\":0,\"chunkCount\":1,\"dataBase64\":\"<base64>\",\"isLast\":true}}"`
- `fileSyncCommit` 送信例:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "esp32lab/call/fileSyncCommit/<deviceName>" -m "{\"v\":1,\"DstID\":\"<deviceId>\",\"SrcID\":\"server-001\",\"id\":\"req-filesync-003\",\"ts\":\"2026-03-09T11:31:00.000Z\",\"op\":\"call\",\"sub\":\"fileSyncCommit\",\"args\":{\"sessionId\":\"filesync-001\",\"targetArea\":\"images\",\"deleteMode\":\"listed\"}}"`
- [厳守] `targetArea=certs` の場合は `/certs` を更新し、接続再開前に証明書メタ情報を `NVS` へ反映する。
- [厳守] `SHA-256` 不一致時は旧ファイルを維持する。
- [厳守] `deleteMode=all` でも `/logs` は削除しない。

## 17. AP モードでの局所更新（実装後に使用）
- [重要] AP モード時も `/images` と `/certs` を局所更新できるようにする。  
  理由: 現地メンテナンス時に MQTT 非依存で更新できるようにするため。
- AP ログイン後の画像アップロード例:
  - `curl.exe -X POST http://192.168.4.1/api/storage/upload -F "targetArea=images" -F "path=/images/top/banner.png" -F "sha256=<sha256>" -F "file=@C:/path/to/banner.png"`
- AP ログイン後の証明書アップロード例:
  - `curl.exe -X POST http://192.168.4.1/api/storage/upload -F "targetArea=certs" -F "path=/certs/mqtt-ca.pem" -F "sha256=<sha256>" -F "file=@C:/path/to/mqtt-ca.pem"`
- [厳守] AP モード更新でも `tmp` 書込み、`SHA-256` 検証、成功後 `rename` を適用する。

## 18. ファイルログ確認（実装後に使用）
- [重要] ファイルログはシリアル出力とは別経路で必要時のみ有効化する。  
  理由: 通常運用時の情報露出を抑えつつ、検証時だけ証跡を残すため。
- AP モードまたは通常 API でログ一覧確認（例）:
  - `curl.exe -s http://192.168.4.1/api/logs/list`
- [厳守] 時刻同期済み時の保存形式は `/logs/<年>/<月日>/<年月日時分秒ms-連番>.log` とする。
- [厳守] 時刻同期できない場合は `/logs/<起動回数>/00001.log` とする。
- [厳守] 再起動時または 1KB 到達時に新しいログファイルへ切り替える。
- [厳守] 30 日または総量 100KB 超過時は古いログから削除する。
