# IoT コマンド仕様書

[重要] 本書は開発・試験で使用するコマンドを定義する。  
理由: 手順の揺れによる再現不可を防ぐため。

## 1. Local MQTT（mosquitto）
- ブローカ起動:
  - `mosquitto -c mosquitto.conf -v`
- Subscribe確認:
  - `mosquitto_sub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/+/status/#" -v`
- Publish確認:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/esp32s3-001/cmd/led" -m "{\"requestId\":\"req-001\",\"ledState\":\"on\"}"`

## 2. PlatformIO（ESP32-S3）
- プロジェクト初期化:
  - `pio project init --board esp32-s3-devkitc-1`
- ビルド:
  - `pio run`
- シリアル監視:
  - `pio device monitor`
- 書込み:
  - `pio run -t upload`

## 3. セキュリティ有効化時の注意
- [厳守] eFuse書込み系コマンドは事前レビュー必須。
- [禁止] 秘密鍵パスを履歴に残す実行方法（平文引数）を使用しない。
- [厳守] MQTT検証はTLS接続コマンドで実施し、平文ポート1883を使わない。
- [禁止] `--insecure` 相当の検証スキップを恒久運用しない。

## 4. LocalServer（TypeScript/Node.js）
- 依存導入:
  - `cd IoT/LocalServer`
  - `npm install`
- 開発起動:
  - `npm run dev`
- 本番ビルド:
  - `npm run build`
- 本番起動:
  - `npm run start`
- MQTT疎通試験:
  - `npm run test:connect`

## 5. LocalServer 自動起動（Task Scheduler）
- タスク登録（管理者PowerShell）:
  - `cd IoT/LocalServer/scripts`
  - `powershell -ExecutionPolicy Bypass -File .\install-task-scheduler.ps1`
- タスク確認:
  - `Get-ScheduledTask -TaskName "IoT_LocalServer_AutoStart"`
- 手動実行:
  - `Start-ScheduledTask -TaskName "IoT_LocalServer_AutoStart"`

## 6. LocalServer OTA配布（HTTPS）
- [重要] 2026-03-07 追加: OTA方式①では `LocalServer` が `4443/TCP` で `firmware.bin` を配布する。  
  理由: ESP32 が `mqtt.esplab.home.arpa:4443` へ HTTPS 接続して OTA を実施するため。
- OTA配布サーバー起動:
  - `cd IoT/LocalServer`
  - `npm run build`
  - `node dist/server.js`
- OTA配布確認:
  - `curl.exe -k https://127.0.0.1:4443/ota/manifest.json`
  - `curl.exe -k https://127.0.0.1:4443/ota/firmware.bin -o NUL`
- Windows Firewall 受信許可（管理者PowerShell）:
  - `netsh advfirewall firewall add rule name="IoT LocalServer OTA 4443" dir=in action=allow protocol=TCP localport=4443`
  - `Get-NetFirewallRule -DisplayName "IoT LocalServer OTA 4443"`
- 到達確認:
  - `Test-NetConnection -ComputerName 172.16.1.59 -Port 4443`
- [厳守] OTA実機試験前に `4443/TCP` の受信許可と `OTA HTTPS started. httpsPort=4443` の両方を確認する。  
  理由: Node起動済でも Windows 側で受信拒否されると ESP32 では `connection refused` になるため。

## 7. ESP32 beta版数更新（書込み前運用）
- [重要] 2026-03-07 追加: 当面の間、ESP32へ書き込むたびに `beta.x` の `x` をインクリメントして、書換有無を識別しやすくする。  
  理由: 実機確認時に「どの書込みが稼働中か」を画面・MQTTログで即座に判別するため。
- 自動更新（`x` を +1）:
  - `cd IoT/LocalServer/scripts`
  - `powershell -ExecutionPolicy Bypass -File .\bump-esp32-beta-version.ps1`
- 版数を明示指定して更新:
  - `powershell -ExecutionPolicy Bypass -File .\bump-esp32-beta-version.ps1 -NextVersion "1.1.0-beta.10"`
- 変更内容の確認のみ（DryRun）:
  - `powershell -ExecutionPolicy Bypass -File .\bump-esp32-beta-version.ps1 -DryRun`
- [厳守] 本スクリプト実行後に `IoT/ESP32/header/version.h` と `IoT/LocalServer/.env` の両方が同一版数になっていることを確認する。  
  理由: OTA指令版数とESP32自己申告版数の不整合を防ぐため。

## 8. LocalServer 設定画面/API
- 設定画面:
  - `http://127.0.0.1:3100/settings.html`
- 設定取得:
  - `curl.exe -s http://127.0.0.1:3100/api/settings`
- 設定更新（例: タイムゾーンとOTA版数）:
  - `powershell -Command "$body=@{timeZone='Asia/Tokyo';otaFirmwareVersion='1.1.0-beta.5'}|ConvertTo-Json; Invoke-RestMethod -Method Put -Uri 'http://127.0.0.1:3100/api/settings' -ContentType 'application/json' -Body $body"`
- ファームウェアアップロード:
  - `curl.exe -X POST -F "firmware=@C:/mydata/project/myproject/learning_public/IoT/ESP32/.pio/build/esp32s3_secure/firmware.bin" http://127.0.0.1:3100/api/settings/firmware-upload`
- [重要] 時刻表示は設定した `timeZone` を基準に `OTA進捗`/`最終更新日時` へ反映される。  
  理由: サーバー運用拠点の現地時刻で状態判断しやすくするため。
