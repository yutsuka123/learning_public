# IoT コマンド仕様書

[重要] 本書は開発・試験で使用するコマンドを定義する。  
理由: 手順の揺れによる再現不可を防ぐため。

## 0. コマンド責任範囲
- [厳守] `LocalServer` 用コマンドは通常運用、`ProductionTool` 用コマンドは工場内専用運用として分離する。
- [厳守] `SecretCore` は `LocalServer` 配下のローカルIPC前提で利用し、TCP/HTTP で直接起動・疎通確認する運用を行わない。
- [厳守] 詳細な責任境界は `モジュール仕様書.md` を親定義とする。

## 1. Local MQTT（mosquitto）
- ブローカ起動:
  - `mosquitto -c mosquitto.conf -v`
- Subscribe確認:
  - `mosquitto_sub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/+/status/#" -v`
- Publish確認:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/esp32s3-001/cmd/led" -m "{\"requestId\":\"req-001\",\"ledState\":\"on\"}"`

## 2. PlatformIO（ESP32-S3）
- プロジェクト初期化:
  - `pio project init --board esp32-s3-devkitc-1`
- ビルド:
  - `pio run`
- シリアル監視:
  - `pio device monitor`
- 書込み:
  - `pio run -t upload`

## 3. セキュリティ有効化時の注意
- [厳守] eFuse書込み系コマンドは事前レビュー必須。
- [禁止] 秘密鍵パスを履歴に残す実行方法（平文引数）を使用しない。
- [厳守] MQTT検証はTLS接続コマンドで実施し、平文ポート1883を使わない。
- [禁止] `--insecure` 相当の検証スキップを恒久運用しない。

## 6. LocalServer OTA配布（HTTPS）
- [重要] 2026-03-07 追加: OTA方式①では `LocalServer` が `4443/TCP` で `firmware.bin` を配布する。  
  理由: ESP32 が `mqtt.esplab.home.arpa:4443` へ HTTPS 接続して OTA を実施するため。
- OTA配布サーバー起動:
  - `cd IoT/LocalServer`
  - `npm run build`
  - `node dist/server.js`
- OTA配布確認:
  - `curl.exe -k https://127.0.0.1:4443/ota/manifest.json`
  - `curl.exe -k https://127.0.0.1:4443/ota/firmware.bin -o NUL`
- サーバー証明書確認（ホスト名付き）:
  - `openssl s_client -connect mqtt.esplab.home.arpa:4443 -servername mqtt.esplab.home.arpa`
- CA指定での検証確認:
  - `openssl s_client -connect mqtt.esplab.home.arpa:4443 -servername mqtt.esplab.home.arpa -CAfile IoT/ESP32/src/MQTT/ca.crt`
- Windows Firewall 受信許可（管理者PowerShell）:
  - `netsh advfirewall firewall add rule name="IoT LocalServer OTA 4443" dir=in action=allow protocol=TCP localport=4443`
  - `Get-NetFirewallRule -DisplayName "IoT LocalServer OTA 4443"`
- 到達確認:
  - `Test-NetConnection -ComputerName mqtt.esplab.home.arpa -Port 4443`
- DNS確認:
  - `nslookup mqtt.esplab.home.arpa 172.17.1.100`
- [厳守] OTA実機試験前に `4443/TCP` の受信許可と `OTA HTTPS started. httpsPort=4443` の両方を確認する。  
  理由: Node起動済でも Windows 側で受信拒否されると ESP32 では `connection refused` になるため。
- [厳守] 本番運用では `curl.exe -k` による検証スキップ確認だけで合格としない。  
  理由: 実運用では証明書検証有効が前提であるため。

### 6.1 OTA HTTPS 証明書検証用 CA 設定（003-0004）
- [重要] ESP32 の OTA は MQTT と同一 CA を使用する。経路: `/certs/mqtt-ca.pem` または `SENSITIVE_MQTT_TLS_CA_CERT`（フォールバック）。
- [厳守] LocalServer OTA サーバー証明書（`certs/server.crt`）と MQTT ブローカー証明書が**同一 CA により署名**されている必要がある。
- CA 投入手順（AP モード経由）:
  1. MQTT CA を ESP32 の `/certs` へ投入: `pushMqttCaByFileSyncE2E.mjs` または AP ログイン後の `targetArea=certs` で `mqtt-ca.pem` を upsert。
  2. ESP32 を再起動し、`/certs` の証明書を NVS メタへ同期させる。
- CA 投入手順（ビルド時フォールバック）:
  1. `sensitiveData.h` の `SENSITIVE_MQTT_TLS_CA_CERT` に OTA サーバーを検証可能な CA 証明書 PEM を設定。
  2. [禁止] 本番ではフォールバック依存にせず、`/certs` 主経路で運用する。
- 7014 試験（証明書検証有効 OTA 成功確認）:
  1. 正常証明書で OTA 取得を実行し、更新完了を確認する。
  2. 異常証明書（ホスト名不一致、信頼外 CA）で OTA を試行し、ESP32 側で拒否されることを確認する。

## 16. `LittleFS` 画像・証明書更新（実装後に使用）
- [重要] 画像一斉更新の第一経路は `MQTT` とし、`fileSync` 系コマンドで局所更新する。  
  理由: 全面書換を避け、`/logs` を保持したまま差分更新したいため。
- `fileSyncPlan` 送信例（画像）:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "esp32lab/call/fileSyncPlan/<deviceName>" -m "{\"v\":1,\"DstID\":\"<deviceId>\",\"SrcID\":\"server-001\",\"id\":\"req-filesync-001\",\"ts\":\"2026-03-09T11:30:00.000Z\",\"op\":\"call\",\"sub\":\"fileSyncPlan\",\"args\":{\"sessionId\":\"filesync-001\",\"targetArea\":\"images\",\"basePath\":\"/images\",\"deleteMode\":\"listed\",\"files\":[{\"path\":\"/images/top/banner.png\",\"size\":10240,\"sha256\":\"<sha256>\",\"action\":\"upsert\"}]}}"`
- `fileSyncChunk` 送信例（画像チャンク）:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "esp32lab/call/fileSyncChunk/<deviceName>" -m "{\"v\":1,\"DstID\":\"<deviceId>\",\"SrcID\":\"server-001\",\"id\":\"req-filesync-002\",\"ts\":\"2026-03-09T11:30:05.000Z\",\"op\":\"call\",\"sub\":\"fileSyncChunk\",\"args\":{\"sessionId\":\"filesync-001\",\"targetArea\":\"images\",\"path\":\"/images/top/banner.png\",\"chunkIndex\":0,\"chunkCount\":1,\"dataBase64\":\"<base64>\",\"isLast\":true}}"`
- `fileSyncCommit` 送信例:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "esp32lab/call/fileSyncCommit/<deviceName>" -m "{\"v\":1,\"DstID\":\"<deviceId>\",\"SrcID\":\"server-001\",\"id\":\"req-filesync-003\",\"ts\":\"2026-03-09T11:31:00.000Z\",\"op\":\"call\",\"sub\":\"fileSyncCommit\",\"args\":{\"sessionId\":\"filesync-001\",\"targetArea\":\"images\",\"deleteMode\":\"listed\"}}"`
- [厳守] `targetArea=certs` の場合は `/certs` を更新し、接続再開前に証明書メタ情報を `NVS` へ反映する。
- [厳守] `SHA-256` 不一致時は旧ファイルを維持する。
- [厳守] `deleteMode=all` でも `/logs` は削除しない。

## 18. ファイルログ確認（実装後に使用）
- [重要] ファイルログはシリアル出力とは別経路で必要時のみ有効化する。  
  理由: 通常運用時の情報露出を抑えつつ、検証時だけ証跡を残すため。
- AP モードまたは通常 API でログ一覧確認（例）:
  - `curl.exe -s http://192.168.4.1/api/logs/list`
- [厳守] 時刻同期済み時の保存形式は `/logs/<年>/<月日>/<年月日時分秒ms-連番>.log` とする。
- [厳守] 時刻同期できない場合は `/logs/<起動回数>/00001.log` とする。
- [厳守] 再起動時または 1KB 到達時に新しいログファイルへ切り替える。
- [厳守] 30 日または総量 100KB 超過時は古いログから削除する。

## 19. MQTT payload暗号化モード運用（strict既定）
- [厳守] 既定運用は `strict` とし、平文payloadを受理しない。
  理由: `k-device` による payload全文暗号化を常時有効にし、監視系クライアントからの平文可視化を防ぐため。
- LocalServer strict 起動（PowerShell）:
  - `$env:MQTT_PAYLOAD_ENCRYPTION_MODE='strict'; npm run start`
- ESP32 strict ビルド（既定）:
  - `python -m platformio run -e esp32s3_secure`
- ESP32 strict 明示ビルド（念のため固定）:
  - `$env:PLATFORMIO_BUILD_FLAGS='-D MQTT_PAYLOAD_SECURITY_MODE=2'; python -m platformio run -e esp32s3_secure`
- strict 動作確認:
  - `curl.exe -X POST http://127.0.0.1:3100/api/admin/auth/login -H "Content-Type: application/json" -d "{\"username\":\"admin\",\"password\":\"<password>\"}"`
  - `curl.exe -X POST http://127.0.0.1:3100/api/admin/commands/secure-ping -H "Authorization: Bearer <token>" -H "Content-Type: application/json" -d "{\"targetDeviceName\":\"<deviceName>\",\"plainText\":\"strict-e2e\"}"`
- [推奨] 平文拒否の確認は `7057` / `7058` を再実行して判定する。

### 19.1 一時ロールバック（障害切り分け専用）
- [新規利用禁止] `compat` 常用運用を行わない。  
  理由: 平文混在リスクが残るため。
- [重要] 障害切り分けで一時的に `compat` へ戻す場合のみ、変更開始/終了時刻と理由を `問題点記録書.md` へ必ず記録する。
- LocalServer 一時切替:
  - `$env:MQTT_PAYLOAD_ENCRYPTION_MODE='compat'; npm run start`
- ESP32 一時切替:
  - `$env:PLATFORMIO_BUILD_FLAGS='-D MQTT_PAYLOAD_SECURITY_MODE=1'; python -m platformio run -e esp32s3_secure -t upload --upload-port COM9`
- [厳守] 切り分け完了後は strict へ戻し、`secure-ping` と平文拒否確認を再実施する。
