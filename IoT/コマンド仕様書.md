# IoT コマンド仕様書

[重要] 本書は開発・試験で使用するコマンドを定義する。  
理由: 手順の揺れによる再現不可を防ぐため。

## 0. コマンド責任範囲
- [厳守] `LocalServer` 用コマンドは通常運用、`ProductionTool` 用コマンドは工場内専用運用として分離する。
- [厳守] `SecretCore` は `LocalServer` 配下のローカルIPC前提で利用し、TCP/HTTP で直接起動・疎通確認する運用を行わない。
- [厳守] 詳細な責任境界は `モジュール仕様書.md` を親定義とする。

## 1. Local MQTT（mosquitto）
- ブローカ起動:
  - `mosquitto -c mosquitto.conf -v`
- Subscribe確認:
  - `mosquitto_sub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/+/status/#" -v`
- Publish確認:
  - `mosquitto_pub -h localhost -p 8883 --cafile ca.crt -u "${MQTT_USERNAME}" -P "${MQTT_PASSWORD}" -t "iot/esp32s3-001/cmd/led" -m "{\"requestId\":\"req-001\",\"ledState\":\"on\"}"`

## 2. PlatformIO（ESP32-S3）
- プロジェクト初期化:
  - `pio project init --board esp32-s3-devkitc-1`
- ビルド:
  - `pio run`
- シリアル監視:
  - `pio device monitor`
- 書込み:
  - `pio run -t upload`

## 3. セキュリティ有効化時の注意
- [厳守] eFuse書込み系コマンドは事前レビュー必須。
- [禁止] 秘密鍵パスを履歴に残す実行方法（平文引数）を使用しない。
- [厳守] MQTT検証はTLS接続コマンドで実施し、平文ポート1883を使わない。
- [禁止] `--insecure` 相当の検証スキップを恒久運用しない。

## 6. LocalServer OTA配布（HTTPS）
- [重要] 2026-03-07 追加: OTA方式①では `LocalServer` が `4443/TCP` で `firmware.bin` を配布する。  
  理由: ESP32 が `mqtt.esplab.home.arpa:4443` へ HTTPS 接続して OTA を実施するため。
- OTA配布サーバー起動:
  - `cd IoT/LocalServer`
  - `npm run build`
  - `node dist/server.js`
- OTA配布確認:
  - `curl.exe -k https://127.0.0.1:4443/ota/manifest.json`
  - `curl.exe -k https://127.0.0.1:4443/ota/firmware.bin -o NUL`
- サーバー証明書確認（ホスト名付き）:
  - `openssl s_client -connect mqtt.esplab.home.arpa:4443 -servername mqtt.esplab.home.arpa`
- CA指定での検証確認:
  - `openssl s_client -connect mqtt.esplab.home.arpa:4443 -servername mqtt.esplab.home.arpa -CAfile IoT/ESP32/src/MQTT/ca.crt`
- Windows Firewall 受信許可（管理者PowerShell）:
  - `netsh advfirewall firewall add rule name="IoT LocalServer OTA 4443" dir=in action=allow protocol=TCP localport=4443`
  - `Get-NetFirewallRule -DisplayName "IoT LocalServer OTA 4443"`
- 到達確認:
  - `Test-NetConnection -ComputerName mqtt.esplab.home.arpa -Port 4443`
- DNS確認:
  - `nslookup mqtt.esplab.home.arpa 172.17.1.100`
- [厳守] OTA実機試験前に `4443/TCP` の受信許可と `OTA HTTPS started. httpsPort=4443` の両方を確認する。  
  理由: Node起動済でも Windows 側で受信拒否されると ESP32 では `connection refused` になるため。
- [厳守] 本番運用では `curl.exe -k` による検証スキップ確認だけで合格としない。  
  理由: 実運用では証明書検証有効が前提であるため。

### 6.1 OTA HTTPS 証明書検証用 CA 設定（003-0004）
- [重要] ESP32 の OTA は MQTT と同一 CA を使用する。経路: `/certs/mqtt-ca.pem` または `SENSITIVE_MQTT_TLS_CA_CERT`（フォールバック）。
- [厳守] LocalServer OTA サーバー証明書（`certs/server.crt`）と MQTT ブローカー証明書が**同一 CA により署名**されている必要がある。
- CA 投入手順（AP モード経由）:
  1. MQTT CA を ESP32 の `/certs` へ投入: `pushMqttCaByFileSyncE2E.mjs` または AP ログイン後の `targetArea=certs` で `mqtt-ca.pem` を upsert。
  2. ESP32 を再起動し、`/certs` の証明書を NVS メタへ同期させる。
- CA 投入手順（ビルド時フォールバック）:
  1. `sensitiveData.h` の `SENSITIVE_MQTT_TLS_CA_CERT` に OTA サーバーを検証可能な CA 証明書 PEM を設定。
  2. [禁止] 本番ではフォールバック依存にせず、`/certs` 主経路で運用する。
- 7014 試験（証明書検証有効 OTA 成功確認）:
  1. 正常証明書で OTA 取得を実行し、更新完了を確認する。
  2. 異常証明書（ホスト名不一致、信頼外 CA）で OTA を試行し、ESP32 側で拒否されることを確認する。

  - `curl.exe -X POST -F "firmware=@C:/mydata/project/myproject/learning_public/IoT/ESP32/.pio/build/esp32s3_secure/firmware.bin" http://127.0.0.1:3100/api/settings/firmware-upload`

  理由: サーバー運用拠点の現地時刻で状態判断しやすくするため。

## 9. OTA再認証（実装後に使用）
- [重要] OTA開始前に、Web UI で管理者パスワード再入力を要求する。  
  理由: OTA は高リスク操作であり、誤操作・離席時操作・なりすまし実行を抑止するため。
- 再認証API確認（例）:
  - `curl.exe -X POST http://127.0.0.1:3100/api/auth/ota/reauth -H "Content-Type: application/json" -d "{\"password\":\"<local-password>\"}"`
- セッション有効確認（例）:
  - `curl.exe -i http://127.0.0.1:3100/api/auth/ota/session`
- [厳守] 再認証成功後の有効期間は 3時間以内とし、ブラウザセッション終了後は再認証を要求する。
- [推奨] Cookie 属性は `HttpOnly` `Secure` `SameSite=Strict` を使用する。
