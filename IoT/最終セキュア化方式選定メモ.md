# IoT 最終セキュア化方式選定メモ

[最重要] 本書は `secure_final` 方針の意思決定を、**案B -> 案A -> 案C** の順でぶれずに進めるための判断基準を定義する。  
理由: セッションを跨いでも同一基準で Go/No-Go 判定し、場当たり的な切替を防ぐため。

[厳守] 最初に案B（IDF寄せ）を評価し、成立困難な根拠が確認された場合のみ案Aへ移行する。  
理由: `nvs_sec_provider` 系 API を正攻法で扱える可能性を先に確認するため。

[厳守] 案Aでも品質条件を満たせない場合のみ案Cへ移行する。  
理由: 本番安全性と開発継続性のバランスを段階的に確保するため。

[禁止] 判定未完了の方式を混在適用しない。  
理由: 試験結果と実機状態の対応関係が崩れ、原因追跡不能になるため。

## 2. 方式定義
### 2.1 案B（第一優先）
- 内容: ESP-IDF寄せで `nvs_sec_provider` 系 API を正式利用し、`secure_final` fail-closed 方針を成立させる。
- 期待: 設計意図（secure固定）と実装手段の整合を取りやすい。
- 主リスク: 既存Arduino資産（Wi-Fi/AP/UI/MQTT周辺）との互換性、移植工数、試験再実施範囲の増大。

### 2.2 案A（第二優先）
- 内容: 現行PlatformIO/Arduino前提で、利用可能APIの範囲に設計を寄せて secure/path と fallback/path を再定義する。
- 期待: 既存機能との整合を維持しながら進めやすい。
- 主リスク: 最終的な fail-closed 要件を満たすための代替実装が複雑化する可能性。

### 2.3 案C（最終手段）
- 内容: 当面は `esp32s3_secure` を主運用とし、`esp32s3_secure_rescue` を回復専用に固定する。
- 期待: 実機運用を止めずに安全側で継続可能。
- 主リスク: `secure_final` の製品版要件は未達のまま残る。

## 3. 分岐条件（Go/No-Go）
### 3.1 案BのGo条件
- `nvs_sec_provider` 系 API を実際に利用可能であること（ビルド/リンク/実機起動で確認）。
- 既存の AP/STA/MQTT/OTA 主要機能に重大な退行がないこと。
- `7099` と `7101` を再実施し、判定可能な証跡が揃うこと。
- [重要] ROM 使用率に実運用マージンがあること（目安: `99%` 常時超過を回避）。`#0026` 未解決時は Go にしない。
- [重要] `DEV-002` の保全とパーティション最適化後確認が完了していること。未完了時は secure 方式判定へ進まない。

### 3.2 案BのNo-Go条件（案Aへ移行）
- API未提供または実用不可が確定。
- 既存機能への影響が大きく、短期で品質回復できない。
- 追加工数が `007-C` 進行計画を大幅に阻害する。

### 3.3 案AのNo-Go条件（案Cへ移行）
- 代替設計でも fail-closed 要件の担保が困難。
- テストで不安定要因が残り、`007-C` の安全判定を満たせない。
- ROM マージン不足が継続し、パーティション再設計または減量策のいずれでも安全域へ戻せない。

## 6. 案B（IDF寄せ）影響度整理（2026-04-07）
[重要] 案Bは「正攻法」である一方、開発基盤の境界（Arduino中心運用）に最も大きく影響する。  
理由: `nvs_sec_provider` を素直に使える代わりに、ビルド・起動・試験の前提を広範囲で見直す可能性が高いため。

### 6.1 総合評価（現時点）
- 実現性: **中**（技術的には可能だが、既存資産との接合設計が必要）
- 修正リスク: **中〜高**
- 進め方の難易度: **高**（段階移行と比較検証を同時に管理する必要）
- 工数感: **中〜大**（1〜2週間級の調査・試験枠を想定。実装範囲次第で増加）
- 修正ボリューム: **中〜大**（ESP32ビルド基盤、NVS初期化経路、試験/運用手順の文書連動）
- 既存処理への影響: **中〜高**（AP/STA/MQTT/OTA の再確認範囲が広い）
- フレームワーク変更影響: **高**（「変更なし」は困難。最小でもハイブリッド運用設計が必要）

### 6.2 観点別詳細
- 修正リスク:
  - ビルド環境差分により `secure` / `secure_final` の挙動が再び分岐し、再現性が落ちるリスク。
  - 既存 Arduino 前提コード（Wi-Fi/AP/UI/MQTT）に副作用が出るリスク。
  - 量産手順・復旧手順（`app0=0x30000` など）との整合崩れリスク。
- 進め方:
  - 第1段階: IDF寄せの最小 PoC（`nvs_sec_provider` 到達確認のみ）
  - 第2段階: 既存機能影響の横断確認（AP/STA/MQTT/OTA/Production連携）
  - 第3段階: 文書・コマンド・試験の同期更新（`7101` 再判定含む）
  - [厳守] 各段階で `Go/No-Go` を確定し、未確定のまま次段へ進まない。
- 工数:
  - 調査/PoC: 2〜3日
  - 接合実装: 3〜5日
  - 試験/証跡整理: 2〜4日
  - 合計目安: 7〜12営業日（環境差分が少ない場合）
- 修正ボリューム:
  - `ESP32` ビルド定義と初期化経路（NVS暗号化まわり）が主対象。
  - `LocalServer` / `SecretCore` 本体は大改修必須ではないが、試験前提・運用手順の追随修正は必要。
  - ドキュメント更新量は中規模（`コマンド仕様書.md`、`試験仕様書.md`、`試験記録書.md`、`todo.md`）。

  - ただし「現行運用を崩さない」条件まで含めると、段階移行設計の巧拙が成否を左右する。

  - [重要] `nvs_sec_provider` を正攻法で使うなら、現行 Arduino 単独運用のままでは限界がある可能性が高い。
  - 実務上は「全面移行」か「IDF機能を限定導入するハイブリッド」の二択になりやすい。

## 7. 案B PoC 結果（2026-04-07 実施）

[最重要] 案B は **No-Go（継続不可）** と判定する。以下に根拠を示す。

### 7.1 技術的事実

| 項目 | 結果 |
|------|------|
| 現行フレームワーク | `framework-arduinoespressif32@3.20017.0` = Arduino-ESP32 **v2.0.17** |
| 内蔵 ESP-IDF | **v4.4.7** |
| `nvs_sec_provider` 導入 IDF バージョン | **v5.2 以降** |
| `nvs_sec_provider.h` | 現行パッケージ配下に **不在**（`#0025` と一致） |
| `nvs_flash_read_security_cfg_v2` | v4.4 系に **不在**。v4.4 は `nvs_flash_read_security_cfg()` のみ |
| `nvs_sec_provider_register_hmac` | v4.4 系に **不在** |
| Arduino-ESP32 v3.x | v3.0.0 = IDF v5.1, v3.2.0 = IDF v5.4.1 |
| v3.x での `nvs_sec_provider` | v5.2+ なら API あり。ただし v3.x は API 全面 Breaking Change |

- [出典] Arduino-ESP32 v2.0.17 リリースノート: `https://github.com/espressif/arduino-esp32/releases/tag/2.0.17` (based on ESP-IDF v4.4.7)
- [出典] `nvs_sec_provider` コンポーネント初出: `https://github.com/espressif/esp-idf/tree/v5.2/components/nvs_sec_provider`
- [出典] Arduino-ESP32 v3.2.0 リリースノート: `https://github.com/espressif/arduino-esp32/releases/tag/3.2.0` (based on ESP-IDF v5.4.1)

### 7.2 案Bの段階分解と実現性

| レベル | 内容 | 実現性 | 影響 |
|--------|------|--------|------|
| B-1: v4.4 内回避策 | `nvs_flash_read_security_cfg()` を直接使う | **不可**。HMAC provider は v4.4 に存在しない | — |
| B-2: Arduino v3.x へ更新 | v2 → v3 メジャー更新で IDF v5.1+ を導入 | **技術的に可能だが大規模** | Wi-Fi/GPIO/SPI/I2C/UART/BLE 全 API に Breaking Change |
| B-3: `framework = espidf` 切替 | Arduino 廃止し IDF ネイティブ移行 | **技術的に可能だが最大規模** | 全既存コード書き直し |
| B-4: ハイブリッド | Arduino + IDF dual framework | **不安定** | PlatformIO dual framework は整合問題の報告多数 |

### 7.3 No-Go 根拠（3.2 の条件すべてに該当）

1. **API未提供が確定**: `nvs_sec_provider` は IDF v5.2+ の新 API。v4.4.7 には一切存在せず、小規模パッチでは到達不可能。
2. **既存機能への影響が大きい**: Arduino v3.x は Breaking Change を含み、AP/STA/MQTT/OTA/LCD/BME280 の全モジュール互換性検証が必要。短期で品質回復できない。
3. **工数が `007-C` を大幅に阻害**: Breaking Change 対応を含めると 2〜4 週間級であり、6.2 の「7〜12 営業日」見込みを超過する。

### 7.3a PlatformIO 3.x 対応観点の補足

- [重要] 「Arduino-ESP32 v3.x へ上げれば案Bが成立するか」は、**PlatformIO 側がその系統を安定運用できるか**も同時に確認が必要である。
- [調査結果][2026-04-07] 少なくとも `platformio/espressif32` の公式系統については、Arduino core 3.0.x を正式サポートしない旨のコミュニティ側整理があり、代替として **`pioarduino/platform-espressif32`** が案内されている。
- [出典] `https://github.com/espressif/arduino-esp32/discussions/10039`
- [意味合い]
  - たとえ Arduino-ESP32 v3.x 自体へ移行できても、**「現行の official PlatformIO 基盤のまま」では完結しない可能性が高い**。
  - その場合、移行対象は Arduino core だけでなく **PlatformIO platform 自体** にも広がる。
  - 本プロジェクトでは `app0=0x30000` の書込み手順、既存の `platformio.ini`、再現済み運用コマンド、量産前文書の整合が重要であり、ビルド基盤まで community fork へ寄せると検証面積がさらに増える。
- [判定への影響]
  - 案Bの No-Go 根拠をさらに補強する材料とする。
  - つまり「Arduino v3.x 化すればよい」という単純な話ではなく、**PlatformIO の採用基盤まで再評価が必要**なため、短期 PoC の範囲を超える。

### 7.4 「根本見直し」の要否

**根本見直しは不要**。

- `nvs_sec_provider` 到達不可はフレームワーク世代差に起因し、設計・アーキテクチャの構造的問題ではない。
- 既存の `secureNvsInit.cpp`（パターン A/B/C）は正しく機能しており、IDF 世代が上がればパターン C に到達可能な設計になっている。
- [将来対応] Arduino-ESP32 v3.x 移行は中長期課題として `todo.md` に残す。

### 7.5 次アクション

- [厳守] 案B は No-Go として確定し、案A（現行 PlatformIO/Arduino 寄せ）の評価へ即移行する。
- [厳守] 案B の PoC 中に案A/C を混在適用していないことを確認済み。
- [推奨] 案A の検討では、v4.4 で利用可能な `nvs_flash_read_security_cfg()` + `nvs_flash_secure_init()`（Flash Encryption ベース）を代替経路として評価する。
- [将来対応] Arduino-ESP32 v3.x（IDF v5.x ベース）への移行計画は `008` 章の後続タスクとして管理する。

## 8. 案A PoC 計画（2026-04-07 追加）

[重要] 案Aは「現行 PlatformIO/Arduino 基盤を維持したまま、v4.4 系で実現可能な secure 経路へ設計を寄せる」方針で評価する。  
理由: 案B のようなフレームワーク大移行を避け、既存の AP/STA/MQTT/OTA/ProductionTool 前提を最大限維持したいから。

### 8.1 案Aの仮説

- v4.4 系では `nvs_sec_provider` / HMAC provider は使えないが、`nvs_flash_read_security_cfg()` + `nvs_flash_secure_init()` による **旧世代 NVS Encryption** は利用可能である。
- 旧世代方式は `nvs_keys` パーティション前提であり、現行 `esp32s3_dev002_16MB_candidate_a.csv` には **`nvs_keys` が未定義** のため、パーティション再設計が必要である。
- `Flash Encryption` を先行する `007-B` の段階順（`FE -> NVS -> SB -> 封鎖`）とは概ね整合する可能性がある。
- [確認済み][2026-04-07] 現行ローカルヘッダ（`framework-arduinoespressif32@3.20017.0`）には `nvs_flash_secure_init()`、`nvs_flash_generate_keys()`、`nvs_flash_read_security_cfg()`、`ESP_PARTITION_SUBTYPE_DATA_NVS_KEYS` が存在する。  
  したがって **「v4.4 では secure NVS API 自体が全く無い」わけではない**。案Aは PoC 実施価値ありと判断する。

### 8.2 案A PoC の成功条件

- [成功条件1] v4.4 系 API で **ビルド/リンク可能** な secure NVS 初期化経路を設計できること。
- [成功条件2] `nvs_keys` 追加を含むパーティション案を、`app0=0x30000`・OTA 2面・`spiffs`・`coredump` を崩さず提示できること。
- [成功条件3] 変更影響が `secureNvsInit.cpp` / `platformio.ini` / パーティションCSV / 文書群の中規模修正に収まり、既存 AP/STA/MQTT/OTA/Production 連携の全面再設計を要しないこと。
- [成功条件4] 不可逆工程に入る前に、少なくとも **コード/ビルド/設計レベルの可否線** を引けること。

### 8.3 案A PoC の失敗条件（案C 検討へ進む条件）

- [失敗条件1] v4.4 系 API では fail-closed 条件を実質的に担保できない。
- [失敗条件2] `nvs_keys` 追加や初期化方式変更により、既存パーティション/OTA/復旧手順の見直しが大規模化する。
- [失敗条件3] secure 経路の成立確認に、PoC 段階で不可逆工程を先行しなければならず、No-Go 前提の短期判定ができない。
- [失敗条件4] 既存 `secureNvsInit` の A/B/C 整理では吸収できず、ESP32 側 secure 設計全体の根本再設計が必要になる。

### 8.4 案A PoC でまず確認すること

1. 現行 v4.4 系ヘッダ/API で使用可能な NVS Encryption 呼び出し形を確定する。
2. `nvs_keys` パーティション追加案を作り、`DEV-002` の 16MB レイアウトへ無理なく挿入できるか確認する。
3. `secureNvsInit.cpp` の `APP_HAS_NVS_SEC_PROVIDER==0` 分岐を、単純な plaintext fallback ではなく **v4.4 secure candidate** として組み替えられるかを設計する。
4. その結果をもって、案A の Go/No-Go を文書で確定する。

### 8.4a 現時点の論点整理

- [有利材料]
  - 現行フレームワークのローカルヘッダに旧世代 secure NVS API が存在する。
  - 現在の `DEV-002` パーティション案には `0x24000-0x2FFFF` の空き領域があり、`app0=0x30000` を維持したまま `nvs_keys`（最小 0x1000）を挿入できる余地がある。
- [注意点]
  - 旧世代方式は `Flash Encryption` 前提であり、**実ランタイムの最終検証は不可逆工程抜きでは完了しない可能性が高い**。
  - したがって PoC の主眼は「コード/ビルド/パーティション/工程整合の可否線を引けるか」に置く。

### 8.5 PoC の非対象

- [厳守] `DEV-001` は変更しない。
- [厳守] `007-B` の eFuse 書込み、`Flash Encryption` 実適用、`Secure Boot` 実適用は行わない。
- [厳守] 案B/A/C を混在適用しない。
- [厳守] `LocalServer` / `SecretCore` の本体改修には広げない。

## 9. 変更履歴
- 2026-04-07c: PlatformIO の Arduino 3.x 対応観点（official PlatformIO ではなく community fork `pioarduino` が案内されていること）を追記し、案B No-Go 根拠を補強した。あわせて案A PoC 計画（成功条件/失敗条件/非対象）を追加。理由: 「Arduino v3.x に上げれば済むのか」という判断を、PlatformIO platform 基盤の観点も含めてぶれずに行えるようにするため。
- 2026-04-07b: 案B PoC を実施し、**No-Go** を確定。根拠: 現行 Arduino-ESP32 v2.0.17 は ESP-IDF v4.4.7 ベースであり、`nvs_sec_provider`（IDF v5.2+ 新 API）に到達不可能。Arduino v3.x へのメジャー更新は Breaking Change が広範で `007-C` 計画を大幅に阻害するため。次は案Aの評価へ即移行する。出典: Arduino-ESP32 v2.0.17 リリースノート、ESP-IDF v5.2 コンポーネントリポジトリ。
- 2026-04-07: 案Bの影響度整理（修正リスク、進め方、工数、修正ボリューム、実現性、既存影響、フレームワーク変更影響）を追加。理由: `008-0045` の出口条件である「技術難易度/工数/リスク/既存機能影響の文書化」を満たし、Go/No-Go判断を定量的に行えるようにするため。
- 2026-04-05: `DEV-002` の保全とパーティション最適化後確認を secure 方式判定の前提条件へ追加。理由: ROM問題を放置したまま方式選定だけ進めても、実機適用順が逆転して再度作業がぶれるため。
- 2026-04-05: ROM マージン条件（`#0026`）を Go/No-Go へ追加。理由: ライブラリ整合が取れても、バイナリ容量が上限近傍のままでは本番運用の更新継続性が担保できないため。
- 2026-04-05: 新規作成。理由: ユーザー指示の「案B優先、問題時は案A、それでも不可なら案C」を運用可能な判定ルールとして固定し、セッション跨ぎでも同じ判断を再現できるようにするため。
