# IoT 冗長化・多重接続仕様書

[重要] 本書は `Server` / `MQTT Broker` / `ESP32` / `WiFi AP` の多重構成、および `k-user` / `k-device` の冗長運用方針を定義する。  
理由: 単一構成前提のまま実装を進めると、将来の地域分散、台数分散、同時稼働冗長構成でデータ構造の作り直しが発生しやすいため。

[厳守] 本書は「現時点で固定する仕様」と「将来対応として予約する仕様」を明確に分けて記載する。  
理由: 先回り設計と現行実装範囲を混同すると、過剰実装と仕様ぶれの両方を招くため。

[禁止] 本書へ実鍵値、実パスワード、実証明書、復号後の `k-user` を記載しない。  
理由: 文書経由の機密漏えいを防止するため。

## 1. 用語
- `Server`
  - `ESP32` を管理する主体。通常運用 UI、REST API、監査、OTA 要求、`SecretCore` 呼出しを担当する。
- `MQTT Broker`
  - メッセージ配送経路を担う主体。`Server` と責務を分けて扱う。
- `SecretCore`
  - Rust による秘密処理層。`k-user` 復号、`k-device` 導出、暗号化ファイル復旧、監査を担当する。
- `WiFi AP`
  - `ESP32` が接続する無線アクセスポイント。大規模建屋で複数存在してよい。
- `serverId`
  - `Server` の論理識別子。
- `brokerId`
  - `MQTT Broker` の論理識別子。
- `wifiApId`
  - `WiFi AP` の論理識別子。
- `kUserProfileId`
  - `k-user` 保護方式、バックアップ方式、冗長構成の関連情報を識別する論理ID。
- `kDeviceProfileId`
  - `k-device` の共有/分離方針を識別する論理ID。

## 2. 現時点で固定する仕様
### 2.1 全体構成
- [厳守] `Server` は複数の `ESP32` を管理できる。  
  理由: これは現行運用でも成立しており、単一個体前提へ戻す理由がないため。
- [厳守] `1 Server - 複数 Broker` の関係を設計上許容する。  
  理由: 地域別配置、台数増加時の分散、将来の負荷分散に備えるため。
- [厳守] `ESP32` は複数 `Server` への同時接続を許容する前提で識別子と保存形式を設計する。  
  理由: 同時稼働冗長構成を後から追加できるようにするため。
- [厳守] システム内には複数 `WiFi AP` が存在してよく、`ESP32` ごとに接続先 `WiFi AP` が異なってよい。  
  理由: 大きな建屋での分散配置を前提とするため。
- [厳守] 現時点で各 `ESP32` が保持する `WiFi AP` 設定は固定1件とする。  
  理由: 現行実装範囲を増やしすぎず、接続先分散の要件だけを先に固定するため。

### 2.2 Server と Broker の関係
- [厳守] `Server` と `MQTT Broker` は別責務とする。  
  理由: 管理主体と通信経路主体を混同すると、冗長化や切替時の設計が不明瞭になるため。
- [推奨] `Server` は `brokerId[]` を保持し、同時接続可否、推奨接続先、運用目的を管理する。  
  理由: 将来の地域分散や台数分散をメタ情報で制御しやすくするため。
- [重要] 複数 Broker 対応でも、当面の推奨運用は `1 Server - 1 Broker` を基本とする。  
  理由: 初期運用と障害解析を単純に保つため。

### 2.3 ESP32 の接続方針
- [厳守] `ESP32` は `serverId` 単位で接続設定を管理できるようにする。  
  理由: 同一個体が複数 `Server` と同時接続する前提を崩さないため。
- [厳守] `ESP32` は `serverId` ごとに少なくとも以下を区別できるようにする。  
  理由: `Server` ごとに接続条件が異なる可能性があるため。
  - `brokerId`
  - `wifiApId`
  - MQTT 接続設定
  - 証明書メタ情報
  - `k-device`
  - 有効/無効フラグ
- [重要] テレメトリや `status` の同報は、同一 `Broker` を複数 `Server` が subscribe することで達成してよい。  
  理由: `ESP32` 側で複雑な送信先分岐を持たせなくてよいため。
- [厳守] OTA は要求元 `Server` の情報に対してのみ `ESP32` が応答する。  
  理由: 同時稼働冗長構成でも、要求元の責任範囲を明確にするため。
- [厳守] 同一種別の要求が複数 `Server` から到着した場合は、早いもの順に処理する。  
  理由: 優先順位の曖昧さを避け、現時点で最小限の競合解決ルールに留めるため。

### 2.4 `k-user` 保護方式
- [厳守] `k-user` 作成時に、少なくとも以下の保護方式を選択可能とする。  
  理由: 運用形態と復旧要件を両立するため。
  - `TPM由来方式`
  - `暗号化ファイル + パスワード方式`
- [厳守] 復号処理は `SecretCore` 内で完結させ、復号済み `k-user` をユーザーへ表示しない。  
  理由: バックアップ性を確保しつつ、秘密値の露出を防ぐため。
- [厳守] `TPM由来方式` を選んだ場合でも、`暗号化ファイル + パスワード` による復旧手段を持たせる。  
  理由: TPM 依存だけでは冗長構成や別PC復旧に対応できないため。
- [重要] バックアップファイルは別PCへコピー可能とし、復号は `SecretCore` が担当する。  
  理由: 第2 `Server` 構築や障害時復旧をユーザーが行えるようにするため。

### 2.5 `k-device` 方針
- [厳守] `k-device` は `Server` ごとに別物とする運用を選択可能とする。  
  理由: `Server` 間を強く分離したい運用があるため。
- [厳守] `k-device` は複数 `Server` 間で共通利用する運用も選択可能とする。  
  理由: 同時稼働冗長構成で鍵管理を単純化したい運用があるため。
- [厳守] `k-device` の共有/分離はユーザー確認のうえで選択する。  
  理由: セキュリティと運用容易性の優先順位が環境ごとに異なるため。
- [重要] MQTT 認証情報はユーザーマターとし、共通化を基本としつつ、必要なら `Server` ごとに分離してよい。  
  理由: 利用Brokerや運用ポリシーにより分離要件が変わるため。

## 3. 将来対応として予約する仕様
- [将来対応] `Server` 配下の複数 `Broker` への同時接続実装。
- [将来対応] 地域別 `Broker` の自動選択。
- [将来対応] 台数増加時の `Broker` 分散制御。
- [将来対応] `ESP32` の複数 `Server` 同時接続の本実装。
- [将来対応] `ESP32` の `serverId` 単位設定の永続化実装。
- [将来対応] `WiFi AP` 複数候補保持と自動切替。
- [将来対応] 冗長 `Server` 間の競合検知、排他、監査統合。
- [将来対応] `kDeviceProfileId` による共通鍵/分離鍵ポリシーの自動移行。

## 4. 識別子と保存メタ情報
- [厳守] 少なくとも以下の識別子を文書・DB・MQTT・NVS メタ情報で共通利用できるようにする。  
  理由: 将来の多重構成で参照崩れを防ぐため。
  - `serverId`
  - `brokerId`
  - `wifiApId`
  - `kUserProfileId`
  - `kDeviceProfileId`
- [推奨] `SecretCore` は `k-user` の保護方式、バックアップ方式、冗長構成可否をメタ情報として保持する。  
  理由: UI と復旧手順の整合を取りやすくするため。

## 5. 運用ルール
- [厳守] `Server` 冗長化を行う場合でも、復号済み `k-user` をファイルへ平文出力しない。  
  理由: バックアップと漏えい防止を両立するため。
- [厳守] `ESP32` は要求元を識別できる情報を保持し、OTA や高リスク要求の応答先を誤らない。  
  理由: 同時稼働 `Server` 間で応答の取り違えを防ぐため。
- [禁止] `Broker` が異なるだけで自動的に別 `Server` とみなす実装。  
  理由: `Server` と `Broker` の責務を混同するため。
- [推奨] 同時稼働冗長構成では、監査ログへ `serverId` / `brokerId` / `deviceId` を必ず記録する。  
  理由: 競合や運用事故の追跡を容易にするため。
