# IoT 製造・セキュア化手順書

[最重要] 本書は `ProductionTool` と実機を用いた製造・初回書込み・セキュア化の手順固定文書である。  
理由: 順番を誤ると復旧不能や証跡欠落を招くため、実施順と停止条件を作業者ごとにぶらさないため。

[厳守] 不可逆工程の順序は **`Flash Encryption -> NVS(HMAC) -> Secure Boot v2 -> 封鎖`** に固定する。  
理由: `RD_DIS`、署名済みバイナリ書込み、JTAG/Download系封鎖には順序依存があり、前後を入れ替えると文鎮化リスクが上がるため。

[禁止] 「途中まで焼けたので続行してよい」という判断を作業者判断だけで行わない。  
理由: 不可逆処理では残工程判断より隔離判断を優先すべきため。

- 工場端末での `ProductionTool` 利用
- 初回FW書込み
- `eFuse` 適用
- 署名済み/暗号化済みバイナリ適用
- 出荷レビューゲート前の証跡整理

## 3. 事前確認
- 対象機の管理番号、`MAC`、接続ポート、現在FW版を `デバイス管理表.md` と一致確認する。
- `ProductionTool`、`SecretCore`、`LocalServer` の起動前提を確認する。
- 当日使用する鍵の識別子だけを確認し、実鍵そのものは文書へ記載しない。
- `espefuse summary` before を保全し、4点証跡の格納先を作成する。
- [厳守] AP/STA/OTA の直前確認が NG のまま不可逆工程へ進まない。

## 4. 実行順序
1. `Flash Encryption`
2. `NVS(HMAC)` 鍵投入
3. `Secure Boot v2`
4. `封鎖`（`DIS_PAD_JTAG`、`DIS_USB_JTAG`、`ENABLE_SECURITY_DOWNLOAD`）
5. 最終 `OTA` 確認
6. `JTAG/シリアル` 安全確認
7. レビューゲート

## 5. 各工程の要点
### 5.1 `Flash Encryption`
- 暗号化済みバイナリの書込みと読戻し確認を行う。
- [厳守] 書込み後は再起動と通常起動確認を必ず行う。
- [重要] ここで NG なら次工程へ進まない。

### 5.2 `NVS(HMAC)`
- `BLOCK_KEY2` へ HMAC鍵を投入する。
- [厳守] `RD_DIS` write-protect より前に必要鍵投入を完了する。
- [重要] 実行時の `NVS` 初期化パターンは `patternA` / `patternC` をログで確認し、設計意図との差異があれば記録する。

### 5.3 `Secure Boot v2`
- 公開鍵 digest を正しい `BLOCK_KEY` へ書込み、`SECURE_BOOT_EN` を有効化する。
- bootloader / app を署名済みバイナリへ置き換える。
- [厳守] 署名済みバイナリ書込み前に `SECURE_BOOT_EN` を入れる場合は、適用手順を計画書どおりに固定し、再起動確認を省略しない。
- [禁止] 未署名バイナリのまま Secure Boot 有効後に通常起動確認を省略しない。

### 5.4 封鎖
- `DIS_PAD_JTAG`、`DIS_USB_JTAG`、`ENABLE_SECURITY_DOWNLOAD` を適用する。
- [厳守] `DIS_USB_SERIAL_JTAG` は適用しない。  
理由: シリアルログを残しつつ、危険操作を拒否する方針のため。
- [厳守] `DIS_DOWNLOAD_MODE` は適用しない。  
理由: `ENABLE_SECURITY_DOWNLOAD` と組み合わせた現行方針で、ROM側の安全確認余地を残すため。

## 6. 文鎮化リスクの高い誤操作
- [禁止] `Secure Boot` 有効化後に未署名バイナリを書き込んで起動確認すること。
- [禁止] `RD_DIS` 系を先に閉じてから必要鍵投入を試みること。
- [禁止] `ENABLE_SECURITY_DOWNLOAD` 適用前の読戻し証跡を採らずに封鎖へ進むこと。
- [禁止] 本番個体で `write_flash` 成否確認のための破壊試験を行うこと。
- [禁止] `DIS_USB_SERIAL_JTAG` を「シリアルを止めたいから」という理由だけで焼くこと。

## 7. 4点証跡
- `ProductionTool` 判定ログ
- ESP32 シリアルログ
- `espefuse summary` before/after
- `試験記録書.md` の当日追記

## 8. レビューゲートで確認すること
- 手順逸脱の有無
- 想定外ログの有無
- 再試行の有無と妥当性
- `7025`、`7100`、`7099` の揃い具合
- 2台目展開可否

## 8.1 実機接続前の固定手順
- [最重要][2026-04-04] 実機をPCへ接続する前に、以下を**上から順に**確認する。
- 1. `デバイス管理表.md` の対象行を開き、今回扱う個体が `DEV-001` であること、`MAC`、`COM`、`FW版`、`最新判定` を声出し確認する。
- 2. `試験記録書.md` の `7095`、`7099`、`7100`、`7101` を開き、当日追記欄の位置を先に確認する。
- 3. `eFuse適用チェックリスト.md` の `PC-001`〜`PC-014` を開き、未記入を埋める順序を確認する。
- 4. `コマンド仕様書.md` の `esp32s3_secure_final`、`7101`、`7100` の実行コマンドを別ウィンドウで参照できる状態にする。
- 5. 4点証跡の保存先を先に作成し、`ProductionTool` 判定ログ、ESP32 シリアルログ、`espefuse summary` before/after、`試験記録書.md` の追記先を対応付ける。
- 6. `SecretCore` / `LocalServer` / `ProductionTool` の起動条件、固定 Pipe 前提、AP 到達性前提を確認する。
- 7. [厳守] ここまで終わるまでは、実機へ `write_flash`、`burn-key`、`burn-efuse` を一切実行しない。
- [重要] 上記 1〜7 がそろった時点で初めて「接続してよい状態」とみなす。

## 8.2 当日記録先対応表
- [厳守][2026-04-04] 当日は、記録先をその場判断で増やさない。以下の固定先へ追記する。

| 記録対象 | 正式な記録先 | 目的 |
|---|---|---|
| 対象機識別、`MAC`、`COM`、最新判定 | `デバイス管理表.md` | 個体取り違え防止 |
| 各段階の4点証跡有無、`007` クローズ判定 | `試験記録書.md` `7095` | 1台目完了判定 |
| 各段階の安定性確認（OTA2回 / AP10分 / 通常10分） | `試験記録書.md` `7099` | 文鎮化兆候の早期検知 |
| JTAG/シリアル安全確認 | `試験記録書.md` `7100` | 出荷前安全条件の確認 |
| 不可逆開始直前の連携健全性確認 | `試験記録書.md` `7101` | `007-B` 進入可否判定 |
| 実行前チェック、停止条件、未記入有無 | `eFuse適用チェックリスト.md` | 進入禁止条件の可視化 |
| 想定外事象、再発防止、隔離理由 | `問題点記録書.md` | 手順起因/個体起因の切り分け |
| 当日の実行順・根拠・go/no-go | `本番セキュア化出荷準備試験計画書.md` と本書 | 実行判断の親定義 |

## 8.3 当日の即時停止条件
- [致命的][2026-04-04] 次のいずれかが起きたら、その場で作業を止めて**隔離判定優先**とする。
- `DEV-001` と実測 `MAC` / `COM` / `firmwareVersion` のいずれかが一致しない。
- `esp32s3_secure` と `esp32s3_secure_final` を取り違え、どちらで書き込むかを説明できない。
- `PC-001`〜`PC-011` に未記入または `NG` がある。
- 4点証跡の保存先が未作成、または after 証跡の採取担当が未定である。
- `7099`、`7100`、`7101` のどれを今から更新するのか説明できない。
- AP/STA/OTA の直前確認、または `ProductionTool` 連携確認で `NG` が出た。
- [禁止] 「接続したついで」「確認だけ」のつもりで不可逆コマンドを先に打たない。

## 9. `ProductionTool` GUI化との関係
- [重要][将来対応] GUI 化しても、工程ロック、対象機再確認、dry-run、停止条件、4点証跡採取は必須のままとする。
- [厳守] GUI 化は操作性向上であり、安全条件の緩和を意味しない。
- [推奨] GUI では工程ごとの `go/no-go` を画面固定し、次工程ボタンを条件付きで解放する。
