# 鍵管理および初期セットアップ設計仕様書（オンプレ独立前提）

[重要] 本仕様は、ユーザー環境ごとに `k-user` を生成し、その鍵からデバイス単位 `k-device` を決定的に生成する鍵管理方式を定義する。  
理由: デバイスごとの通信鍵を安全に生成しつつ、別PCへの複製悪用を抑止し、同一PC再インストール時の復旧性も確保するため。

## 1. 目的
- デバイスごとの通信鍵を安全に生成する。
- デバイス鍵の再配布を不要にする。
- 鍵を別PCへコピーして利用されることを防止する。
- 同一PCでの再インストール時には復旧可能とする。

## 2. 前提
- ユーザーごとにローカルPC上へ LocalServer をセットアップする。
- ユーザーごとにシステムは完全独立する。
- ユーザー間で鍵・データを共有しない。
- [重要] LocalServer本体は TypeScript / Node.js を継続利用する。
- [重要] 秘密処理は別層 `SecretCore` が担当する。
- [厳守] 現行の鍵保護方式は **TPM を前提** とする。
- [重要] 本書の対象は `k-user / k-device` 系であり、ESP32 セキュアブートや Flash Encryption に使う `k-iot` 系は別管理とする。
- [将来対応] TPM 非搭載環境や macOS 向けの同等ハードウェア拘束方式は別仕様として定義する。

## 2.1 責任範囲の親定義
- [厳守] `ProductionTool` / `LocalServer` / `SecretCore` / `ESP32` の責任範囲は `モジュール仕様書.md` を親定義とする。
- [厳守] 本書はそのうち `LocalServer` と `SecretCore` が扱う `k-user / k-device` 系の鍵管理境界を定義する。
- [禁止] `ProductionTool` が通常運用の `k-user / k-device` 管理主体になる前提を持ち込まない。

## 2.2 高リスクワークフロー方針
- [厳守] `k-user / k-device` 系の高リスク処理は `SecretCore` の Rust 主導ワークフローで実行する。
- [厳守] `LocalServer` は入力検証、開始要求、進捗表示、結果表示のみを担当し、逐次通信手順を実行しない。
- [厳守] 正規ワークフロー名は `runPairingSession()`、`runKeyRotationSession()`、`runSignedOtaCommand()` とする。
- [厳守] `LocalServer` が受け取る状態は `queued` / `running` / `waiting_device` / `verifying` / `completed` / `failed` に限定する。

## 3. 鍵構造
```text
TPM
 │
 └─ ランダム秘密 S_random
        │
        └─ HKDF + アプリ固有識別子
              │
              └─ k-user
                     │
                     └─ HKDF + MACアドレス
                           │
                           └─ k-device
```

### 3.1 鍵の役割
| 鍵/値 | 説明 |
|------|------|
| `S_random` | TPMで保護されるランダム秘密 |
| `k-user` | ユーザー環境のルート鍵 |
| `k-device` | デバイスごとの通信鍵 |
| `wrapped_secret` | `S_random` を TPM で暗号化した保存物 |

### 3.2 鍵命名方針
- [厳守] 本書では `K_master_user` 旧名称を廃止し、正式名称を `k-user` とする。
- [厳守] デバイス鍵の正式名称を `k-device` とする。
- [旧仕様] 旧文書に残る `K_master_user` / `K_device` は順次 `k-user` / `k-device` に読み替える。

## 4. TPM による秘密管理
### 4.1 初回インストール時
- `S_random` を CSPRNG で生成する。
- TPM 鍵で `S_random` を暗号化（ラップ）する。
- 暗号化後の `wrapped_secret` をファイル保存する。

```text
wrapped_secret = TPM_Encrypt(S_random)
```

### 4.2 保存方針
- [厳守] ディスクへ保存するのは `wrapped_secret` のみとする。
- [厳守] `S_random` の平文をファイル保存しない。
- [厳守] `wrapped_secret` の保存先は LocalServer の専用管理領域とし、一般設定ファイルと分離する。
- [禁止] `S_random` / `k-user` / `k-device` の実値をログ・画面・ドキュメントへ出力しない。

## 5. `k-user` の生成
### 5.1 起動時処理
- `wrapped_secret` を読み込む。
- TPM により復号する。
- `S_random` を取得する。
- HKDF により `k-user` を生成する。

```text
k-user = HKDF(
    ikm  = S_random,
    salt = S_app,
    info = "k-user-v1"
)
```

### 5.2 補足
- `S_app` はアプリケーション固有識別子（ソフトウェア埋め込み定数）とする。
- [厳守] `k-user` はディスクへ平文保存しない。
- [厳守] TypeScript 本体へ `k-user` の raw value を返却しない。

## 6. `k-device` の生成
### 6.1 device_id
- [厳守] `device_id` には ESP32 eFuse base MAC を使用する。
- [禁止] Wi-Fi MAC は device_id 用途に使用しない。

### 6.2 生成式
```text
k-device = HKDF(
    ikm  = k-user,
    salt = SHA256(mac_address),
    info = "k-device-v1"
)
```

### 6.3 設計意図
- デバイス鍵の永続保存を不要にする。
- 同一入力から決定的に再生成できる。
- `k-user` が漏えいしない限り、MAC アドレス既知だけでは `k-device` は求められない。

## 7. 鍵生成タイミング
### 7.1 `k-user`
- LocalServer 初回インストール時
- 鍵再発行実施時

### 7.2 `k-device`
- デバイス登録時
- デバイス通信開始時

## 8. 初期セットアップ・ペアリング
- [重要] 初期セットアップ時は `wrapped_secret` の生成と保存を実施する。
- [重要] 初回ペアリング時は、別仕様のペアリング手順に従って `device_id` を取得し、必要な時点で `k-device` を生成する。
- [厳守] 初回投入および再ペアリングは AP モード接続中の `runPairingSession()` を正規経路とし、その内部の `createPairingBundle` を基本経路とする。
- [厳守] `createPairingBundle` は `runPairingSession()` の内部処理として `SecretCore` が生成主体となり、`LocalServer` は入力検証、進行管理、監査表示のみを担当する。
- [厳守] AP モード投入では固定公開鍵検証と ECDH 都度セッション鍵を使用し、TypeScript へ raw `k-device` を返さない。
- [補足] 本書は「鍵をどう生成・再生成するか」を主対象とし、AP モードや初回書込み手順の詳細はペアリング仕様側で管理する。

## 9. Wi-Fi設定変更設計
### 9.1 通信方式
- MQTT over TLS
- Wi-Fi設定はアプリ層で AES-256-GCM 暗号化

### 9.2 暗号鍵
- [厳守] Wi-Fi設定更新の暗号鍵は `k-device` を使用する。

### 9.3 更新フロー
- MQTT で暗号化設定受信
- 復号
- 新設定を未確定保存
- 接続試験
- MQTT再接続成功時に確定
- 失敗時ロールバック

## 10. ローカル系モジュールの責務
### 10.1 `LocalServer`
- `SecretCore` の workflow 呼出し要求
- `device_db` 管理
- `public_id`、`keyVersion`、ペアリング状態のメタ情報管理
- AP セッション管理、再ペアリング進行管理、監査表示

### 10.2 `SecretCore`
- TPM による `wrapped_secret` 復号
- `k-user` の生成
- `k-device` の生成
- `runPairingSession()` 内部での `createPairingBundle` 生成
- `runPairingSession()`
- `runKeyRotationSession()`
- `runSignedOtaCommand()`
- デバイスとの暗号通信に必要な署名/暗号 payload 生成
- [重要][2026-03-21] `runKeyRotationSession()` は新 `k-user` 系統へ切り替えた request を受け、現段階では `runPairingSession()` と同じ secure bundle 送達・AP `state=applied` 完了判定経路を再利用する。理由: `LocalServer` が逐次暗号通信手順を保持せずに鍵切替を進めるため。

### 10.3 `ESP32`
- `k-device`、Wi-Fi、MQTT、OTA、認証情報の NVS 保存
- current/previous の 2 スロット鍵運用
- bundle 検証/復号

### 10.4 責務ルール
- [厳守] raw key を扱う暗号処理は `SecretCore` と `ESP32` のみで行う。
- [厳守] `LocalServer` は raw `k-user` / raw `k-device` を保持しない。
- [厳守] `LocalServer` は高リスク処理の逐次通信手順を保持せず、workflow の進捗と結果のみを扱う。

## 11. バックアップ
### 11.1 バックアップ対象
- `wrapped_secret`
- `k-user-backup.enc`（`k-user` 復旧用の暗号化バックアップファイル）
- `device_db`（デバイス一覧）
- [実装メモ][2026-04-24] 現行の `SecretCore` は `export_k_user_backup_file()` / `import_k_user_backup_file()` で `k-user` バックアップを扱い、LocalServer 側は `backup-k-user-for-7090.ps1` / `restore-k-user-after-7090.ps1` で `keyStore.json` を退避・復元する。加えて `backup-device-db-for-7090.ps1` / `restore-device-db-after-7090.ps1` で `settings.json` / `securityState.json` / `keyStore.json` / `wrapped_secret.bin` / `wrapped_k_user.bin` の復旧用スナップショットを扱う。  
  理由: `wrapped_secret` と `device_db` の責務を分離したまま、現行の運用コマンドと設計を一致させるため。

### 11.2 バックアップ方針
- [重要] `wrapped_secret` 単体は TPM で保護されるため、別PCでは復号できない。
- [重要] ただし `k-user-backup.enc` とパスワードを用いる復旧経路を許容する。
- [厳守] バックアップ物に `S_random` / `k-user` / `k-device` の平文を含めない。
- [推奨] `device_db` には鍵実値ではなく、デバイス識別情報と管理メタデータのみを保存する。

## 12. 復旧
### 12.1 同一PC再インストール時
1. `wrapped_secret` を復元する。
2. `device_db` を復元する。
3. TPM により `wrapped_secret` を復号する。
4. `k-user` を再生成する。
5. `k-device` を再生成する。

### 12.2 効果
- デバイス鍵の再配布が不要となる。
- LocalServer 側の再インストールでも同一PCであれば復旧可能となる。
- [重要] 暗号化バックアップファイルとパスワードを用いることで、別PCへの冗長復旧も可能となる。
- [重要] `device_db` を含む復旧用スナップショットを併用することで、`public_id`、`keyVersion`、ペアリング状態などの管理メタ情報も同時に戻せる。

## 13. セキュリティ前提
- [厳守] `k-user` はディスクに平文保存しない。
- [厳守] TPM が存在しない環境では `wrapped_secret` を復号できない。
- [厳守] ただし、別途定義した `k-user-backup.enc` とパスワードによる復旧経路は許容する。
- [厳守] 復号済み `k-user` は `SecretCore` の Rust モジュール外へ出さない。
- [重要] サーバー侵害時には `k-user` の機密性は保証されない。
- [重要] 本設計の主眼は「別PC複製の防止」と「同一PC再インストール復旧」の両立である。

## 14. 想定外事象
- PC故障
- TPM初期化
- TPM交換
- マザーボード交換

### 14.1 対応方針
- [重要] TPM前提復旧（`wrapped_secret`）が使えない場合は、`k-user-backup.enc` + パスワードによる復旧を優先する。
- [重要] 復旧不能な場合は鍵の再生成が必要となる。
- [重要] 鍵再生成時は、デバイス鍵の再配布または再登録が必要となる。

## 15. サポート方針
- [厳守] メーカーは `k-user` / `S_random` を保持しない。
- [厳守] メーカー共通鍵（`K_master_global`）を導入しない。
- [推奨] サポート時は `wrapped_secret` を含まない診断パッケージを別途生成する。
