# 鍵管理および初期セットアップ設計仕様書（オンプレ独立前提）

[重要] 本仕様は、ユーザー環境ごとに `k-user` を生成し、その鍵からデバイス単位 `k-device` を決定的に生成する鍵管理方式を定義する。  
理由: デバイスごとの通信鍵を安全に生成しつつ、別PCへの複製悪用を抑止し、同一PC再インストール時の復旧性も確保するため。

## 1. 目的
- デバイスごとの通信鍵を安全に生成する。
- デバイス鍵の再配布を不要にする。
- 鍵を別PCへコピーして利用されることを防止する。
- 同一PCでの再インストール時には復旧可能とする。

## 2. 前提
- ユーザーごとにローカルPC上へ LocalServer をセットアップする。
- ユーザーごとにシステムは完全独立する。
- ユーザー間で鍵・データを共有しない。
- [重要] LocalServer本体は TypeScript / Node.js を継続利用する。
- [重要] 秘密処理は別層 `SecretCore` が担当する。
- [厳守] 現行の鍵保護方式は **TPM を前提** とする。
- [重要] 本書の対象は `k-user / k-device` 系であり、ESP32 セキュアブートや Flash Encryption に使う `k-iot` 系は別管理とする。
- [将来対応] TPM 非搭載環境や macOS 向けの同等ハードウェア拘束方式は別仕様として定義する。

## 3. 鍵構造
```text
TPM
 │
 └─ ランダム秘密 S_random
        │
        └─ HKDF + アプリ固有識別子
              │
              └─ k-user
                     │
                     └─ HKDF + MACアドレス
                           │
                           └─ k-device
```

### 3.1 鍵の役割
| 鍵/値 | 説明 |
|------|------|
| `S_random` | TPMで保護されるランダム秘密 |
| `k-user` | ユーザー環境のルート鍵 |
| `k-device` | デバイスごとの通信鍵 |
| `wrapped_secret` | `S_random` を TPM で暗号化した保存物 |

### 3.2 鍵命名方針
- [厳守] 本書では `K_master_user` 旧名称を廃止し、正式名称を `k-user` とする。
- [厳守] デバイス鍵の正式名称を `k-device` とする。
- [旧仕様] 旧文書に残る `K_master_user` / `K_device` は順次 `k-user` / `k-device` に読み替える。

## 4. TPM による秘密管理
### 4.1 初回インストール時
- `S_random` を CSPRNG で生成する。
- TPM 鍵で `S_random` を暗号化（ラップ）する。
- 暗号化後の `wrapped_secret` をファイル保存する。

```text
wrapped_secret = TPM_Encrypt(S_random)
```

### 4.2 保存方針
- [厳守] ディスクへ保存するのは `wrapped_secret` のみとする。
- [厳守] `S_random` の平文をファイル保存しない。
- [厳守] `wrapped_secret` の保存先は LocalServer の専用管理領域とし、一般設定ファイルと分離する。
- [禁止] `S_random` / `k-user` / `k-device` の実値をログ・画面・ドキュメントへ出力しない。

## 5. `k-user` の生成
### 5.1 起動時処理
- `wrapped_secret` を読み込む。
- TPM により復号する。
- `S_random` を取得する。
- HKDF により `k-user` を生成する。

```text
k-user = HKDF(
    ikm  = S_random,
    salt = S_app,
    info = "k-user-v1"
)
```

### 5.2 補足
- `S_app` はアプリケーション固有識別子（ソフトウェア埋め込み定数）とする。
- [厳守] `k-user` はディスクへ平文保存しない。
- [厳守] TypeScript 本体へ `k-user` の raw value を返却しない。

## 6. `k-device` の生成
### 6.1 device_id
- [厳守] `device_id` には ESP32 eFuse base MAC を使用する。
- [禁止] Wi-Fi MAC は device_id 用途に使用しない。

### 6.2 生成式
```text
k-device = HKDF(
    ikm  = k-user,
    salt = SHA256(mac_address),
    info = "k-device-v1"
)
```

### 6.3 設計意図
- デバイス鍵の永続保存を不要にする。
- 同一入力から決定的に再生成できる。
- `k-user` が漏えいしない限り、MAC アドレス既知だけでは `k-device` は求められない。

## 7. 鍵生成タイミング
### 7.1 `k-user`
- LocalServer 初回インストール時
- 鍵再発行実施時

### 7.2 `k-device`
- デバイス登録時
- デバイス通信開始時

## 8. 初期セットアップ・ペアリング
- [重要] 初期セットアップ時は `wrapped_secret` の生成と保存を実施する。
- [重要] 初回ペアリング時は、別仕様のペアリング手順に従って `device_id` を取得し、必要な時点で `k-device` を生成する。
- [補足] 本書は「鍵をどう生成・再生成するか」を主対象とし、AP モードや初回書込み手順の詳細はペアリング仕様側で管理する。

## 9. Wi-Fi設定変更設計
### 9.1 通信方式
- MQTT over TLS
- Wi-Fi設定はアプリ層で AES-256-GCM 暗号化

### 9.2 暗号鍵
- [厳守] Wi-Fi設定更新の暗号鍵は `k-device` を使用する。

### 9.3 更新フロー
- MQTT で暗号化設定受信
- 復号
- 新設定を未確定保存
- 接続試験
- MQTT再接続成功時に確定
- 失敗時ロールバック

## 10. ローカルサーバーの責務
- TPM による `wrapped_secret` 復号
- `k-user` の生成
- `k-device` の生成
- デバイスとの暗号通信処理
- [厳守] 暗号処理はすべて LocalServer 内部で行う。

## 11. バックアップ
### 11.1 バックアップ対象
- `wrapped_secret`
- `device_db`（デバイス一覧）

### 11.2 バックアップ方針
- [重要] `wrapped_secret` は TPM で保護されるため、別PCでは復号できない。
- [厳守] バックアップ物に `S_random` / `k-user` / `k-device` の平文を含めない。
- [推奨] `device_db` には鍵実値ではなく、デバイス識別情報と管理メタデータのみを保存する。

## 12. 復旧
### 12.1 同一PC再インストール時
1. `wrapped_secret` を復元する。
2. `device_db` を復元する。
3. TPM により `wrapped_secret` を復号する。
4. `k-user` を再生成する。
5. `k-device` を再生成する。

### 12.2 効果
- デバイス鍵の再配布が不要となる。
- LocalServer 側の再インストールでも同一PCであれば復旧可能となる。

## 13. セキュリティ前提
- [厳守] `k-user` はディスクに平文保存しない。
- [厳守] TPM が存在しない環境では `wrapped_secret` を復号できない。
- [厳守] 別PCへファイルコピーしても鍵は利用できない。
- [重要] サーバー侵害時には `k-user` の機密性は保証されない。
- [重要] 本設計の主眼は「別PC複製の防止」と「同一PC再インストール復旧」の両立である。

## 14. 想定外事象
- PC故障
- TPM初期化
- TPM交換
- マザーボード交換

### 14.1 対応方針
- [重要] 上記発生時は鍵の再生成が必要となる。
- [重要] この場合、デバイス鍵の再配布または再登録が必要となる。

## 15. サポート方針
- [厳守] メーカーは `k-user` / `S_random` を保持しない。
- [厳守] メーカー共通鍵（`K_master_global`）を導入しない。
- [推奨] サポート時は `wrapped_secret` を含まない診断パッケージを別途生成する。
